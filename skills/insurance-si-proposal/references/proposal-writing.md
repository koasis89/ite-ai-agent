# 제안서 작성 가이드 (목차별 + 장표 구성)

## 1. PPTX 장표 구성 원칙

국내 SI 제안서는 가로형(16:9 또는 A4 가로) 슬라이드 문서다. 장표마다 다음 구조를 지킨다:

```
┌─────────────────────────────────────────────┐
│ [목차 경로] Ⅱ. 전략 및 방법론 > 2. 추진전략      │ ← 상단 소제목(현재 위치)
│ ■ 헤드메시지: 이 장표의 결론 한 문장            │ ← 거버닝 메시지
├─────────────────────────────────────────────┤
│                                             │
│   본문: 다이어그램·표 중심, 텍스트는 보조        │
│   (개념도, 프로세스 흐름, 비교표, 구성도)        │
│                                             │
├─────────────────────────────────────────────┤
│ 하단: 관련 요구사항 ID │ 페이지 번호            │
└─────────────────────────────────────────────┘
```

- **헤드메시지만 이어 읽어도 제안서 전체 논리가 성립**해야 한다. 초안 작성 후 헤드메시지만 뽑아 읽어보고 흐름이 끊기면 재구성한다.
- 텍스트 나열보다 **구조화된 시각물**(단계 다이어그램, As-Is/To-Be 비교, 매트릭스, 아키텍처 구성도)을 우선한다. pptx 스킬의 도형·표 기능으로 구현한다.
- 한 장표 하나의 주장. 주장이 둘이면 장표를 나눈다.
- 숫자를 쓴다: "신속한 대응" 대신 "장애 접수 후 30분 내 초동 대응".

## 2. 표준 목차별 작성 가이드

공제·보험 SI RFP의 전형적 목차(Ⅰ~Ⅶ)를 기준으로 한다. 실제 RFP가 목차를 지정하면 그것을 그대로 따르고, 아래는 각 파트에 담을 내용의 가이드로 쓴다.

### Ⅰ. 일반현황
- 제안사 일반현황·연혁·조직·재무·유사실적: **사용자 제공 자료만** 사용. 없으면 플레이스홀더. RFP 지정 서식(서식 1~5호 등)이 있으면 해당 서식 항목에 맞춘다.
- 수행조직: 발주기관 협의체(공동수급이면 지분율 명시), PM-PL-파트 구조도, 업무분장표. PM 재직 요건을 확인해 반영한다.
- 투입인력: 핵심인력(PM/PL) 중심, 불필요한 학력·자격 나열은 배제하라는 지침이 일반적이다.

### Ⅱ. 전략 및 방법론 — 승부처 1
- **사업이해도**: 이 파트가 첫인상이다. ① 발주기관 현황과 문제의식을 발주기관 자신보다 구조적으로 정리(업무 가치사슬 위에 문제점 매핑), ② 사업의 목적·범위·전제조건 재구성, ③ 목표시스템 개념도(채널-코어-AI-데이터-인프라 계층 구성도), ④ 제안의 특징·강점 요약. RFP의 추진배경 문구를 소재로 쓰되 그대로 베끼지 말고 재해석한다.
- **추진전략**: RFP 반복 키워드에 응답하는 3~4개 전략 축 + 각 축의 실행 방안. 평가기준에 "위험요소 고려"가 있으면 리스크 기반 전략(예: 데이터 전환 리스크→검증 3단계, 오픈 리스크→단계적 오픈)을 명시적으로 포함한다.
- **적용기술**: 실현 가능성이 평가 포인트다. 기술별로 "무엇을-어디에-왜"와 검증 방법(프로토타입, PoC)을 쓴다. 과시적 신기술 나열은 역효과.
- **표준 프레임워크**: 전자정부 표준프레임워크 등 적용 계획 + **예상 문제점과 대응방안**(평가기준이 명시적으로 요구). 문제점을 안 쓰면 감점이다.
- **개발 방법론**: 방법론 절차·기법 + 적용 경험 + **단계별 산출물 목록과 제출 시기**.

### Ⅲ. 기술 및 기능 — 승부처 2 (배점 최대)
- **기능요구사항**: 요구사항을 업무 영역별로 그룹핑하고, 영역별로 [현황·요구 요지 → 구현방안(화면·프로세스·데이터 관점) → 기대효과] 구조로 쓴다. 각 장표에 대응 요구사항 ID를 표기한다. AI 기능은 "AI 보조 + 담당자 최종 검토 + 근거 제시" 구조로 쓴다.
- **보안요구사항**: 민감정보(진료·사고 정보) 보호를 중심으로 관리적·기술적·물리적 보안 체계, 개발보안(시큐어코딩), 취약점 점검, 개인정보 처리 방안.
- **데이터요구사항**: **전환(마이그레이션) 계획·검증 방법·오류 데이터 처리**가 평가기준에 명시되는 것이 보통이다. 전환 대상 분석→정비→전환→검증(건수·금액 대사, 표본 검증)→오류 처리 절차를 단계 다이어그램으로 제시한다.
- **시스템 운영요구사항**: 운영 절차·모니터링 + **비상사태 대응방안**(장애 등급별 대응, 백업/복구).
- **제약사항**: RFP의 COR 항목별 준수 방안을 표로 정리한다.

### Ⅳ. 성능 및 품질
- 성능: 목표 지표(응답시간, 동시사용자, OCR 인식률 등)와 달성 방안·측정 방법. 장비 구성도(클라우드면 자원 구성·확장 설계).
- 인터페이스: 시스템 인터페이스(연계 방식 대안 비교 — 장단점 분석이 평가기준)와 사용자 인터페이스(UX/UI 설계·검증 계획, 웹 접근성)를 구분해서 쓴다.
- 테스트·품질: 단위→통합→시스템→성능→인수 테스트 체계, 품질 지표·검토 절차. 배점이 작으므로 표준적으로 간결하게.

### Ⅴ. 프로젝트 관리
- 관리방법론: 일정·위험(이슈)·보안·형상/산출물 관리 + **문제 발생 시 보고 체계**. WBS 수준의 일정계획(단계별 마일스톤, 주요 산출물 시점).
- 일정: 사업기간에 맞는 단계 구성(분석-설계-개발-테스트-전환-안정화). 오픈 시점과 데이터 전환 리허설 시점을 명시하면 신뢰를 얻는다.

### Ⅵ. 프로젝트 지원
- 품질보증(SP인증 등 보유 인증 명시 — 우대 항목), 시험운영, 교육훈련(대상자별: 현업/운영자/관리자), 유지관리·하자보수(기간·조직·SLA), 기밀보안, 비상대책(백업·복구·장애 대응).
- 공제조합은 IT 인력이 적으므로 교육·기술이전·안정화 지원을 구체적으로 쓰면 배점 대비 신뢰 효과가 크다.

### Ⅶ. 상생협력 및 하도급
- 중소기업 참여비율, 하도급 계획(없으면 "하도급 없음"을 명시 — 통상 최고 등급), 하도급이 있으면 대금지급 방식·적정성.

## 3. 제안요약서

제안서 완성 후 별도 파일로 만든다. 통상 10~20장. 구성: 사업이해 1~2장 → 파트별 핵심 방안 각 1~2장 → 차별화 요약 1장. **본문 장표의 축약이 아니라 평가위원이 채점 직전에 훑는 요약본**이므로, 기술평가 항목 순서에 맞춰 재배열하고 각 장에 본문 페이지 참조를 단다.

## 4. 제안설명회(PT) 자료

통상 15~20분 발표용 15~25장. 제안서 목차 순서가 아니라 **스토리 순서**로 재구성한다:

1. 왜 우리인가 (도메인 이해 + 핵심 실적, 1~2장)
2. 이 사업을 무엇이라고 이해했는가 (문제의 재정의, 1~2장)
3. 어떻게 해결하는가 (전략 축별 핵심 방안, 5~8장 — 시간의 절반)
4. 리스크는 어떻게 통제하는가 (데이터 전환·오픈·운영, 2~3장)
5. 약속 (일정·품질·지원 체계, 1~2장)

PM이 직접 발표하는 요건이 일반적이므로, 발표자 노트에 장표별 발언 요지를 담는다. 질의응답 대비로 예상 질문 10개와 답변 요지를 별도 정리해 주면 좋다.
