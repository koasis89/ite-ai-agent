# 전체 스킬 실행 계약 레이어 설계안

작성일: 2026-07-03
대상 저장소: ite-ai-agent
기준 카탈로그: src/catalog/manifest.json

추가 기준(누락 보완): skills

## 1. 목적

Electron 환경에서 docx 기능만 개별 구현하는 방식에서 벗어나, 모든 스킬을 동일한 실행 계약 레이어로 수용한다.
이 설계는 다음을 보장한다.

- 스킬별 입력/출력 검증의 일관성
- 실행 실패 시 표준 에러 코드와 표준 오류 봉투
- 활성/비활성, deprecated, merged, internal 상태에 대한 공통 정책
- 외부 코드 재사용 시 클린룸 방식 준수

## 2. 적용 범위

- Electron UI, Main IPC, Preload 브릿지, Runtime 실행기
- manifest의 skills 전체 항목
- skills 하위 로컬 스킬 전체 항목
- 향후 skills 추가 시 동일 구조 확장

## 3.3 실행 흐름

1. Renderer가 actionId와 payload를 전달
2. Main에서 registry 조회
3. status-policy 검증
4. inputSchema 검증
5. executor 실행
6. outputSchema 검증
7. 표준 결과 봉투 반환

## 4. 상태별 정책

- active
  - 실행 가능
  - 계약 검증 필수
- deprecated
  - 기본 실행 차단
  - 옵션으로 허용 가능하나 경고 강제
- merged
  - canonical action으로 자동 리다이렉트
- alias
  - canonical로 정규화 후 실행
- internal
  - UI 노출 금지
  - 내부 플로우에서만 호출 허용

## 5. 외부 코드 가져오기 전략

## 5.1 원칙

- 구현 복사 금지, 패턴만 차용
- 타입/함수명/오류체계는 저장소 표준으로 재작성
- 외부 코드 직접 포함 시 출처, 라이선스, 수정 이력 의무 기록

## 5.2 레퍼런스 패턴 반영

- ts-rest에서 차용
  - 계약 우선 설계
  - 입력/출력 양방향 검증
- tRPC에서 차용
  - procedure 단위 input/output parser
  - 오류 포맷 표준화
- openai-node에서 차용
  - schema 기반 structured parse
  - strict schema 옵션 정책화

## 5.3 라이선스 및 보안 게이트

- 도입 전 체크리스트
  - 라이선스 유형 확인
  - 유지보수 활동성 확인
  - 취약점 여부 확인
- 도입 후 체크리스트
  - NOTICE 또는 문서 출처 기록
  - 보안 스캔 파이프라인 포함
  - 계약 테스트 통과 필수

### 6.1.1 우선순위 규칙

- 문서/오피스 내보내기 요청은 manifest shortcut보다 6.1 로컬 스킬 매핑을 우선 적용
- `docx`, `xlsx`는 actionId를 명시적으로 분리
  - `word.export-template`
  - `excel.export-template`
- Renderer 스킬 목록 노출 시 6절(manifest)과 6.1절(local)을 병합하되, 동일 기능이면 local을 우선 표시

## 7. Electron 연계 구성

## 7.1 Main IPC

- omx:skills:invoke
- omx:skills:list
- omx:skills:status-policy

## 7.2 Preload API

- invokeSkill(action)
- listSkillContracts()
- getSkillExecutionPolicy()

## 7.3 Renderer UI

- 읽기 전용 스킬 카탈로그 패널
  - name
  - status
  - canonical
  - enabled
- 실행 패널
  - active만 기본 노출
  - deprecated는 고급 옵션에서만 노출

## 8. 관측성과 감사

- 모든 실행은 requestId 기반 로그 남김
- 실패는 code 단위 집계
- deprecated 또는 merged 호출 비율 별도 집계
- internal 호출은 actor 감사 로그 필수

## 9. 품질 게이트

- 계약 테스트
  - input reject 케이스
  - output reject 케이스
- 정책 테스트
  - deprecated 차단
  - merged canonical 라우팅
  - internal 접근 통제
- 통합 테스트
  - Renderer -> Preload -> IPC -> Executor -> 응답

## 10. 단계별 적용 제안

1. 공통 코어 계약과 에러 체계 구현
2. docx를 파일럿 action으로 이전
3. execution/planning 계열 스킬로 확장
4. shortcut/utility 계열 정책 반영
5. 전체 회귀 테스트 및 운영 지표 대시보드 반영
