## 1. 정의

### OMX 데스크톱 에이전트란?

**OMX 데스크톱 에이전트**는 OMX CLI 코어 엔진을 기반으로 동작하는 Electron 데스크톱 애플리케이션입니다. 클로드 AI(Claude Desktop) 스타일의 세련된 채팅 UX를 제공하면서, 내부적으로는 AI 에이전트가 태스크 계획(Planning) → 실행(Execution) → 완료(Completion) 수명주기를 자동으로 관리합니다.

> **핵심 원칙**: Electron 앱은 LLM SDK 또는 복잡한 프롬프트 로직을 직접 보유하지 않습니다. 모든 AI 지능은 OMX CLI 코어 엔진이 담당하며, Electron은 이를 **비차단(Non-blocking) 스트림으로 조율하는 수려한 상태 비저장 뷰(Stateless View)** 역할만 수행합니다.

### 핵심 용어

| 용어 | 설명 |
|------|------|
| **OMX CLI** | 로컬에 설치된 AI 에이전트 실행 엔진. `omx` 명령으로 구동 |
| **Truth Boundary** | `.omx/` 디렉터리. 에이전트의 상태, 로그, 결과의 유일한 원본 |
| **Ndjson** | 줄바꿈 구분 JSON. CLI 스트림과 훅 로그의 표준 통신 포맷 |
| **IPC** | Electron Main ↔ Renderer 간 프로세스 간 통신 채널 |
| **Interlude** | 에이전트가 사용자의 판단을 기다리는 차단형 대기 상태 |
| **Drift** | 실행 로그(`hooks-*.jsonl`)와 매니페스트(`manifest.json`) 간의 설정 불일치 |

## 2. 역할 및 구성 요소

### 전체 아키텍처 구조

```
┌─────────────────────────────────────────┐
│         Electron Renderer (React)        │
│  채팅 UI / 대시보드 / 오버레이 컴포넌트  │
└──────────────────┬──────────────────────┘
                   │ contextBridge IPC
┌──────────────────┴──────────────────────┐
│         Electron Main (Interop 브로커)   │
│  CLIWrapper / StateWatcher / IPC 채널   │
└──────────────────┬──────────────────────┘
                   │ child_process.spawn
┌──────────────────┴──────────────────────┐
│         OMX CLI Core Engine              │
│  ask.ts / sparkshell.ts / MCP 서버      │
│  .omx/state/*.json (Truth Boundary)     │
│  .omx/logs/hooks-*.jsonl (Hook Logs)    │
└─────────────────────────────────────────┘
```

### 레이어별 역할

#### Electron Renderer (UI 레이어)
- **역할**: 사용자가 보는 모든 화면을 React 컴포넌트로 렌더링
- **특징**: 상태를 직접 보유하지 않음. IPC 이벤트를 수신하여 화면만 갱신
- **주요 컴포넌트**: `ChatContainer`, `LifecycleDashboard`, `TaskTimeline`, `WikiOverlay`, `DeferredSkillsNotice`, `ErrorGuideOverlay`

#### Electron Main (브로커 레이어)
- **역할**: CLI 프로세스를 관리하고, 파일 시스템을 감시하며, Renderer와 IPC로 통신
- **주요 모듈**:

| 모듈 | 파일 | 역할 |
|------|------|------|
| EnvChecker | `src/main/env/env-checker.ts` | 앱 시작 시 OMX 환경 진단 |
| CLIWrapper | `src/main/cli/cli-wrapper.ts` | `omx` 명령 비동기 실행 |
| StateWatcher | `src/main/state/state-watcher.ts` | `.omx/state/` 파일 실시간 감시 |
| HookTailer | `src/main/logs/hook-tailer.ts` | `.omx/logs/` 훅 로그 스트림 구독 |
| DriftDetector | `src/main/ops/drift-detector.ts` | 로그-매니페스트 정합성 검증 |
| MultiStateValidator | `src/main/state/multi-state-validator.ts` | 복합 상태 전이 규칙 검증 |

#### OMX CLI Core (AI 지능 레이어)
- **역할**: 실제 LLM 추론, 태스크 계획 및 실행, MCP 서버 구동
- **Truth Boundary**: `.omx/state/*.json`, `.omx/logs/hooks-*.jsonl` — Electron이 절대적 원본으로 신뢰

## 3. 설치 및 수행 방법

### 3.1 사전 요구 사항

| 항목 | 버전 | 확인 명령 |
|------|------|-----------|
| Node.js | 20 LTS 이상 | `node --version` |
| OMX CLI | 최신 버전 | `omx --version` |
| Codex 인증 | 로그인 완료 | `codex login status` |

> **OMX CLI 설치 안내**: Linux/macOS에서는 `리눅스 Node설치.md` 문서를 참고하세요.

### 3.2 OMX 환경 초기화

앱을 처음 사용하기 전에 OMX 환경을 설정해야 합니다.

```bash
# 1. OMX 환경 설정 (최초 1회)
omx setup

# 2. 환경 진단 — 정상 여부 확인
omx doctor --json

# 3. Codex 인증 상태 확인
codex login status

# 4. Codex 인증이 만료되거나 인증이 필요한 경우, 다음 명령으로 로그인 진행 
codex logout
codex login
```
`omx doctor` 출력에서 `"ok": true`가 반환되고, `codex login status`에서 `authenticated` 상태가 확인되어야 앱이 정상 구동됩니다.

### 3.3 의존성 설치

```bash
# 워크스페이스 루트에서 실행
npm install
```

### 3.4 개발 모드 실행

```bash
# Electron 개발 서버 시작 (Hot Reload 포함)
cd Electron
npm run dev:electron
```

### 3.5 프로덕션 빌드 및 실행

```bash
# 1. 타입 체크
npx tsc --noEmit

# 2. 전체 빌드
npm run build

# 3. 실행 파일 패키징
npm run dist

# 패키징 결과물은 dist/ 폴더에 생성됨
```

### 3.6 앱 구동 시 자동 진단 흐름

앱이 시작되면 다음 순서로 환경 진단이 자동 수행됩니다.

```
1. omx --version 체크    → CLI 설치 여부 및 버전 검증
2. omx doctor --json     → 설정 파일(.codex/config.toml, hooks.json) 유효성 확인
3. codex login status    → Codex 인증 정합성 확인
4. omx exec 핸드셰이크   → 런타임 셸 실행 권한 검증 (응답: "OMX-EXEC-OK")
```

진단 중 이상이 감지되면 앱 상단에 **환경 오류 배너**가 표시됩니다.

### 3.7 테스트 실행

```bash
# 전체 단위 테스트
npm run test

# 특정 테스트 파일 실행
npx vitest run src/test/env-checker.test.ts
npx vitest run src/test/EL-224.test.ts
npx vitest run src/test/drift-detector.test.ts
npx vitest run src/test/EL-226.test.ts
```

## 4. UI 화면 기능 사용법

### 4.1 기본 채팅 화면 (메인 뷰)

앱을 실행하면 다음 구조의 메인 화면이 나타납니다.

```
┌───────────────────────────────────┬───────────────────────┐
│  💬 메인 채팅 (Chat Stream)       │  📋 에이전트 타임라인 │
│                                   │                       │
│  👤 User                          │  [✅] task-001        │
│  명령어를 입력하세요.             │       Read Task 완료  │
│                                   │                       │
│  🤖 Agent (Planner)               │  [✅] task-002        │
│  ▼ 사고 과정 (Reasoning)          │       Claim Task 완료 │
│    1. 사용자 요청 분석 중...      │                       │
│    2. 계획 수립 중...             │  [↻] task-003 진행중  │
│                                   │       Wiki Search...  │
│  [ 🔍 위키 검색 중... ] ← 진행   │                       │
│                                   │                       │
├───────────────────────────────────┴───────────────────────┤
│  메시지를 입력하세요...                              [전송] │
└───────────────────────────────────────────────────────────┘
```

#### 채팅 입력

- 하단 입력창에 명령 또는 질문을 입력하고 **Enter** 키 또는 **전송 버튼**을 클릭합니다.
- 에이전트가 응답 중일 때는 스트리밍 방식으로 텍스트가 실시간 출력됩니다.

#### 사고 과정 아코디언 (`▼ 사고 과정`)

- 에이전트의 내부 추론 단계가 접기/펴기 형태로 표시됩니다.
- 제목 행을 클릭하면 상세 추론 과정을 펼쳐볼 수 있습니다.
- 최종 답변에 집중하려면 접어두면 됩니다.

#### 도구 호출 배지 (`[ 🔍 도구명 실행 중... ]`)

- 에이전트가 `read-task`, `wiki query` 등의 도구를 호출할 때 인라인으로 표시됩니다.
- 완료 시 `[ ✅ 완료 ]` 상태로 자동 전환됩니다.

#### 스트림 완료/실패 표시

- 응답이 정상 종료되면 결과 끝에 `Stream finished.`가 표시됩니다.
- 실행이 실패한 경우(상태이력 `[❌] 실패`)에는 `Stream finished.` 대신 **`❌ 실행 실패: <실패 원인>`** 형태로 실제 실패 원인이 표시됩니다.
  - 사용량 초과, 인증 만료, `turn.failed` 등 CLI가 전달한 오류 메시지가 그대로 노출됩니다.
  - 원인을 특정할 수 없을 때는 `알 수 없는 오류로 실행이 실패했습니다.` 안내가 표시됩니다.

### 4.2 페르소나 선택 (PersonaSwitcher)

화면 **상단 헤더 우측**의 페르소나 드롭다운으로 에이전트의 역할 프롬프트(`prompts/{role}.md`)를 전환할 수 있습니다. 동일한 질문이라도 선택한 페르소나에 따라 응답 관점과 산출물 형식이 달라집니다.

```
┌───────────────────────────────────────────────────────────┐
│  OMX Desktop Agent                      [ 🏛️ PI Consultant ▼ ] │
└───────────────────────────────────────────────────────────┘
```

#### 제공 페르소나

| 아이콘 | 페르소나 | 역할 ID | 설명 |
|--------|----------|---------|------|
| 🤖 | 기본 봇 | `default` | 범용 어시스턴트(프롬프트 미적용) |
| 🏛️ | PI Consultant | `pi-architect` | PI 컨설팅 PL & 애플리케이션 아키텍트 |
| 📋 | Code Reviewer | `code-reviewer` | 코드 리뷰 전문가 |
| 🧭 | Architect | `architect` | 시스템 설계/진단 (읽기 전용) |

#### 사용 방법

1. 헤더의 페르소나 버튼을 클릭해 드롭다운을 엽니다.
2. 원하는 페르소나를 선택하면 즉시 적용됩니다(다음 메시지부터 반영).
3. **PI Consultant** 선택 시, 아키텍처 결정 기록(ADR)·Gap 분석·As-Is 분석 등 PI 컨설팅 산출물 작성에 최적화된 응답을 받을 수 있습니다.

> 페르소나는 메시지 전송 시점의 선택 값이 적용됩니다. `기본 봇`을 선택하면 별도 역할 프롬프트 없이 동작합니다.

### 4.3 모델 선택 및 Gemini 사용법

채팅 입력창 **하단 툴바**의 모델 선택기에서 사용할 LLM 모델을 전환합니다. 클릭하면 위쪽으로 목록이 팝업됩니다.

```
┌───────────────────────────────────────────────────────────┐
│  메시지 입력…                                              │
├───────────────────────────────────────────────────────────┤
│  [ 📎 ]  [ 📄 템플릿 ▾ ]            [ 🧠 Gemini 2.5 Pro ▾ ] │
└───────────────────────────────────────────────────────────┘
```

#### 모델 그룹

| 그룹 | 모델 | 특징 |
|------|------|------|
| **테스트** | Echo, Echo Reverse | 로컬 처리(네트워크 불필요). 입력을 그대로/역순으로 반환하여 UI·스트림 동작 검증용 |
| **표준** | GPT-4o, GPT-4o mini, o3, o4-mini, Claude Sonnet 4.5 | OpenAI / Anthropic 모델 |
| **Gemini** | Gemini 2.5 Pro, 2.5 Flash, 1.5 Pro | Google Gemini 모델(별도 API 키 필요) |

#### Gemini API 키 설정

Gemini 모델은 Google API 키가 필요합니다. 키가 설정되지 않은 상태에서 Gemini 모델을 선택하면 채팅 영역 상단에 **인라인 키 설정 패널**이 나타납니다.

```
┌───────────────────────────────────────────────────────────┐
│  🔑 Gemini API 키 설정                                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ AIzaSy...                                           │  │
│  └─────────────────────────────────────────────────────┘  │
│                              [ 저장 ]   [ 삭제 ]          │
└───────────────────────────────────────────────────────────┘
```

1. [Google AI Studio](https://aistudio.google.com/)에서 발급받은 API 키(`AIzaSy`로 시작, 39자 이상)를 입력합니다.
2. **저장** 버튼을 클릭하면 키가 OS 키체인(`safeStorage`)에 **암호화 저장**되며, 입력란의 원문은 즉시 비워집니다.
3. 저장이 완료되면 Gemini 모델이 활성화되고 키 설정 패널은 사라집니다.
4. 키를 제거하려면 **삭제** 버튼을 클릭합니다.

> **보안**: API 키는 평문으로 디스크에 저장되지 않으며, OS 수준의 보안 저장소에 암호화되어 보관됩니다. 키 형식이 올바르지 않으면 저장이 거부됩니다.

### 4.4 산출물 템플릿 퀵메뉴 (TemplateQuickMenu)

채팅 입력창 하단 툴바의 **`📄 템플릿`** 버튼을 클릭하면, PI 컨설팅 표준 산출물 작성 프롬프트를 입력창에 바로 주입할 수 있습니다.

#### 제공 템플릿

| 템플릿 | 용도 |
|--------|------|
| **ADR (아키텍처 결정)** | 결정/배경/대안/근거/결과를 포함한 아키텍처 결정 기록 작성 |
| **Gap 분석** | As-Is 대비 To-Be 갭 분석 보고서 작성 |
| **WBS** | 마일스톤/태스크/일정 기반 프로젝트 WBS 작성 |
| **As-Is L6 분석/설계** | Level 6(업무규칙/CRUD/인터페이스) 분석·설계 산출물 작성 |
| **공수 산정** | 화면-서비스 호출관계·난이도·개발공수(M/M) 산정 산출물 작성 |

#### 사용 방법

1. 입력창 하단의 `📄 템플릿` 버튼을 클릭합니다.
2. 작성할 산출물 종류를 선택하면, 해당 템플릿(`templates/pi-consulting/*.md`) 기반 작성 지시문이 입력창에 자동 입력됩니다.
3. 필요 시 내용을 보완한 뒤 전송합니다.

> 템플릿 퀵메뉴는 **PI Consultant 페르소나**와 함께 사용할 때 가장 효과적입니다.

### 4.5 컨텍스트 파일 첨부 (PDF 텍스트 추출 포함)

채팅 입력창 하단의 **`📎`(첨부)** 버튼으로 파일을 첨부하면, 파일 내용이 컨텍스트로 추출되어 메시지 앞에 함께 전송됩니다.

#### 지원 형식

| 분류 | 형식 | 처리 방식 |
|------|------|-----------|
| 텍스트 | `.md`, `.txt`, `.json`, `.yaml/.yml`, `.ts/.tsx`, `.js/.jsx`, `.py`, `.rs`, `.go`, `.java`, `.cs`, `.html`, `.css`, `.sql`, `.xml` | 본문을 직접 읽어 컨텍스트로 추가 |
| **PDF** | `.pdf` | **페이지별 텍스트를 자동 추출**(pdfjs 기반)하여 컨텍스트로 추가 |
| 기타 바이너리 | 위 외 형식 | 추출하지 않고 안내 문구만 표시 |

#### 사용 방법

1. `📎` 버튼을 클릭해 파일을 선택합니다(여러 개 동시 선택 가능).
2. PDF는 페이지 순서대로 텍스트가 추출되어 첨부 칩으로 표시됩니다.
3. 첨부 칩의 `✕`를 눌러 제거할 수 있습니다.

> **참고**
> - 첨부 내용은 길이 제한(약 4,000자)이 적용되며, 초과 시 `일부만 포함됨` 표시와 함께 잘려서 전송됩니다.
> - **스캔/이미지 PDF**(글자가 이미지인 경우)는 추출 가능한 텍스트가 없어 `추출 가능한 텍스트가 없습니다` 안내가 표시됩니다(OCR 미지원).
> - 추출 중 오류가 발생하면 실패 사유가 첨부 칩에 표시됩니다.

### 4.6 에이전트 타임라인 사이드바


화면 우측에 위치한 사이드바로, 현재 진행 중인 태스크 수명주기를 실시간으로 추적합니다.

#### 태스크 상태 아이콘

| 아이콘 | 상태 | 설명 |
|--------|------|------|
| `[✅]` | 완료 | 태스크 정상 완료 |
| `[↻]` | 진행 중 | 현재 실행 중인 태스크 |
| `[⚠️]` | 경고 | 사용자 확인 필요 |
| `[❌]` | 실패 | 태스크 실행 오류 |
| `[⏸]` | 차단됨 | 선행 조건 미충족으로 대기 |

#### 수명주기 대시보드

- 상단에 **System Health** 지표(Drift Rate, 활성 태스크 수)가 표시됩니다.
- 각 태스크를 클릭하면 상세 정보 패널이 열립니다.

### 4.7 차단형 인터뷰 모드 (Interlude)

에이전트가 중요한 판단을 사용자에게 요청할 때 활성화됩니다.

```
┌───────────────────────────────────────────────────────────┐
│  (채팅 영역이 어두워지며 비활성화됨)                       │
│                                                           │
├───────────────────────────────────────────────────────────┤
│  🚨 [동결 모드] 사용자 판단 및 승인 대기 중               │
├───────────────────────────────────────────────────────────┤
│  🤖 [Verifier]                                            │
│  "프로덕션 적용 전, dry-run 모드로 먼저 테스트하시겠습니까?" │
│                                                           │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  추가 맥락을 입력하세요... (선택 사항)              │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                           │
│     [ ❌ 취소 및 롤백 ]          [ ✅ 승인 및 계속 ]      │
└───────────────────────────────────────────────────────────┘
```

#### 사용 방법

1. 에이전트의 질문 내용을 확인합니다.
2. 필요하면 **추가 맥락 입력란**에 보충 설명을 입력합니다.
3. **승인 및 계속 (Enter)**: 에이전트가 다음 단계를 진행합니다.
4. **취소 및 롤백**: 현재 작업을 중단하고 이전 상태로 되돌립니다.

> 인터뷰 모드에서는 다른 채팅 입력이 비활성화됩니다. 반드시 승인 또는 취소로 응답해야 이후 대화가 가능합니다.

### 4.8 지연 스킬 알림 토스트 (Deferred Skills)

`team + ralph` 등 복수 스킬이 동시에 요청될 때 서브 스킬이 지연된 경우, 화면 **우측 하단**에 플로팅 토스트로 표시됩니다.

```
                           ┌──────────────────────────────┐
                           │  ⏳ 지연된 스킬 대기 중 (1)  │
                           │     • ralph                  │
                           └──────────────────────────────┘
```

#### 동작 방식

- 주 스킬 실행이 완료되면 지연된 스킬이 자동으로 활성화됩니다.
- 토스트는 스킬이 활성화되는 순간 자동으로 사라집니다.
- 여러 스킬이 지연된 경우 숫자가 누적되어 표시됩니다(예: `대기 중 (3)`).

### 4.9 상태 충돌 경고 배너 (Rollback Warning)

에이전트가 `in_progress` 상태에서 `planning` 상태로 불법 롤백이 시도될 때 화면 **최상단**에 인라인 배너가 나타납니다.

```
┌───────────────────────────────────────────────────────────┐
│  ⚠️  상태 충돌이 감지되었습니다.  [ 🧹 상태 초기화 ]      │
└───────────────────────────────────────────────────────────┘
```

#### 대처 방법

1. **상태 초기화 버튼 클릭**: `omx state clear` 명령이 실행되어 상태가 초기화됩니다.
2. 배너가 사라진 후 다시 원하는 명령을 입력하면 정상 동작합니다.

> 상태 초기화 시 현재 진행 중인 태스크의 중간 결과는 초기화될 수 있습니다. 필요한 내용은 미리 메모해두세요.

---

### 4.10 정합성 위기 배너 (Drift Critical)

플러그인 로그와 매니페스트 원본 간의 불일치율(Drift Rate)이 **10% 이상**으로 감지되면 상단에 경고 배너가 표시됩니다.

```
┌───────────────────────────────────────────────────────────┐
│  🔴  설정 정합성 위기 감지됨.  [ 🔄 환경 재동기화 ]  [ 상세 보기 ] │
└───────────────────────────────────────────────────────────┘
```

#### 대처 방법

| 버튼 | 동작 |
|------|------|
| **환경 재동기화** | `omx setup`을 재실행하여 로컬 환경을 매니페스트 원본과 동기화 |
| **상세 보기** | 어떤 플러그인이 미등록 상태인지 목록을 팝업으로 표시 |

---

### 4.11 에러 복구 가이드 오버레이 (Error Guide)

시스템 예외 발생 시 화면 중앙에 단계별 복구 가이드 팝업이 나타납니다.

```
┌────────────────────────────────────┐
│  ❌  에러 코드: DRIFT_CRITICAL     │
├────────────────────────────────────┤
│  설정 정합성 위기 — 복구 방법:    │
│                                    │
│  1. 실행 중인 에이전트를 종료합니다. │
│  2. omx setup 명령을 실행합니다.  │
│  3. omx doctor로 환경을 재확인합니다. │
│  4. 앱을 재시작합니다.            │
│                                    │
│  [ 🔄 환경 재동기화 ]  [ 상태 초기화 ] │
│                              [✕ 닫기] │
└────────────────────────────────────┘
```

#### 에러 코드별 의미

| 에러 코드 | 의미 | 주요 조치 |
|-----------|------|-----------|
| `DRIFT_CRITICAL` | 설정 정합성 위기 (Drift Rate ≥ 10%) | 환경 재동기화 (`omx setup`) |
| `STATE_ROLLBACK` | 금지된 상태 롤백 감지 | 상태 초기화 (`omx state clear`) |
| `CLI_UNREACHABLE` | OMX CLI 응답 없음 | CLI 설치 확인, `omx doctor` 재실행 |
| `ADAPTER_DEGRADED` | 외부 어댑터 응답 저하 | 어댑터 서비스 상태 확인 |
| `UNKNOWN` | 알 수 없는 예외 | 앱 재시작 후 지속 시 로그 확인 |

#### 오버레이 닫기

- **✕ 닫기 버튼** 클릭 또는 **배경 영역 클릭**으로 닫을 수 있습니다.
- 오버레이를 닫아도 문제가 해결되지 않으면, 가이드의 조치 단계를 순서대로 수행하세요.

### C. 문제 해결 빠른 참조

| 증상 | 원인 | 해결 |
|------|------|------|
| 앱 시작 시 환경 오류 배너 | OMX CLI 미설치 또는 인증 만료 | `omx doctor` 실행 후 재시작 |
| 채팅 응답이 없음 | CLI 프로세스 응답 없음 | 앱 재시작 또는 `omx doctor` 확인 |
| Drift Critical 배너 | 미등록 플러그인 호출 증가 | **환경 재동기화** 버튼 클릭 |
| 상태 충돌 배너 반복 | 상태 파일 오염 | **상태 초기화** 후 재시작 |
| 인터뷰 모드 잠김 | 네트워크 지연으로 응답 지연 | 잠시 대기 후 취소 버튼 클릭 |
| 결과 끝에 `❌ 실행 실패` 표시 | CLI 실행 오류(사용량 초과, 인증 만료 등) | 표시된 실패 원인 확인 후 조치, `codex login status`로 인증 점검 |
| Gemini 모델 선택 불가 | API 키 미설정 | 키 설정 패널에서 `AIzaSy...` 키 저장 |
| Gemini 키 저장 거부 | 키 형식 오류 | `AIzaSy`로 시작하는 39자 이상 키 입력 |
| PDF 첨부 시 `추출 가능한 텍스트가 없습니다` | 스캔/이미지 PDF | 텍스트형 PDF 사용(OCR 미지원) |
