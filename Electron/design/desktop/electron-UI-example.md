클로드 데스크탑(Claude Desktop)의 세련되고 미니멀한 UX 철학을 반영하여, 앞서 정의한 티켓들의 요구사항이 실제 화면에서 어떻게 렌더링되어야 하는지 **구조적 와이어프레임(Wireframe) 모식도**로 구현해 드립니다.

프론트엔드(React/Vue 등) 개발팀이 이 텍스트 기반의 UI 설계도를 바탕으로 실제 컴포넌트를 분리하고 디자인 시스템을 입힐 수 있습니다.

---

### 🎨 1. 기본 채팅 및 도구 실행 뷰 (EL-213, EL-219, EL-221 반영)

**핵심 포인트**: 에이전트의 사고 과정(Thinking)은 아코디언으로 숨기고, 도구 호출(Tool Call)은 인라인 애니메이션 배지로 처리하며, 전체 상태는 우측 사이드바로 분리합니다.

```text
+---------------------------------------------------+---------------------------+
| 💬 메인 대화 (Chat Stream)                        | 📋 에이전트 타임라인 (HUD)|
+---------------------------------------------------+---------------------------+
| 👤 User                                           |                           |
| 현재 아키텍처 문서를 분석하고 요약해줘.           | [✅] task-001             |
|                                                   |      Read Task 완료       |
| 🤖 Agent (Planner)                                |                           |
| +-----------------------------------------------+ | [✅] task-002             |
| | ▼ 사고 과정 (Reasoning)                       | |      Claim Task 완료      |
| |   1. 사용자가 아키텍처 문서 요약을 요청함.    | |  💡 [Note: 충돌 없음]     |
| |   2. omx_wiki 디렉터리를 먼저 검색해야 함.    | |                           |
| |   3. wiki query 도구를 호출할 계획.           | | [↻] task-003 (In Prog)  |
| +-----------------------------------------------+ |      Wiki Search...       |
|                                                   |                           |
|  [ 🔍 프로젝트 지식베이스(Wiki) 검색 중... ] ⬅️ 인라인 프로그레스 (Pulsing) |
|                                                   |                           |
| 문서를 확인했습니다. 현재 아키텍처는...           |                           |
|                                                   |                           |
+---------------------------------------------------+---------------------------+
| Type your message...                              |                      [↑]  |
+---------------------------------------------------+---------------------------+

```

* **컴포넌트 설계 가이드**:
* `<ReasoningAccordion />`: `subType: "reasoning"` 토큰이 들어오면 렌더링되는 접기/펴기 영역.
* `<ToolCallBadge />`: `type: "tool_call"` 이벤트 수신 시 렌더링. 작업이 완료되면 `[ ✅ 위키 검색 완료 ]` 로 부드럽게 트랜지션.

### 🚨 2. 차단형 인터뷰 모드 (EL-215, EL-216 반영)

**핵심 포인트**: 에이전트가 판단을 요구할 때(`askuserQuestion` 또는 `needs-input`), 기존 채팅창은 비활성화(Dim)되고 사용자의 시선을 집중시키는 전용 모달 폼이 최하단에 올라옵니다.

```text
+-------------------------------------------------------------------------------+
| 👤 User: 환경 설정 파일을 프로덕션 기준으로 업데이트 해줘.                    |
|                                                                               |
| 🤖 Agent (Executor)                                                           |
| (채팅 내역 영역은 시각적으로 어두워짐 - Dim & Disabled)                       |
|                                                                               |
|                                                                               |
+-------------------------------------------------------------------------------+
| 🚨 [동결 모드] 사용자 판단 및 승인 대기 중                                    |
+-------------------------------------------------------------------------------+
| 🤖 [Verifier]                                                                 |
| "프로덕션 적용 전, 안전을 위해 'dry-run' 모드로 먼저 테스트하시겠습니까?"     |
|                                                                               |
|  +-------------------------------------------------------------------------+  |
|  | (선택) 에이전트에게 전달할 추가 맥락을 입력하세요...                    |  |
|  +-------------------------------------------------------------------------+  |
|                                                                               |
|       [ ❌ 작업 취소 및 롤백 ]             [ ✅ 승인 및 계속 (Enter) ]        |
+-------------------------------------------------------------------------------+

```

* **컴포넌트 설계 가이드**:
* `<ChatContainer dim={isInterlude} />`: 인터류드 상태일 때 포커스를 잃게 함.
* `<InterludeActionPanel />`: `wakeable_only=true` 신호가 들어왔을 때 하단에 고정되는 원자성 락 뷰(Lock View). 답변을 Submit 하면 락이 풀릴 때까지 '처리 중...' 스피너 렌더링.

### ⚠️ 3. 복합 상태 예외 처리 및 지연 스킬 알림 (EL-224, EL-225 반영)

**핵심 포인트**: 다중 스킬이 호출되었을 때 자동 실행을 막는 '지연(Deferred) 토스트'와 불법적인 롤백 시도시 나타나는 '액션 배너'입니다.

```text
+-------------------------------------------------------------------------------+
| ⚠️ 상태 충돌 감지됨: 지원되지 않는 롤백입니다. [ 🧹 상태 초기화(Clear) ]    | ⬅️ 상단 배너
+-------------------------------------------------------------------------------+
| 💬 메인 대화 (Chat Stream)                                                    |
|                                                                               |
| 👤 User: team 명령과 ralph 명령을 동시에 실행해줘.                            |
|                                                                               |
| 🤖 Agent (Orchestrator)                                                       |
| Team 오케스트레이션 구동을 시작합니다.                                        |
|                                                                               |
|                                                                               |
|                                             +-------------------------------+ |
|                                             | ⏳ 지연된 스킬 대기 중 (1)    | | ⬅️ 플로팅 토스트
|                                             |    • ralph                    | |
|                                             +-------------------------------+ |
+-------------------------------------------------------------------------------+
| Type your message...                                                          |
+-------------------------------------------------------------------------------+

```

* **컴포넌트 설계 가이드**:
* `<GlobalAlertBanner />`: `DriftRate`가 높거나 금지된 상태 전이가 발생할 때 상단에 표시. `omx state clear` 또는 `omx setup`을 백그라운드에서 실행하는 복구 버튼(Action Button) 포함.
* `<DeferredSkillToast />`: `team + ralph` 같은 중복 명령 파싱 시, 메인 실행을 방해하지 않게 우측 하단에 조용히 띄워두는 플로팅 알림 뷰.

### 👩‍💻 개발팀 적용 가이드

프론트엔드 팀은 이 레이아웃을 바탕으로 다음과 같이 React/Vue 컴포넌트 트리를 구성하면 됩니다.

1. 상태(State)는 철저히 IPC에서 보내주는 `omx:lifecycle-change` 이벤트 페이로드(`{ status: 'in_progress', interlude: false, deferred: ['ralph'] }`)에 전적으로 의존합니다.
2. 프론트엔드 단에서 타이머를 돌리거나 상태를 추측하여 UI를 변경하지 않습니다. (진실의 경계 원칙 고수)
3. 애니메이션 라이브러리(Framer Motion 등)를 사용하여 아코디언이 열리거나 인라인 프로그레스 배지가 나타날 때 클로드처럼 부드러운 전환(Transition) 효과를 주면 완성도가 극대화됩니다.
