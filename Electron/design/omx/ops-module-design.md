# OMX Ops 모듈 개별 설계서

## 0. 문서 정보
- 모듈: ops
- 기준 분석 문서: Electron/analysis/ops-module-analysis.md
- 상태: Draft v0.1
- 작성일: 2026-05-29
- 문서 목적: 운영 진단과 드리프트 점검 계층의 읽기 전용 분석 계약을 정의
- 용어 기준: 한글 우선 표기, 최초 1회 영문 병기
- 업데이트 기준: drift detector 입력 원천, 결과 구조체, 진단 기준 버전 변경 시 2, 3, 4, 7장을 우선 갱신

## 1. 개요와 주요기능
- 개요: ops 모듈은 실행 로그와 정책 기준 사이의 차이를 읽기 전용으로 탐지해 운영 이상 징후를 요약 가능한 구조체로 제공하는 진단 계층이다.
- 주요기능:
  - 드리프트 탐지: .omx/logs와 manifest 기반으로 운영 상태와 기준 차이를 분석
  - 진단 요약 제공: UI/IPC가 소비 가능한 구조화 결과 반환
  - graceful degradation: 로그 누락, 파손, 권한 문제 시 진단 가능 범위를 유지
  - 기준 버전 관리: 진단 정책 버전 차이를 결과에 반영 가능하도록 설계

## 2. 책임과 경계
- 책임:
  - 운영 로그와 manifest를 읽어 정책 드리프트 여부를 판단하는 읽기 전용 분석 수행
  - 누락/파손 상황에서도 부분 결과와 경고를 반환
  - 상위 계층이 표시하거나 브로드캐스트하기 쉬운 구조체를 제공
- 비책임:
  - 운영 상태 수정, 로그 복구, 정책 파일 자동 갱신은 담당하지 않는다.
- 경계:
  - ops는 관측 및 진단 계층이며, remediation 실행은 다른 운영 워크플로가 담당한다.

## 3. 외부/내부 인터페이스
- 외부 인터페이스:
  - drift-detector.ts가 제공하는 drift 분석 함수 집합
- 내부 인터페이스:
  - .omx/logs 읽기
  - manifest 메타데이터 읽기
  - 정책 기준 버전 비교 유틸리티
- 호출자:
  - main/ipc 운영 대시보드, 진단 화면, 자동 점검 트리거

## 4. 데이터 구조와 계약
- 입력 계약:
  - 로그 디렉터리와 manifest는 읽기 전용 입력으로 취급
  - 누락된 파일은 예외보다 경고/부분 결과로 처리 가능해야 한다.
- 출력 계약:
  - 결과는 UI/IPC가 즉시 소비 가능한 요약 구조체를 반환
  - 드리프트 여부, 원인 요약, 경고 목록, 기준 버전 정보를 포함하는 방향으로 유지
- 계약 원칙:
  - ops 결과는 분석 시점의 스냅샷이며 상태 변경을 유도하지 않는다.
  - 경로/권한 오류는 치명적 실패보다 진단 불완전 경고로 표현한다.

## 5. 상태 전이와 불변식
- 진단 상태 전이:
  - input scan -> analyze -> summarize -> reported
  - source missing/corrupt 시 degraded analyze 경로 허용
- 불변식:
  - ops 계층은 입력 파일을 수정하지 않는다.
  - 동일 기준 버전에서 동일 입력이면 결정적 결과를 반환해야 한다.
  - 로그 일부 누락이 전체 진단 중단의 기본 원인이 되어서는 안 된다.

## 6. 핵심 시퀀스
- 드리프트 탐지 시퀀스:
  1. drift detector가 manifest와 로그 입력 경로를 수집
  2. 정책 기준 버전과 실제 로그/실행 흔적을 비교
  3. 차이점과 이상 징후를 요약 구조체로 정리
  4. UI/IPC 또는 운영 대시보드가 결과를 소비
- graceful degradation 시퀀스:
  1. 로그 누락/권한 오류를 감지
  2. 이용 가능한 입력만으로 부분 분석 수행
  3. source warning 또는 degraded 상태를 결과에 포함

## 7. 오류 처리 및 복구
- 로그 누락:
  - missing source로 표시하고 가능한 범위의 분석만 수행
- 로그 파손:
  - corrupt warning을 남기고 해당 소스만 제외
- 경로/권한 오류:
  - 접근 불가 사실을 진단 결과에 포함하고 상위에 복구 힌트 제공
- 기준 버전 불일치:
  - 결과에 버전 차이를 명시해 해석 오차를 줄임

## 8. 보안/성능 고려
- 보안:
  - ops는 읽기 전용 분석만 허용해 운영 상태 변경 위험을 차단한다.
  - 로그/manifest 경로 접근은 허용된 workspace 범위로 제한해야 한다.
- 성능:
  - 요약 가능한 구조체를 우선 생성해 UI/IPC 비용을 낮춘다.
  - 대용량 로그는 필요한 범위만 읽도록 후속 세분화가 필요하다.

## 9. 테스트 케이스 맵
- 단위:
  - drift 판단 로직
  - 기준 버전 비교
  - 요약 구조체 생성
- 통합:
  - 로그 + manifest 입력 조합별 진단 결과
  - 누락/파손 입력의 graceful degradation
- 회귀:
  - 읽기 전용 보장
  - 경로/권한 오류 시 전체 실패 대신 부분 결과 반환

## 10. 오픈 이슈
- drift-detector의 실제 결과 스키마를 별도 canonical 문서로 고정할 필요가 있음
- 기준 버전 관리 위치와 배포 전략을 정의할 필요가 있음
- ops 진단 결과를 sidecar 또는 renderer UI에 어떻게 노출할지 연결 정책 정리가 필요
