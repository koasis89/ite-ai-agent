## 1. 문서 개요

### 1.1 작성 목적
이 문서는 OMX 전체를 처음 보는 개발자가 “무엇이 어디에 있고, 어떤 순서로 호출되며, 어디까지가 공개 계약인지”를 빠르게 이해하도록 돕는 공식 상세 설계서이다. 특히 초보 개발자가 자주 헷갈리는 용어를 많이 풀어 설명하고, 모듈별 책임과 공개 API를 연결해서 읽을 수 있게 구성한다.

### 1.2 범위(In Scope)
- Electron Main/Renderer/Preload 구조
- OMX CLI, hooks, runtime, state, IPC, logs, hud, team, workflow, agent, adapter 계층
- .omx/state 기반 상태 파일과 session-scoped 상태
- 공개 API, 이벤트 채널, CLI 커맨드, 주요 데이터 계약
- 보안, 성능, 테스트, 마이그레이션, 변경 관리
- 모듈별 상세 설계 13장 전 항목

### 1.3 비범위(Out of Scope)
- 내부 구현 코드의 라인별 상세 설명
- 특정 버그 수정 이력의 재현
- 외부 서비스의 운영 정책 원문 전체
- 각 모듈 내부 helper 함수의 세부 구현
- UI 픽셀 단위 스타일링 세부 사항

### 1.4 독자 대상
- OMX를 처음 접한 신규 개발자
- Electron, IPC, CLI, 상태 파일 구조를 빠르게 파악해야 하는 개발자
- 기능 추가나 리팩터링 전에 호출 관계를 확인하려는 개발자
- 문서를 기준으로 API 변경 영향을 검토하는 리뷰어

| Planner | 계획 수립 담당 에이전트 역할 | 요구사항 정리, 단계 설계, 리스크 식별 | 먼저 길을 그리는 역할 | `agents`, `planning` |
| Executor | 구현/수정 담당 에이전트 역할 | 코드/문서 실제 변경 수행 | 실제 손을 움직이는 역할 | `agents`, 실행 전 모듈 |
| Verifier | 검증 담당 에이전트 역할 | 테스트/정합성/완료 판정 | "끝났다"를 증거로 확인하는 역할 | `agents`, `test`, `logs` |
| 자가 적응형 기억 엔진 | 턴 결과와 사용자 피드백을 분석해 프로젝트 특화 규칙/스택을 학습하는 계층 | 세션 간 지식 누적, 문맥 재사용, 운영 규칙 적응 | 사람의 해마처럼 경험을 축적해 다음 턴 품질을 높임 | `src/adapt/`, `wiki`, `state` |
| Autopilot Mode | 복수 에이전트의 순차/병렬 협업 루프를 제어하며 목표 달성까지 단계를 갱신하는 모드 | 장기 과제 자동 실행, 단계형 파이프라인 운영 | "자동 주행"처럼 단계 전개를 시스템이 관리 | `pipeline`, `team`, `goal-workflows` |
| Ralph Mode | 기술적 blocking 전까지 사용자 개입 없이 연속 최적화 루프를 수행하는 모드 | 지속 실행, 검증, 수정 루프 | "멈추지 말고 계속" 원칙의 단일 실행 루프 | `ralph`, `runtime` |
| Deep-Interview Mode | 입력 모호성이 높을 때 심층 질의 라운드를 강제하는 모드 | 요구사항 명확화, 오판 방지, 승인 오남용 차단 | 애매하면 구현 전에 반드시 질문으로 명확화 | `question`, `hooks`, `planning` |
| input_lock | Deep-Interview 중 성급한 승인 문자를 일시 차단하는 필드 | 인터뷰 라운드 강제 유지 | `yes/y/proceed`로 조기 종료되는 실수를 막는 안전장치 | `question`, `interlude` |
| Blocking | 기술적으로 현재 레인을 계속 진행할 수 없는 상태 | Ralph/Team 루프 중단 및 재계획 트리거 | 단순 지연이 아니라 "진행 불가" 상태로 이해 | `runtime`, `team`, `ralph` |
| Hook | 런타임 이벤트 시 자동 실행되는 트리거 | 입력/도구/종료 시점 자동화 | 이벤트 기반 자동 스크립트 갈고리 | `hooks`, `scripts` |
| Hook Event | 훅이 발행하는 표준 이벤트 레코드 | 로그/알림/상태 반영의 공통 입력 | 시스템 활동의 공통 로그 단위 | `hooks`, `logs`, `notifications` |
| MCP | 도구/서버/상태를 표준 연결하는 프로토콜 계층 | 외부 도구 서버 연동, stdio 서버 관리 | "플러그인 표준 포트"처럼 이해 | `mcp`, 외부 MCP 서버 |
| ACP | 외부 에이전트 런타임 증거를 읽고 브리지하는 통합 표면 | adapt/hermes 연동 관측 | 외부 에이전트 세계와 OMX를 잇는 제어 평면 | `adapt`, `openclaw` |
| Exclusive File Locking | 동시 수정 충돌을 막기 위한 파일 시스템 기반 배타 락 전략 | AGENTS.md 같은 단일 중요 파일 보호 | 동시에 여러 프로세스가 수정해 깨지는 문제를 예방 | `utils`, `hooks`, 파일 락 구현부 |
| `agents-md.lock/` | AGENTS.md 수정 시 생성되는 배타 락 디렉터리 | 락 소유권 표시 및 충돌 차단 | "지금 내가 수정 중" 표식 디렉터리 | 락 디렉터리 경로 |
| Zombie Lock(좀비 락) | 프로세스가 죽었지만 락만 남아있는 고아 락 상태 | 영구 대기/교착 방지 대상 | 락이 안 풀려 모두 멈추는 상황을 의미 | 락 감시 루틴 |
| PID 생존 감지 (`process.kill(pid, 0)`) | 락 소유 PID가 살아있는지 시그널 0으로 확인하는 점검 방식 | 좀비 락 자동 회수 판단 기준 | 프로세스가 실제 살아있는지 안전하게 확인하는 표준 트릭 | 락 회수 로직 |
| Lock 회수/파기 | 비정상 사멸 소유자의 락을 정리하는 복구 동작 | 장기 교착 해소, 자동 복구 | 죽은 소유자의 잠금을 시스템이 대신 치운다 | 락 관리 유틸 |
| Team | tmux 기반 멀티에이전트 실행 모드 | 리더-워커, 태스크 큐, 메일박스 운영 | 여러 작업자를 동시에 운용하는 팀 실행 | `team`, `sidecar` |
| Team Worker | Team 모드에서 실제 작업을 수행하는 실행 주체 | inbox/task/status 갱신 및 결과 보고 | 팀원 프로세스로 이해 | `team` |
| Team Dispatch | 리더가 워커에게 작업을 배포하는 전달 계층 | 병렬 작업 분배, 우선순위 전달 | "업무 배정" 기능 | `team` |
| Team Mailbox | 리더/워커 메시지 교환 채널 | 상태 보고, 질의/응답, 제어 메시지 | 팀 내부 메신저 | `team` |
| State | 현재 실행 상태를 파일/메모리로 표현한 값 | 모드 동기화, HUD/UI 반영, 재개 근거 | 시스템의 현재 위치를 기록하는 지도 | `state`, `.omx/state` |
| Session Scope | 세션 ID 기준으로 격리된 상태/로그/질문 범위 | 세션 간 충돌 방지, 재현성 확보 | 작업 단위를 폴더로 분리한다고 이해 | `state`, `logs`, `question` |
| Ledger | 상태 변화를 시간 순으로 남기는 append-only 기록 | 감사 추적, 원인 분석, 완료 증거 | "히스토리 장부" | `goal-workflows`, `performance-goal`, `ralph` |
| Snapshot | 특정 시점 상태를 읽기 전용으로 묶은 값 | 비교/검증/표시 기준점 | 사진처럼 그 시점 상태를 고정 저장 | `runtime`, `sidecar`, `hud` |
| Atomic Write | 임시 파일 후 rename으로 교체하는 안전 쓰기 | 파일 손상/부분 쓰기 방지 | 저장 중 비정상 종료에도 파일 일관성 유지 | `state`, `utils` |
| Preflight | 실행 전 환경/조건 사전 점검 단계 | 실패 조기 발견, 불필요 실행 방지 | 출발 전 체크리스트 | `env`, `cli` |
| Graceful Degradation | 일부 실패에도 핵심 기능을 유지하는 전략 | 외부 장애 시 서비스 연속성 확보 | 전부 멈추지 않고 가능한 범위만 유지 | `runtime`, `notifications`, `openclaw` |
| Idempotent | 같은 작업을 반복해도 결과가 안정적인 성질 | 재시도 안전성, 장애 복구 품질 | 재실행해도 부작용이 작아야 함 | `state`, `services`, `scripts` |

### 2.3 전제 및 제약사항
- Electron Main은 권한이 큰 작업을 담당하므로 Renderer가 직접 파일 시스템이나 프로세스를 만지지 않도록 설계한다.
- 공개 API는 내부 helper보다 우선하여 문서화한다.
- 상태 파일은 가능한 한 원자적 쓰기와 검증을 거친다.
- 초보 개발자가 읽는 문서이므로, 약어는 처음 등장 시 풀어쓴다.
- 일부 모듈은 채널명이나 컴포넌트 이름이 공개 API 역할을 한다.

## 3. 시스템 컨텍스트

### 3.1 상위 아키텍처 개요
OMX는 크게 네 층으로 이해하면 쉽다.
- 실행 진입층: CLI, hooks, scripts, main
- 오케스트레이션층: pipeline, planning, team, question, goal-workflows, ralph, ralplan, performance-goal, autoresearch
- 관측/표현층: logs, hud, sidecar, notifications, renderer, components
- 기반/공통층: state, runtime, ipc, services, utils, env, mcp, catalog, auth, agents, adapt, openclaw, wiki, core

이 구조의 핵심은 “사용자 입력이 들어오면 CLI나 Renderer에서 시작되고, Main/IPC를 거쳐 상태와 워크플로로 연결되며, 결과는 다시 Renderer/HUD/로그로 반영된다”는 점이다.

### 3.2 실행 환경(Electron Main/Renderer/Preload)
- Main: 파일 읽기/쓰기, process spawn, IPC handler 등록, state synchronizing을 담당한다.
- Renderer: 채팅 화면, 대시보드, 상태 패널, 마크다운 렌더링을 담당한다.
- Preload: Renderer가 안전하게 호출할 수 있는 API만 노출한다.

초보자 관점에서 중요한 점은 다음과 같다.
- Renderer는 “화면”이다.
- Main은 “권한과 실행”이다.
- Preload는 “안전한 통로”다.

### 3.3 외부 연동 경계(OMX CLI, MCP, OpenClaw 등)
- OMX CLI: 터미널에서 실행하는 사용자 명령의 첫 진입점이다.
- MCP: 외부 서버를 표준 방식으로 띄우고 연결한다.
- OpenClaw: 알림/연동 게이트웨이로 외부 시스템에 메시지를 전달한다.
- Tmux: HUD, 팀 워커, 사이드카 같은 터미널 기반 운영 보조에 사용된다.
- Git/Codex/Filesystem: 상태 파일, 릴리스, 워크트리, 문서 생성에 자주 사용된다.

## 4. 논리 아키텍처 상세

### 4.1 모듈 계층 구조

OMX는 **두 개의 실행 세계**가 맞물려 동작한다. 하나는 Node.js 기반의 OMX CLI 런타임(`src/`)이고, 다른 하나는 Electron 데스크톱 앱(`Electron/src/`)이다. 아래 계층 구조는 이 두 세계를 함께 정리한 것이다.

#### 전체 계층 다이어그램

```
┌──────────────────────────────────────────────────────────────────┐
│                     [ 사용자 진입점 ]                              │
│  CLI (omx <cmd>)            Electron Renderer (채팅 UI / 대시보드) │
└────────────┬─────────────────────────────────┬───────────────────┘
             │                                 │ window.electronAPI (preload.ts)
             ▼                                 ▼
┌──────────────────────┐         ┌─────────────────────────────────┐
│  [진입/라우팅 계층]    │         │      [Electron Main 계층]        │
│  src/hooks/          │         │  Electron/src/main/             │
│  src/cli/            │         │    ├── ipc/   (9개 IPC 브리지)   │
│  src/scripts/        │         │    ├── core/  (앱 수명주기)       │
│  src/modes/          │         │    ├── cli/   (CLI 프록시)        │
└────────┬─────────────┘         │    ├── env/   (환경 점검)         │
         │                       │    ├── state/ (상태 파일 감시)     │
```

#### 계층 간 호출 제약

- **Renderer → Main**: `window.electronAPI`(preload 경유) 외 직접 접근 금지
- **Main → Renderer**: `ipcMain.handle` / `webContents.send` IPC 이벤트만 허용
- **상태 계층**: 모든 쓰기는 atomic write(`tmp → rename`) 원칙 준수
- **통합 계층**: 워크플로·런타임이 호출하는 방향만 허용, 역방향 강결합 금지
- **utils**: 가장 하위 공통 기반. 다른 계층을 import 하지 않는다.

### 4.2 모듈 간 의존성 규칙

모듈 간 의존성은 **단방향 흐름** 원칙을 따른다. 상위 계층이 하위 계층을 호출하는 방향은 허용하지만, 역방향 강결합(하위 → 상위 직접 import)은 금지한다. 이 규칙을 어기면 순환 의존성이 생겨 테스트와 유지보수가 어려워진다.

#### 계층 간 허용/금지 의존성 표

| 의존 방향 | 허용 여부 | 비고 |
|---|---|---|
| 진입/라우팅 → 워크플로·런타임 | ✅ 허용 | hooks/cli가 pipeline/team/ralph를 시작 |
| 진입/라우팅 → 상태/기반 | ✅ 허용 | 진입 시 상태 읽기 |
| 워크플로 → 실행/런타임 | ✅ 허용 | pipeline이 exec/subagents 호출 |
| 워크플로 → 상태/기반 | ✅ 허용 | 단계 전이 결과를 state에 기록 |
| 실행/런타임 → 상태/기반 | ✅ 허용 | runtime이 state/logs에 기록 |
| 통합/외부연동 → 상태/기반 | ✅ 허용 | adapt/auth가 상태 읽기 |
| UI/관측 → 상태/기반 | ✅ 읽기 전용 | hud/notifications가 상태를 읽어 표시 |
| Renderer → Electron Main | ✅ IPC 경유만 | `window.electronAPI`(preload) 외 직접 접근 금지 |
| Electron Main → Renderer | ✅ IPC 경유만 | `ipcMain.handle` / `webContents.send` 이벤트 발송 |
| 상태/기반 → 워크플로 | ❌ 금지 | state/utils가 pipeline/team을 직접 import 하면 안 됨 |
| 상태/기반 → UI/관측 | ❌ 금지 | state가 hud/notifications를 알면 안 됨 |
| utils → 다른 모든 계층 | ❌ 금지 | utils는 외부 의존 없는 순수 기반 |
| types → 다른 모든 계층 | ❌ 금지 | 타입만 정의, 로직 없음 |

#### 핵심 의존성 규칙 상세

**규칙 1 — Renderer ↔ Main 격리**
- Renderer(`Electron/src/renderer/`)는 Node.js API, 파일 시스템, 프로세스 실행에 직접 접근할 수 없다.
- 모든 요청은 `preload.ts`가 노출한 `window.electronAPI`를 통해 Main으로 위임한다.
- Main이 Renderer 내부 React 상태를 직접 수정하는 것도 금지. 반드시 IPC 이벤트로 전달한다.

**규칙 2 — 단방향 상태 흐름**
- 상태 쓰기는 워크플로·런타임 → `src/state/` 방향으로만 흐른다.
- UI/관측 계층은 `src/state/`를 **읽기 전용**으로 소비하고 직접 변경하지 않는다.
- Electron Main의 `state/`(상태 파일 감시)는 OMX 런타임 `src/state/` 파일 변경을 감시해 IPC로 Renderer에 중계하는 역할만 한다.

**규칙 3 — utils / types 순수 기반 원칙**
- `src/utils/`와 `src/types/`는 다른 OMX 모듈을 import 하지 않는다.
- 순환 의존성 방지의 최후 보루이므로, 유틸에 로직을 과도하게 넣지 않는다.

**규칙 4 — 통합 계층 단방향 호출**
- `src/adapt/`, `src/openclaw/`, `src/auth/`, `src/catalog/`, `src/agents/` 등 통합 계층은
  워크플로·런타임이 필요할 때 호출하는 방향만 허용한다.
- 통합 계층이 먼저 워크플로나 런타임 로직을 직접 호출하면 안 된다(역방향 강결합 금지).

**규칙 5 — hooks 확장점 원칙**
- `src/hooks/`는 OMX 런타임 이벤트에 반응하는 **진입점**이다.
- hooks가 워크플로를 시작하는 것은 허용하지만, 워크플로 내부에서 hooks를 직접 import 해 재진입하면 순환이 생긴다. 이벤트 발행(pub/sub) 방식으로 분리한다.

#### 의존성 위반 감지 방법
- TypeScript 컴파일러 path alias와 `tsconfig.json`을 이용해 레이어 간 import 검사
- `biome.json` lint 규칙으로 금지 패턴 탐지
- 새 모듈 추가 시 PR 리뷰에서 계층 위치 확인 필수

### 4.3 이벤트/명령 흐름 개요

OMX의 이벤트·명령 흐름은 **두 진입 경로**에서 시작해 같은 상태/관측 계층으로 수렴한다. 아래에서 각 경로를 상세히 설명한다.

#### 흐름 다이어그램

```
[ CLI 진입 경로 ]                        [ Electron Renderer 진입 경로 ]

사용자 터미널 입력                          채팅 입력 / 버튼 클릭
     │                                           │
     ▼                                           ▼
src/cli/  ──────────────────────    window.electronAPI (preload.ts)
     │                                           │
     ▼                                           ▼
src/hooks/                             Electron Main (ipc/)
  keyword-detector.ts                    ├── cli-ipc (CLI 실행 위임)
  triage-heuristic.ts                    ├── stream-bridge-ipc (스트림 중계)
  session.ts                             ├── state-ipc (상태 구독)
     │                                   └── task-ipc / ops-ipc 등
     │                                           │
     └──────────────┬────────────────────────────┘
                    │
                    ▼
           [ 실행/워크플로 계층 ]
            src/runtime/
            src/pipeline/
            src/team/  / src/ralph/
            src/question/ / src/planning/
            src/goal-workflows/
                    │
          ┌─────────┴──────────┐
          ▼                    ▼
     src/state/           src/logs/
   (상태 파일 갱신)        (세션/턴 기록)
          │
          ▼
  Electron Main 상태 감시 (main/state/)
          │  IPC 이벤트
          ▼
    Electron Renderer
      LifecycleDashboard
      TaskTimeline / TodoPanel
      ChatContainer
          │
          ├──► src/hud/       (tmux 계기판 ANSI 갱신)
          └──► src/notifications/ (알림 외부 발송)
```

#### 경로 A — CLI 명령 흐름 (상세)

1. **입력 수신**: 사용자가 터미널에서 `omx <command>` 를 실행한다.
2. **CLI 파싱**: `src/cli/`가 명령어와 인자를 파싱하고 기본 유효성을 검사한다.
3. **Hook 라우팅**: `src/hooks/keyword-detector.ts`가 입력 텍스트를 스캔해 키워드를 탐지한다. `triage-heuristic.ts`가 작업 규모와 실행 레인(solo/team/ralph 등)을 결정한다.
4. **모드/워크플로 선택**: `src/modes/`에서 현재 활성 모드를 확인하고 적합한 워크플로(pipeline/team/ralph/question 등)를 시작한다.
5. **실행**: `src/runtime/`이 실제 에이전트 명령을 실행하고 `src/exec/`이 외부 프로세스를 시작한다.
6. **상태 기록**: 실행 중 상태 변화는 `src/state/`에 atomic write로 저장하고, `src/logs/`에 세션·턴 단위로 기록된다.
7. **관측 반영**: `src/hud/`가 tmux 계기판을 갱신하고, `src/notifications/`가 외부 게이트웨이로 주요 이벤트를 발송한다.

#### 경로 B — Electron Renderer 명령 흐름 (상세)

1. **입력 수신**: 사용자가 채팅창에 메시지를 입력하거나 UI 버튼을 누른다.
2. **preload 경유**: `window.electronAPI`(preload.ts가 노출)를 통해 Main에 요청을 보낸다. Renderer가 직접 파일/프로세스에 접근하지 않는 보안 경계다.
3. **IPC 핸들러 처리**: `Electron/src/main/ipc/`의 해당 IPC 핸들러(`cli-ipc`, `stream-bridge-ipc` 등)가 요청을 받아 OMX CLI 실행이나 상태 조회를 수행한다.
4. **OMX 런타임 호출**: Main이 필요에 따라 `src/runtime/`, `src/pipeline/` 등을 호출한다(CLI 프로세스 spawn 또는 직접 모듈 호출).
5. **스트림 이벤트 중계**: 실행 중 발생하는 token/tool/done 이벤트를 `stream-bridge-ipc`가 `omx:stream-*` IPC 채널로 Renderer에 실시간 전달한다.
6. **상태 파일 감시**: `main/state/`가 `.omx/state` 파일 변경을 감시(fs.watch)하고, 변경 시 `state-ipc`를 통해 Renderer에 최신 상태를 push한다.
7. **UI 갱신**: Renderer의 `LifecycleDashboard`, `TaskTimeline`, `TodoPanel`, `ChatContainer` 등이 IPC 이벤트를 수신해 화면을 갱신한다.

#### 공통 이벤트 유형

| 이벤트 유형 | 발생 시점 | 소비 모듈 |
|---|---|---|
| `omx:stream-token` | LLM 토큰 스트림 중 | `ChatContainer`, HUD |
| `omx:stream-tool-call` | 도구 호출 시작 | `ChatContainer`, `TaskTimeline` |
| `omx:stream-done` | 실행 완료 | `ChatContainer`, `LifecycleDashboard` |
| `omx:lifecycle-update` | 에이전트 상태 변경 | `LifecycleDashboard` |
| `omx:task-update` | 태스크 상태 변경 | `TaskTimeline`, `TodoPanel` |
| `omx:interlude` | 사용자 확인 요청 | `ChatContainer` (확인 UI) |
| `omx:runtime-hook-event` | 훅 트리거 | 로그, 알림, 상태 |
| `omx:state-change` | `.omx/state` 파일 변경 | `LifecycleDashboard`, HUD |

#### 흐름 불변 원칙
- **Renderer 쓰기 금지**: Renderer는 상태를 직접 쓰지 않는다. Main을 통해서만 변경을 요청한다.
- **스트림 순서 보장**: stream-token 이벤트는 streamId 기준으로 순서를 유지해야 한다.
- **상태 일관성**: 상태 파일 쓰기는 항상 atomic write(`tmp → rename`)를 사용해 부분 쓰기로 인한 오염을 방지한다.
- **hook 재진입 방지**: hooks 내부에서 다시 hooks를 직접 호출하는 재진입 패턴은 금지한다.

## 5. 런타임 시퀀스

런타임 시퀀스는 OMX가 **사용자 입력을 받아 최종 결과를 화면에 반영하기까지의 전체 흐름**을 단계별로 정리한 것이다. 각 섹션은 독립된 시나리오가 아니라 서로 이어지는 흐름이다. 초보 개발자는 5.1 → 5.2 → 5.3 → 5.4 순서로 읽으면 전체 그림을 파악할 수 있다.


### 5.1 사용자 입력 처리 시퀀스

**한 줄 요약**: 사용자가 무언가를 입력하면 OMX가 그것을 어떤 작업인지 판단하고 적합한 실행 경로로 보낸다.

#### 시퀀스 다이어그램

```
사용자
  │  ① 입력 (텍스트 / 키워드 명령 / 파일 경로 등)
  ▼
[입력 수신 계층]
  src/cli/            ─── CLI 직접 실행 경로
  window.electronAPI  ─── Electron Renderer 경로 (preload.ts 경유)
  │
  │  ② 입력 유효성 검사
  │    - 빈 입력 거부
  │    - 경로 traversal 패턴 차단
  │    - 세션 ID / 슬러그 안전 문자 검증
  ▼
[라우팅 계층]
  src/hooks/keyword-detector.ts
  │  ③ 키워드 탐지
  │    - $ralph, $team, $ralplan 등 명시 키워드 → 해당 워크플로 강제 진입
  │    - 탐지 없음 → triage로 이동
  ▼
  src/hooks/triage-heuristic.ts
  │  ④ 작업 규모·레인 결정
  │    - 단순 질문 → solo 실행 (runtime 직접)
  │    - 다단계 계획 → pipeline
  │    - 팀 분업 필요 → team
  │    - 지속적 완료 루프 → ralph
  ▼
[실행 레인 진입]
  src/modes/   ← 현재 활성 모드 확인
  │  ⑤ 워크플로 또는 runtime 시작
  ▼
[스트림 / 상태 기록 분리]
  │  ⑥ LLM 스트림 토큰 → stream-bridge-ipc → Renderer (실시간)
  │     도구 호출 이벤트 → TaskTimeline / TodoPanel 갱신
  │     상태 변경 → src/state/ atomic write
  ▼
[결과 반영]
  ChatContainer    ← 메시지 스트림 표시
  LifecycleDashboard ← 에이전트 상태 갱신
  TaskTimeline     ← 태스크 단계 표시
  TodoPanel        ← 할일 목록 갱신
  src/hud/         ← tmux 계기판 갱신
  src/notifications/ ← 외부 알림 발송
```

#### 단계별 상세 설명

| 단계 | 담당 모듈 | 수행 내용 | 실패 시 처리 |
|---|---|---|---|
| ① 입력 수신 | `src/cli/`, `preload.ts` | 텍스트 수신, 인코딩 정규화 | 연결 오류 → 재연결 시도 |
| ② 유효성 검사 | `src/cli/`, IPC 핸들러 | 빈 입력·경로·슬러그 검증 | 오류 메시지 반환, 실행 중단 |
| ③ 키워드 탐지 | `src/hooks/keyword-detector.ts` | `$keyword` 패턴 스캔 | 탐지 실패 → triage로 fallback |
| ④ 레인 결정 | `src/hooks/triage-heuristic.ts` | 작업 규모·모드 결정 | 판단 불가 → solo 기본 레인 |
| ⑤ 워크플로 시작 | `src/modes/`, 각 워크플로 | 워크플로 초기화, 상태 쓰기 | 초기화 실패 → 5.4 오류 시퀀스 |
| ⑥ 스트림/상태 분리 | `stream-bridge-ipc`, `src/state/` | 토큰 스트림·상태 병렬 기록 | 스트림 끊김 → done 이벤트 강제 발송 |
| ⑦ 결과 반영 | Renderer 컴포넌트, HUD, 알림 | UI 갱신, 알림 발송 | 렌더 오류 → ErrorGuideOverlay 표시 |


### 5.2 라우팅/계획/실행 시퀀스

**한 줄 요약**: 입력이 들어온 뒤 OMX가 계획을 세우고 단계별로 실행하며 완료를 판정하는 흐름이다.

#### 시퀀스 다이어그램

```
[입력 라우팅 결과 수신]
        │
        ▼
   src/planning/
   │  ① 계획 문서 읽기
   │    - .omx/plans/prd-*.md  (요구사항)
   │    - .omx/plans/test-spec-*.md (테스트 스펙)
   │    - 없으면 planning 단계 skip
        │
        ▼
   src/pipeline/
   │  ② 단계 순서(DAG) 생성
   │    - 각 단계: { id, title, deps[], tool?, status }
   │    - 의존 단계 완료 확인 후 다음 단계 실행
        │
        ▼ (단계 실행 중)
   ┌────────────────────────────────────────┐
   │           실행 레인 분기                │
   │                                        │
   │  solo      → src/runtime/ 직접 실행    │
   │  team      → src/team/ (tmux 멀티 워커)│
   │  ralph     → src/ralph/ (완료 루프)    │
   │  ralplan   → src/ralplan/ (합의 검증)  │
   │  question  → src/question/ (대화 발행) │
   └────────────────────────────────────────┘
        │
        ▼ (각 단계 완료 후)
   src/goal-workflows/
   │  ③ handoff 처리 / ledger 기록
   │    - 단계 상태 → src/state/ 갱신
   │    - 완료 항목 → ledger.jsonl 추가
   │    - 다음 단계 deps 해제
        │
        ▼
   src/verification/
   │  ④ 완료 증거 검증 게이트
   │    - 테스트 통과 여부 확인
   │    - 아티팩트 존재 확인
   │    - 실패 시 → ralph 루프 재진입
        │
        ▼
   [전체 완료 / 다음 워크플로 handoff]
```

#### 실행 레인별 특성

| 레인 | 담당 모듈 | 특징 | 완료 조건 |
|---|---|---|---|
| **solo** | `src/runtime/` | 단일 에이전트, 즉시 실행 | LLM 응답 완료 + stream-done |
| **team** | `src/team/` | tmux 멀티 워커, 병렬 분업 | 모든 워커 완료 + 통합 |
| **ralph** | `src/ralph/` | 반복 완료 루프, 감사 증거 기반 | verification 게이트 통과 |
| **ralplan** | `src/ralplan/` | 다중 검토자 합의 | 합의 임계값 충족 |
| **question** | `src/question/` | 사용자 확인 대기 | 사용자 응답 수신 |
| **autoresearch** | `src/autoresearch/` | 자동 반복 연구 | 목표 조건 충족 |

#### 단계 상태 전이

```
not-started → in-progress → completed
                   │
                   ▼ (실패 시)
               failed → (복구 가능?) → in-progress (재시도)
                              │ (불가)
                              ▼
                           aborted
```

### 5.3 상태 동기화 및 UI 반영 시퀀스

**한 줄 요약**: 런타임에서 상태 파일이 변경되면 Electron Main이 감지해 Renderer 화면에 반영하는 흐름이다.

#### 시퀀스 다이어그램

```
[OMX 런타임 실행 중]
        │
        │ 상태 변경 발생
        ▼
   src/state/
   │  ① atomic write (.omx/state/*.json)
   │    - tmp 파일에 먼저 쓴 뒤 rename으로 교체
   │    - 부분 쓰기로 인한 오염 방지
        │
        ▼
   Electron Main / main/state/
   │  ② fs.watch로 .omx/state 디렉토리 감시
   │    - 변경 감지 시 파일 읽기 + 파싱
   │    - 디바운스 처리(연속 변경 병합)
        │
        ▼
   state-ipc.ts (Electron Main → Renderer)
   │  ③ omx:state-change IPC 이벤트 발송
   │    payload: { key, value, updatedAt }
        │
        ▼
   Electron Renderer
   │  ④ 컴포넌트별 이벤트 구독
   │
   ├── LifecycleDashboard  ← lifecycle 상태 변경 구독
   ├── TaskTimeline        ← task 상태 변경 구독
   ├── TodoPanel           ← todo 목록 갱신 구독
   ├── ChatContainer       ← 메시지 스트림 구독
   ├── AdapterStatusBar    ← 어댑터 연결 상태 구독
   └── KnowledgePanel      ← wiki 갱신 구독
        │
        ▼
   [병렬 터미널 관측]
   src/hud/
   │  ⑤ watch loop (폴링 주기 설정 가능)
   │    - 상태 파일 직접 읽기
   │    - ANSI 이스케이프로 tmux 계기판 갱신
        │
   src/sidecar/
   │  ⑥ 팀 워커 상태 요약 터미널 패널 갱신
        │
   src/logs/
      ⑦ 세션/턴/툴 호출 기록 (JSONL append)
         - session-*.jsonl
         - turn-*.jsonl
         - tool-*.jsonl
```

#### 상태 전파 지연 목표

| 경로 | 목표 지연 | 비고 |
|---|---|---|
| atomic write → fs.watch 감지 | < 50 ms | OS 파일 시스템 이벤트 |
| fs.watch → IPC 발송 | < 10 ms | 디바운스 후 즉시 |
| IPC 발송 → Renderer 갱신 | < 16 ms | 1 프레임(60fps) 이내 |
| 스트림 토큰 → ChatContainer | < 50 ms | 실시간 타이핑 효과 |
| 상태 변경 → HUD 갱신 | < 200 ms | watch loop 주기 내 |

### 5.4 오류/중단/재시도 시퀀스

**한 줄 요약**: 실패가 발생했을 때 OMX가 원인을 판단하고 복구하거나, 사용자에게 확인을 요청하거나, 안전하게 종료하는 흐름이다.

#### 오류 분류 및 처리 경로

```
오류 발생
    │
    ▼
[오류 분류]
    │
    ├── 네트워크/외부 API 오류
    │       ↓
    │   retry with backoff (최대 3회)
    │       ↓ 실패 지속
    │   Graceful Degradation (핵심 기능만 유지)
    │       ↓
    │   src/notifications/ 로 오류 알림 발송
    │
    ├── 상태 파일 손상 / lock 충돌
    │       ↓
    │   tmp 파일 정리 + 마지막 유효 상태로 rollback
    │       ↓ rollback 불가
    │   workspace state → user home state fallback
    │
    ├── 사용자 확인 필요 (ambiguous / destructive)
    │       ↓
    │   src/question/ 또는 interlude 발행
    │   (omx:interlude IPC → ChatContainer 확인 UI)
    │       ↓ 사용자 응답 대기
    │   승인 → 실행 재개
    │   거부 → 해당 단계 skip 또는 abort
    │
    ├── 단계 실패 (pipeline/team/ralph 내)
    │       ↓
    │   상태: in-progress → failed
    │       ↓ 복구 가능 판단
    │   재시도 가능 → in-progress 재진입 (최대 retry 횟수 제한)
    │   재시도 불가 → aborted, ledger에 실패 기록
    │
    └── 치명적 오류 (프로세스 충돌 등)
            ↓
        Main 프로세스 crash handler 동작
        ErrorGuideOverlay 표시 (Renderer)
        최후 상태 스냅샷 저장 (복구용)
```

#### 오류 유형별 담당 모듈 및 처리 전략

| 오류 유형 | 감지 위치 | 복구 전략 | 사용자 노출 여부 |
|---|---|---|---|
| LLM API 타임아웃 | `src/runtime/` | 지수 백오프 재시도 → 오류 메시지 | 최종 실패 시만 노출 |
| MCP 서버 연결 실패 | `src/mcp/` | 재연결 시도 → 해당 도구 비활성화 | `AdapterStatusBar` 표시 |
| 상태 파일 lock 충돌 | `src/state/` | lock 해제 대기 → timeout 후 강제 해제 | 숨김 (자동 처리) |
| 워크플로 단계 실패 | `src/pipeline/` | 재시도 or question 발행 | `TaskTimeline` 실패 표시 |
| 사용자 확인 필요 | `src/question/` | interlude 발행, 응답 대기 | `ChatContainer` 확인 UI |
| Renderer 렌더 오류 | `renderer/App.tsx` | Error Boundary 캐치 | `ErrorGuideOverlay` 표시 |
| 외부 알림 실패 | `src/openclaw/` | 로컬 로그로 fallback | 숨김 (로그 기록) |
| 인증 만료 | `src/auth/` | 슬롯 교체(hotswap) 시도 | 교체 실패 시 `ApiKeySettings` 안내 |

#### 재시도 설계 원칙

- **Idempotent 보장**: 같은 단계를 다시 실행해도 결과가 안정적이어야 한다. 상태 파일에 단계 ID를 기록해 중복 실행을 막는다.
- **최대 재시도 횟수**: 단계별 기본 3회. 초과 시 `failed` 상태로 전이하고 ledger에 기록한다.
- **지수 백오프**: 외부 API 재시도는 1초 → 2초 → 4초 간격으로 늘린다. 최대 대기 30초.
- **부분 완료 보존**: 재시도 시 이미 완료된 하위 단계는 다시 실행하지 않는다. 상태 파일에서 완료 여부를 확인한다.
- **중단 안전**: `$cancel` 키워드 또는 interlude 거부 시 현재 단계를 clean하게 중단하고, in-flight 도구 호출은 응답을 기다리거나 timeout 후 종료한다.

## 6. 데이터/상태 모델

데이터/상태 모델은 OMX의 모든 모듈이 **공유하는 자료 구조 계약**이다. 모듈 간 통신에서 불일치가 생기는 주요 원인은 각 모듈이 서로 다른 타입을 가정하기 때문이다. 여기서 정의하는 형식을 단일 소스로 사용해야 한다.

---

### 6.1 공통 데이터 계약

#### 6.1.1 Message

LLM 대화 히스토리의 최소 단위. `ChatContainer` 렌더링과 runtime 추론 히스토리 양쪽에서 사용된다.

```ts
interface Message {
  id: string;          // UUID v4, 유일성 보장
  role: 'user' | 'assistant' | 'system' | 'tool';
  content: string;     // 텍스트 본문 (마크다운 가능)
  toolName?: string;   // role='tool' 일 때 도구 이름
  toolCallId?: string; // tool call과 tool result 연결 키
  createdAt: string;   // ISO 8601, 정렬 및 표시용
}
```

> **규칙**: `id`는 생성 시 부여하고 이후 변경하지 않는다. `content`가 빈 문자열인 경우 UI에서 필터링해야 한다.

---

#### 6.1.2 Task

단계 실행 단위. `pipeline`, `team`, `ralph`, `TaskTimeline` 컴포넌트가 공유한다.

```ts
interface Task {
  id: string;                     // UUID v4
  title: string;                  // 사람이 읽는 단계 이름
  status: TaskStatus;             // 아래 열거형 참조
  deps: string[];                 // 선행 task id 목록 (DAG)
  assignee?: string;              // team 모드에서 워커 이름
  startedAt?: string;             // ISO 8601
  completedAt?: string;           // ISO 8601
  retryCount: number;             // 현재까지 재시도 횟수
  artifactPaths?: string[];       // 생성된 파일 경로 목록
}

type TaskStatus =
  | 'not-started'
  | 'in-progress'
  | 'completed'
  | 'failed'
  | 'aborted'
  | 'skipped';
```

> **규칙**: `deps` 에 있는 모든 task가 `completed` 상태여야 해당 task를 `in-progress`로 전환할 수 있다.

---

#### 6.1.3 Lifecycle

현재 활성 OMX 모드와 실행 상태. `LifecycleDashboard`, `modes`, `state`가 공유한다.

```ts
interface Lifecycle {
  status: 'idle' | 'running' | 'paused' | 'error';
  activeMode?: string;         // 현재 실행 중인 모드 이름 (ralph, team 등)
  mergedModes: string[];       // 중첩 활성 모드 목록
  updatedAt: string;           // ISO 8601
  audit?: AuditEntry[];        // ralph 완료 증거 목록
}

interface AuditEntry {
  claim: string;               // 검증 주장 ("tests pass", "artifact exists" 등)
  verified: boolean;
  evidence?: string;           // 증거 경로 또는 요약
  checkedAt: string;
}
```

---

#### 6.1.4 Stream Event (LLM 스트림)

LLM 응답을 실시간으로 전달할 때 사용하는 이벤트. `stream-bridge-ipc`가 Renderer로 전송한다.

```ts
// 텍스트 토큰 청크
interface StreamTokenEvent {
  type: 'token';
  streamId: string;
  textChunk: string;
}

// 도구 호출 시작
interface StreamToolCallEvent {
  type: 'tool-call';
  streamId: string;
  toolCallId: string;
  toolName: string;
  args?: Record<string, unknown>;
}

// 스트림 완료
interface StreamDoneEvent {
  type: 'done';
  streamId: string;
  exitCode: 0 | 1;
  reason?: string;   // 정상: undefined, 중단: 'cancelled', 오류: 오류 메시지
}
```

> **규칙**: `streamId`로 여러 스트림을 구별한다. Renderer는 `streamId`가 다른 이벤트를 독립된 스트림으로 취급한다.

#### 6.1.5 Goal / Workflow State

`goal-workflows`, `ralph`, `ralplan` 등이 `.omx/state/` 에 저장하는 워크플로 영속 상태.

```ts
interface WorkflowState {
  version: number;            // 스키마 버전 (현재 1)
  workflow: string;           // 워크플로 이름 ('ralph', 'team', 'goal' 등)
  slug: string;               // 작업 고유 식별자 (파일명 안전 문자만)
  status: 'active' | 'completed' | 'aborted';

  artifactDir: string;        // 생성 아티팩트 경로
  startedAt: string;
  completedAt?: string;
}

interface LedgerEntry {
  stepId: string;
  event: 'started' | 'completed' | 'failed' | 'retried';
  at: string;                 // ISO 8601
  detail?: string;
}
```

### 6.2 상태 파일(.omx/state) 모델

#### 파일 구조

```
.omx/
├── state/
│   ├── lifecycle.json          ← Lifecycle 구조체 (실시간 갱신)
│   ├── tasks.json              ← Task[] 배열 (현재 실행 단계)
│   ├── todo.json               ← 할일 목록
│   ├── session.json            ← 현재 세션 메타데이터
│   └── workflow-{slug}.json    ← WorkflowState (워크플로별 1개)
├── logs/
│   ├── session-{id}.jsonl      ← 세션 이벤트 (append-only)
│   ├── turn-{id}.jsonl         ← 턴별 대화 기록
│   └── tool-{id}.jsonl         ← 도구 호출 기록
├── plans/
│   ├── prd-{slug}.md           ← 요구사항 문서 (planning 단계 산출물)
│   └── test-spec-{slug}.md     ← 테스트 스펙
└── project-memory.json         ← 크로스 세션 메모리
```

#### 쓰기 규칙

| 파일 유형 | 쓰기 방식 | 읽기 우선순위 |
|---|---|---|
| `*.json` | atomic write (tmp → rename) | workspace `.omx/` → user home `.omx/` |
| `*.jsonl` | append-only | 세션 범위 우선 |
| `plans/*.md` | 덮어쓰기 허용 (planning 단계) | 항상 workspace 범위 |
| `project-memory.json` | atomic write | workspace → home |

> **원칙**: `.json` 파일은 임시 파일에 먼저 쓰고 `rename`으로 교체해 부분 쓰기를 방지한다. 읽을 때 파싱 실패 시 빈 기본값으로 fallback하고 오류를 로그에 남긴다.

### 6.3 IPC Payload 계약

Electron Main ↔ Renderer 사이의 IPC 채널은 채널마다 고정 payload 형식을 사용한다.

#### 채널별 Payload 정의

| 채널 이름 | 방향 | Payload 구조 | 설명 |
|---|---|---|---|
| `omx:stream-token` | Main→Renderer | `StreamTokenEvent` | 토큰 청크 실시간 전달 |
| `omx:stream-tool-call` | Main→Renderer | `StreamToolCallEvent` | 도구 호출 알림 |
| `omx:stream-done` | Main→Renderer | `StreamDoneEvent` | 스트림 완료 |
| `omx:state-change` | Main→Renderer | `{ key: string; value: unknown; updatedAt: string }` | 상태 파일 변경 통보 |
| `omx:task-update` | Main→Renderer | `{ tasks: Task[] }` | 태스크 목록 전체 갱신 |
| `omx:lifecycle-update` | Main→Renderer | `Lifecycle` | 라이프사이클 갱신 |
| `omx:interlude` | Main→Renderer | `{ question: string; choices?: string[] }` | 사용자 확인 요청 |
| `omx:interlude-response` | Renderer→Main | `{ answer: string }` | 사용자 응답 |
| `omx:cli-run` | Renderer→Main | `{ command: string; sessionId?: string }` | CLI 명령 실행 요청 |
| `omx:env-get` | Renderer→Main | `{ key: string }` | 환경 변수 조회 |

#### 오류 Payload

모든 IPC 응답에서 오류 발생 시 다음 형식을 사용한다.

```ts
interface IpcError {
  code: string;       // 머신 가독 오류 코드 ('NOT_FOUND', 'PERMISSION_DENIED' 등)
  message: string;    // 사람 가독 설명
  detail?: unknown;   // 추가 디버그 정보 (선택)
}
```

### 6.4 버전/호환성 정책

#### 스키마 버전 관리

- 모든 영속 JSON 파일에 `version` 필드를 포함한다. 현재 버전은 `1`이다.
- 런타임이 파일을 읽을 때 `version` 필드를 먼저 확인하고, 미지원 버전이면 경고 후 기본값으로 fallback한다.

#### 변경 허용/금지 기준

| 변경 유형 | 허용 여부 | 처리 방법 |
|---|---|---|
| 새 선택 필드 추가 | ✅ 허용 | 즉시 추가 가능, 기존 독자는 무시 |
| 새 필수 필드 추가 | ⚠️ 제한 | `version` 올림 + 마이그레이션 코드 필요 |
| 기존 필드 이름 변경 | ❌ 금지 | deprecated 별칭 → 마이그레이션 기간 종료 후 삭제 |
| 기존 필드 타입 변경 | ❌ 금지 | 새 필드 추가 후 구 필드 deprecated 처리 |
| 열거형 값 추가 | ✅ 허용 | 독자 측 unknown 값 처리 코드 필요 |
| 열거형 값 삭제 | ❌ 금지 | deprecated 기간 최소 2 마이너 버전 |

#### IPC 채널 호환성

- 채널 이름(`omx:*`)은 변경하지 않는다.
- payload 형식 변경 시 기존 필드를 유지한 채로 새 필드를 추가하는 방식으로만 확장한다.
- Renderer 코드는 알 수 없는 채널 이벤트를 조용히 무시해야 한다 (미래 Main 버전 대비).

## 7. 인터페이스 설계

### 7.1 Main-Renderer IPC 채널 설계
- Renderer는 `window.electronAPI`만 사용한다.
- Main은 채널별로 invoke/on handler를 등록한다.
- 채널은 역할별로 나눈다: stream, state, task, interlude, env, cli, ops, adapter.
- 채널 간 payload 구조는 단일 소스로 관리하는 것이 이상적이다.

### 7.2 CLI 실행 인터페이스 설계
- `omx <command> [subcommand] [args...]` 형태를 따른다.
- CLI는 읽기/실행/검증/도구 호출을 묶는 최상위 표면이다.
- 커맨드 결과는 종료 코드와 표준 출력/에러로 표현한다.
- 장기 실행 명령은 스트림이나 watch 모드가 될 수 있다.

### 7.3 MCP/Adapter 인터페이스 설계
- MCP는 서버 자동 시작, 상태 경로 해석, 생명주기 telemetry를 담당한다.
- Adapter는 외부 목표 시스템에 대한 읽기/보고/초기화 표면을 제공한다.
- OpenClaw는 외부 통지와 연동을 처리한다.
- auth/catalog/agents/adapt는 설치·구성·관찰용 인터페이스를 제공한다.

## 8. 보안 및 권한 설계

### 8.1 인증/권한 모델
- Main이 권한이 큰 연산을 담당한다.
- Renderer는 직접 권한이 큰 작업을 할 수 없다.
- auth는 슬롯 기반 인증 전환을 관리한다.
- safeStorage 같은 OS 보호 기능은 민감 정보 저장에 사용한다.

### 8.2 입력 검증/무결성
- 경로는 traversal 공격을 막기 위해 검증한다.
- 세션 ID, 슬러그, 모드 이름은 안전한 문자 집합만 허용한다.
- JSON은 파싱 실패 시 안전하게 fallback한다.
- UI에서 받는 payload는 절대 완전히 신뢰하지 않는다.

### 8.3 민감정보/키 관리
- API 키와 토큰은 평문 저장을 피한다.
- 로그에는 민감한 값을 남기지 않는다.
- auth, services, logs, notifications는 마스킹 정책을 지켜야 한다.

### 8.4 감사 추적(Audit)
- ledger.jsonl과 이벤트 로그는 변경 추적을 위한 핵심 수단이다.
- team, ralph, goal-workflows, performance-goal은 상태 변화 이유를 남긴다.
- 알림과 hooks는 언제 어떤 트리거가 발생했는지 기록하는 것이 중요하다.

## 9. 성능 및 안정성 설계

### 9.1 성능 목표 및 지표
- 빠른 초기 로드
- 스트림 응답의 낮은 지연
- 상태 파일 읽기의 최소 I/O
- UI 리렌더 최소화
- tmux/HUD watch loop의 과도한 CPU 사용 방지

### 9.2 병목 구간/최적화 전략
- 상태 파일은 필요한 것만 읽는다.
- 스트림 token은 바로 렌더하되 전체 화면을 매번 다시 그리지 않는다.
- 반복 검사 항목은 캐시한다.
- tmux와 IPC 같은 외부 호출은 최소한으로 사용한다.

### 9.3 장애 복구/폴백 전략
- workspace state가 없으면 user home state를 fallback으로 본다.
- 외부 게이트웨이 실패 시 로컬 안내 메시지를 사용한다.
- 여러 번 실패해도 앱 전체가 죽지 않도록 설계한다.
- task claim 충돌, state lock 충돌, hook dispatch 실패는 복구 경로를 둔다.

### 9.4 관측성(로그/메트릭/트레이스)
- logs는 세션과 툴 호출을 기록한다.
- hud는 현재 활성 상태를 표시한다.
- notifications는 실행 중 주요 이벤트를 외부로 전달한다.
- sidecar는 팀 상태 요약을 보여준다.

#### 13.34 components 모듈 설계 (예시 섹션)
- 기준 문서: Electron/analysis/components-module-analysis.md
- 안내: 본 섹션은 템플릿 작성 방식 예시다. 다른 모듈은 동일 구조(개요/주요기능 + 13.xx.1~13.xx.9)로 확장 작성한다.
- 개요: `components` 모듈은 Renderer UI를 기능 단위로 분리한 프레젠테이션 계층이며, 채팅/상태/설정/보조 오버레이를 조합해 사용자 워크플로우를 구성한다.
- 주요기능:
  - 채팅 상호작용: `ChatContainer`를 중심으로 메시지 입력, 스트리밍 응답, 인터류드 모드, 컨텍스트 첨부 처리
  - 상태 시각화: `LifecycleDashboard`, `TodoPanel`, `TaskTimeline`으로 실행 상태/할일/이력 표시
  - 모델 및 환경 제어: `ModelSelector`, `ApiKeySettings`, `AdapterStatusBar`를 통한 모델 선택, API 키 상태, 어댑터 상태 표시
  - 보조 UX: `WikiOverlay`, `DeferredSkillsNotice`, `ErrorGuideOverlay`, `KnowledgePanel` 기반의 안내/지식 탐색/오류 피드백 제공

##### 13.34.1 책임과 경계
- `components`는 기능 단위 UI 캡슐화를 담당한다.
- 도메인 상태 소스는 Main/IPC이며, 컴포넌트는 구독/표현/입력 수집에 집중한다.
- 주요 축:
  - 대화 축: `ChatContainer`
  - 상태 축: `LifecycleDashboard`, `TodoPanel`, `TaskTimeline`
  - 설정/보조 축: `ModelSelector`, `ApiKeySettings`, `AdapterStatusBar`, `WikiOverlay`

##### 13.34.2 외부/내부 인터페이스
- 외부 인터페이스: `window.electronAPI` 구독/요청 메서드
- 내부 인터페이스: 컴포넌트 props 기반 조합
  - 예: `ChatContainer.onSendMessage`, `ModelSelector.onChange`, `LifecycleDashboard.stateDir`

##### 13.34.3 데이터 구조와 계약
- 메시지 모델: `{ id, role, content }`
- 할일 모델: `{ id, title, status }`
- 수명주기 모델: `{ status, activeMode?, mergedModes, updatedAt, audit? }`
- UI는 수신 payload를 신뢰하지 않고 기본값으로 방어 병합한다.

##### 13.34.4 상태 전이와 불변식
- 채팅 전송 후 스트리밍 시작 전에는 입력 상태를 초기화한다.
- 인터류드 활성 시 일반 입력창은 비활성/오버레이 모드로 전환한다.
- 스트림 완료 시 임시 스트리밍 텍스트를 확정 메시지로 반영한다.

##### 13.34.5 핵심 시퀀스
- 사용자 메시지 전송:
  1. 입력/첨부 컨텍스트 결합
  2. `onSendMessage`로 상위 전달
  3. 스트리밍 token 수신 시 assistant 버블 갱신
  4. done 수신 시 메시지 확정 및 디버그 상태 초기화

##### 13.34.6 오류 처리 및 복구
- IPC 미존재/구독 실패 시 컴포넌트 크래시 없이 no-op 처리
- stream error는 사용자 가시 영역(에러 버블)로 표준 출력
- 구독 해제(cleanup)를 강제해 메모리 누수를 방지

##### 13.34.7 보안/성능 고려
- Markdown 렌더링은 컴포넌트 매핑으로 표시 규칙을 제어한다.
- 긴 텍스트/테이블 렌더링에서 overflow 스타일을 적용해 레이아웃 붕괴를 방지한다.
- 불필요한 리렌더를 줄이기 위해 상태 업데이트 범위를 최소화한다.

##### 13.34.8 테스트 케이스 맵
- 채팅 입력/전송/엔터 키 동작.
- 인터류드 진입/해제 시 UI 잠금 및 복귀.
- todo/lifecycle/tool_call 이벤트 수신 시 화면 갱신.

##### 13.34.9 오픈 이슈
- 컴포넌트별 스타일 분리(CSS module 또는 토큰화) 필요.
- 장문 대화 이력 가상 스크롤 도입 검토.

- A3: 워크플로 완료는 ledger와 snapshot reconciliation을 함께 요구한다.
