# OMX Core 모듈 개별 설계서

## 0. 문서 정보
- 모듈: core
- 기준 분석 문서: Electron/analysis/core-module-analysis.md
- 상태: Draft v0.1
- 작성일: 2026-05-29

## 1. 개요와 주요기능
- 개요: core 모듈은 외부 명령 실행의 최소 공통 기반 계층으로, spawn 기반 비동기 실행과 표준 출력 라인 파싱을 통해 상위 파이프라인이 안정적으로 이벤트를 소비하도록 지원한다.
- 주요기능:
  - 비동기 명령 실행: 동기 실행 금지 원칙을 지키며 자식 프로세스를 실행
  - 출력 라우팅: stdout/stderr 라인 단위 처리와 JSON 파싱 가드 적용
  - 이벤트 분기: onEnvelope와 onRawLine 경로를 분리해 파이프라인 오염 최소화
  - 종료 정합성: 종료 코드와 비정상 종료를 done 이벤트 의미론에 맞춰 전달

## 2. 책임과 경계
- 책임:
  - 명령 실행 수명주기(start, stream, done, error)를 표준 형태로 제공
  - 출력 라인 파싱 실패가 상위 흐름 전체를 중단시키지 않도록 격리
  - 운영체제별 실행 차이를 최소화하는 공통 실행 규칙 유지
- 비책임:
  - 비즈니스 상태 전이 결정, UI 메시지 렌더링, 사용자 정책 판단은 담당하지 않는다.
- 경계:
  - core는 실행 기반 레이어이며, 상위 계층(cli/ipc/services)이 도메인 정책을 결정한다.

## 3. 외부/내부 인터페이스
- 외부 인터페이스:
  - execute-command 유틸리티(명령, 인자, 콜백 인수 기반)
  - 콜백 경로: onEnvelope, onRawLine, onDone, onError
- 내부 인터페이스:
  - Node child_process spawn API
  - 라인 단위 버퍼 분해기 및 JSON 파싱 가드
- 호출자:
  - main/cli 실행 계층, ipc 스트림 브리지, 운영 점검 파이프라인

## 4. 데이터 구조와 계약
- 실행 입력 계약:
  - `{ command: string, args: string[], cwd?, env?, shell? }`
- 스트림 이벤트 계약:
  - envelope 라인: JSON parse 성공 시 구조화 이벤트로 전달
  - raw 라인: parse 실패 시 원문 라인으로 전달
- 종료 계약:
  - done 이벤트는 `exitCode` 또는 종료 원인 정보를 포함해야 한다.
  - error 이벤트는 사용자 표시용 메시지와 내부 진단 정보를 분리한다.
- 계약 원칙:
  - stderr 신호는 노이즈 제어 정책에 따라 상위 계층에서 표시 강도를 결정한다.

## 5. 상태 전이와 불변식
- 실행 상태 전이:
  - idle -> spawning -> streaming -> completed | failed | cancelled
- 출력 처리 상태:
  - buffering -> line-split -> envelope-emit | raw-emit
- 불변식:
  - 동기 실행 API를 사용하지 않는다.
  - done 이후 추가 stream 이벤트를 발행하지 않는다.
  - JSON 파싱 실패는 raw 경로로 우회되며 전체 실행을 중단시키지 않는다.

## 6. 핵심 시퀀스
- 명령 실행 시퀀스:
  1. 호출자가 실행 인자를 전달
  2. core가 spawn으로 프로세스를 시작
  3. stdout 라인을 분해해 JSON 파싱 시도
  4. 성공 시 onEnvelope, 실패 시 onRawLine 콜백 호출
  5. 프로세스 종료 시 onDone 호출
- 오류 시퀀스:
  1. spawn 예외 또는 실행 오류 발생
  2. error 객체를 표준 메시지로 정규화
  3. onError 호출 후 실패 종료로 전이

## 7. 오류 처리 및 복구
- Windows 인용(quoting) 오류:
  - 실행 인자 정규화 규칙을 상위 호출자와 일치시켜 재시도 가능성을 확보
- 파싱 오류:
  - envelope 경로 실패 시 raw 라인 처리로 복구
- 비정상 종료:
  - exitCode와 종료 원인을 함께 전달해 상위에서 실패/취소를 구분
- stderr 노이즈:
  - 사용자 메시지와 내부 로그 레벨을 분리해 과도한 UI 노출을 억제

## 8. 보안/성능 고려
- 보안:
  - 명령/인자 입력은 상위 계층에서 화이트리스트 또는 검증 정책을 적용
  - 셸 실행 필요 시 escaping 규칙을 강제
- 성능:
  - 라인 단위 처리로 대용량 스트림에서 메모리 폭증을 완화
  - 불필요한 JSON 재파싱을 줄이는 단일 패스 처리 유지

## 9. 테스트 케이스 맵
- 단위:
  - JSON 파싱 성공/실패 분기(onEnvelope/onRawLine)
  - 종료 코드별 done 이벤트 정합성
- 통합:
  - cli/ipc 경유 실행에서 스트림 이벤트 순서 보장
  - stderr 노이즈 제어 정책 반영 검증
- 회귀:
  - Windows 인용 규칙으로 인한 실행 실패 재발 방지
  - done 이후 이벤트 발행 차단 검증
