# OMX Env 모듈 개별 설계서

## 0. 문서 정보
- 모듈: env
- 기준 분석 문서: Electron/analysis/env-module-analysis.md
- 상태: Draft v0.1
- 작성일: 2026-05-29
- 문서 목적: 실행 전 환경 검증 게이트의 기준과 실패 처리 규칙 정의
- 용어 기준: 한글 우선 표기, 최초 1회 영문 병기
- 업데이트 기준: 필수 파일/경로/점검 메시지 변경 시 1, 4, 7장을 우선 갱신

## 1. 개요와 주요기능
- 개요: env 모듈은 실행 전 전제조건을 점검하는 안전 게이트로, 프로젝트 구조와 필수 파일/환경을 확인해 사전 실패를 감지하고 사용자에게 명확한 안내를 제공한다.
- 주요기능:
  - 필수 조건 점검: 경로, 파일, 환경 변수, 런타임 의존성 검증
  - 조기 차단: 실행 중 오류 대신 실행 전 오류로 전환해 복구 비용 최소화
  - 가시성 제공: 점검 결과를 IPC/시스템 메시지로 전달

## 2. 책임과 경계
- 책임:
  - 실행 가능 여부를 사전에 판정하고 실패 원인을 구조화해 반환
  - OS별 경로 해석 차이를 흡수해 일관된 점검 결과 제공
  - 사용자 안내 메시지 품질을 유지
- 비책임:
  - 실제 실행 재시도, 자동 수정, UI 렌더링은 담당하지 않는다.
- 경계:
  - env는 검증 계층이며 실행 제어는 main/ipc/cli에서 수행한다.

## 3. 외부/내부 인터페이스
- 외부 인터페이스:
  - env-checker 기반 점검 함수(실행 전 preflight)
  - 점검 결과 객체(success, issues, hints)
- 내부 인터페이스:
  - 파일시스템 존재 확인/권한 확인
  - OS 경로 정규화 유틸
  - 시스템 메시지 브리지(호출 계층에서 사용)

## 4. 데이터 구조와 계약
- 점검 결과 계약:
  - `{ ok: boolean, checks: CheckResult[], summary: string }`
  - `CheckResult = { item, status: 'pass'|'warn'|'fail', reason?, hint? }`
- 실패 계약:
  - fail 항목이 1개 이상이면 실행 차단
  - warn 항목은 실행 가능하되 사용자 고지
- 메시지 계약:
  - 사용자 메시지는 원인과 조치 힌트를 함께 제공

## 5. 상태 전이와 불변식
- 점검 상태 전이:
  - pending -> checking -> pass | warn | fail
- 실행 게이트 상태 전이:
  - unknown -> allowed | blocked
- 불변식:
  - fail 결과에서 실행 허용으로 전이하지 않는다.
  - 동일 입력 환경에 대해 점검 결과는 결정적이어야 한다.
  - 점검 실패 원인은 최소 1개의 구체적 reason을 포함해야 한다.

## 6. 핵심 시퀀스
- preflight 시퀀스:
  1. main/ipc가 실행 전 env-checker 호출
  2. 필수 파일/경로/환경 검증 수행
  3. check 결과를 pass/warn/fail로 집계
  4. fail이면 실행 차단 및 안내 메시지 반환
  5. pass/warn이면 실행 계층으로 제어 전달

## 7. 오류 처리 및 복구
- 경로 해석 오류:
  - OS별 separator/절대경로 규칙 차이를 정규화해서 재판정
- 누락 파일:
  - 누락 항목과 기대 경로를 메시지로 명시
- 권한 오류:
  - 권한 부족과 파일 미존재를 구분해 안내
- 메시지 품질:
  - 원인-only 메시지를 금지하고 조치 힌트를 필수 포함

## 8. 보안/성능 고려
- 보안:
  - 환경 점검 과정에서 민감 환경 변수 값은 노출하지 않는다.
  - 경로 정보 출력 시 필요한 최소 범위만 보여준다.
- 성능:
  - 점검 항목은 경량 파일 체크 중심으로 유지
  - 중복 체크를 피하기 위해 호출 단위 결과 캐시 전략 검토

## 9. 테스트 케이스 맵
- 단위:
  - 누락 파일 탐지 정확도
  - OS별 경로 해석 일관성
  - 결과 집계(pass/warn/fail) 규칙
- 통합:
  - 실행 전 차단/허용 경로가 main 흐름에서 정상 동작하는지 검증
- 회귀:
  - 점검 실패 메시지에서 원인/조치 힌트 누락 재발 방지

## 10. 오픈 이슈
- 점검 항목 표준 카탈로그(필수/권장/선택)를 별도 문서로 분리할지 검토 필요
- warn 기준 임계치(실행 허용 범위) 정책 통일 필요
- 점검 결과 텔레메트리 수집 범위(익명화 포함) 정의 필요
