# Phase 2 Milestone 5 구현 티켓: 팀 이벤트 구독 기반 stdin 제어 및 블로킹 UI 구현

**Reference Architecture:** [ADR-001-electron-agent-architecture.md](./ADR-001-electron-agent-architecture.md)

이 문서는 [ite-ai-roadmap.md](./ite-ai-roadmap.md) Phase 2의 두 번째 마일스톤인 "팀 이벤트 구독 기반 `stdin` 제어 및 블로킹 UI 구현" 범위를 실제 구현 가능한 티켓으로 분해한 실행 체크리스트다.

에이전트 가동 중 사용자의 판단이나 추가 입력이 필요한 '차단형 인터랙션' 지점을 정교하게 제어하고, CLI의 `stdin`과 Electron UI를 유기적으로 연결하는 것을 목표로 한다.

## 🎯 Phase 2 Milestone 5 범위
- **`stdin` 제어 파이프라인**: CLI 프로세스를 종료하지 않고 입력 대기 상태로 유지하며 데이터를 주입하는 인터페이스.
- **인터류드(Interlude) 신호 처리**: `wakeable_only=true` 계약 조건과 다중 차단 신호를 통합 감지하여 UI 모드 전환.
- **차단형 인터뷰 UI**: 채팅창을 입력 대기 상태로 전환하고, 사용자 입력 이후 디스크 상태 전이 확인 전까지 원자성 락 유지.

## 🎫 EL-214. CLI `stdin` 쓰기 파이프라인 및 백프레셔(Backpressure) 안전 제어

- **우선순위**: P0
- **실행 순서**: 12번째
- **전제 티켓**: Phase 2 Milestone 4 (Ndjson 스트림 인프라 확보)
- **그룹**: Core (Stream Pipeline)
- **목표**: 일시 중단 상태로 대기 중인 CLI 서브 프로세스에 사용자의 텍스트 컨텍스트를 유실 없이 안전하게 주입할 수 있는 `stdin` 쓰기 파이프라인을 구축한다.
- **대상 경로**:
  - `src/main/cli/stdin-writer.ts` (신규)
- **핵심 구현 로직 (보완됨)**:
  1. **스트림 버퍼 안전 가드 (추가)**: 대용량 맥락 입력 시 Node.js 스트림 백프레셔 현상에 대응하기 위해 `child.stdin.write(data)`가 `false`를 리턴할 경우, 추가 쓰기를 일시 정지하고 **`child.stdin.once('drain', ...)` 이벤트를 수신한 뒤 안전하게 잔여 버퍼를 스트리밍하는 논리 구현**.
  2. **개행 규격 준수**: CLI 내부의 `readline` 인터페이스가 대기 상태를 안전하게 돌파할 수 있도록 모든 데이터 끝에 정식 엔드마커(`\n`) 개행 처리를 엄격히 래핑하여 주입.
  3. **예외 수명주기 차단**: 프로세스가 이미 죽었거나 가동 중이지 않은 상태에서 `stdin` 쓰기를 시도할 경우 크래시를 방지하기 위해 `child.killed` 상태를 사전 검증하는 방어 가드 탑재.
- **DoD (완료 기준)**:
  - 런타임에 외부에서 주입된 텍스트가 CLI의 표준 입력으로 정상 전달됨.
  - 백프레셔 상황에서도 입력 데이터 유실 없이 순차 주입됨.
  - 프로세스가 종료되지 않고 다음 이벤트를 기다리는 상태를 유지함.
- **체크리스트**:
  - [ ] 비동기 가동 중인 자식 프로세스의 `stdin` 쓰기 래퍼 모듈 자산화 완료
  - [ ] **[스트림 가드 게이트]** 버퍼 한계 도달 시 유실을 막아주는 drain 이벤트 추적 매커니즘 구현
  - [ ] 프로세스 종료 유무 사전 조회를 통한 예외 방지 구조 적용
  - [ ] `EL-214.test.ts` 에서 고속 버스트 쓰기 데이터 모킹을 통한 스트림 파이프 무결성 테스트 통과

## 🎫 EL-215. 인터류드(Interlude) 및 차단 신호 감지 로직 구현

- **우선순위**: P0
- **실행 순서**: 13번째
- **전제 티켓**: EL-213 (Ndjson 파서)
- **그룹**: Logic (Event Triage)
- **목표**: Ndjson 스트림으로부터 정식 계약에 정의된 `wakeable_only=true` 대상 인터류드(일시 중단) 신호를 정밀 분류 및 감지하여, 메인 프로세스의 수명주기 상태를 즉각 전환한다.
- **대상 경로**:
  - `src/main/cli/interlude-triager.ts` (신규)
- **핵심 구현 로직 (보완됨)**:
  1. **다중 차단 유형 Zod 확장 (추가)**: 단순히 `askUserQuestion` 뿐만 아니라 상호 운용성 계약 문서에 근거한 **`worker_merge_conflict` (통합 충돌 해결 대기), `pre-tool-use` (도구 실행 전 명시적 사용자 승인 대기), `needs-input` (추가 컨텍스트 대기) 상태를 하나의 수렴형 스키마로 선언**.
  2. **페르소나 텍스트 결합**: 감지된 차단 신호에 맞춰 현재 홀딩된 에이전트의 런타임 역할명(`planner`, `executor`, `verifier` 등)과 질문 본문을 일체형 패킷으로 추출.
  3. **무상태 디스패치**: 조건 만족 즉시 `webContents.send('omx:interlude-start', payload)` 채널을 개방하여 Renderer 레이어로 다중 신호 상태를 즉각 송신.
- **DoD (완료 기준)**:
  - CLI에서 인터류드 이벤트가 발생하면 Main Process가 다중 차단 유형을 감지하고 정확한 페이로드를 Renderer로 전송함.
- **체크리스트**:
  - [ ] `interop-team-mutation-contract-ko.md` 기반의 인터류드 통합 Zod 스키마 검증기 구현
  - [ ] 단순 질문, 병합 충돌, 도구 실행 전 승인 단계에 대한 분기 바인딩 완료
  - [ ] Renderer 연동용 인터류드 시작 알림 IPC 이벤트 채널 등록 완료
  - [ ] `EL-215.test.ts` 단위 테스트 구동 시 다중 차단 모크 Ndjson 라인 전송 상태 분류 검증 통과

## 🎫 EL-216. 차단형 인터뷰 모드 UI 제어 및 원자성(Atomicity) 락 구현

- **우선순위**: P1
- **실행 순서**: 14번째
- **전제 티켓**: EL-215 (인터류드 감지), EL-214 (stdin 쓰기)
- **그룹**: UI/UX Integration
- **목표**: 에이전트가 인터류드 상태에 진입할 때 채팅창을 차단형(Blocking) 인터뷰 모드로 강제 전환하고, 클로드 데스크탑 스타일의 포커스 집중형 UI를 제공한다.
- **대상 경로**:
  - `src/main/ipc/interlude-ipc.ts` (신규)
  - `src/renderer/components/ChatContainer.tsx` (UI 컴포넌트 리팩토링 및 가드 결합)
- **핵심 구현 로직 (UI 특화)**:
  1. **포커스 집중형 딤(Dim) 처리**: `omx:interlude-start` IPC 수신 시, 이전 채팅 내역 영역의 opacity를 낮추고(Dim), 최하단 입력창 영역에 사용자의 답변을 요구하는 폼(Form)이나 승인/거절 버튼 컴포넌트를 강조하여 렌더링.
  2. **페르소나 뱃지 노출**: 어떤 역할(예: `Planner`, `Verifier`)이 질문을 던졌는지 시각적으로 구분할 수 있는 페르소나 아바타/뱃지를 인터류드 폼 최상단에 부착.
  3. **입력 동결 및 원자성 락**: 답변 제출 시 UI 즉각 해제를 방지하고, 내부에 '처리 중(Processing...)' 스피너를 유지하며 워커(Watcher)의 `.omx/state/` 갱신 이벤트 수신 시까지 락을 고수.
- **DoD (완료 기준)**:
  - 인터류드 신호 전송 시 채팅 모드가 딤(Dim) 처리된 포커스 인터뷰 모드로 즉시 전환됨.
  - 전송 버튼 클릭 후 디스크의 상태가 갱신될 때까지 UI에 스피너가 표시되고 락이 동작함.
  - 취소 버튼 클릭 시 정상적인 태스크 롤백 루틴이 실행됨.
- **체크리스트**:
  - [ ] 배경 Dim 처리 및 포커스 입력 폼 컴포넌트(인터뷰 모드) 구현
  - [ ] 질문자 페르소나 식별용 UI 뱃지/아이콘 연동
  - [ ] 사용자 입력 제출 시 스피너 유지 및 워처 상태 기반 락 해제 로직 구현
  - [ ] 세션 롤백(작업 취소) 버튼 UI 및 취소 시 이벤트 핸들러 바인딩
  - [ ] `interlude-ui.integrated.test.ts` 에서 질문 인입 -> 사용자 입력 -> 락 해제 완료 통합 시나리오 테스트 성공
