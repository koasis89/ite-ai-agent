Oh-My-Codex (OMX) 시스템의 핵심 오케스트레이션 및 다중 에이전트 제어 엔진인 **`src/agents/` (에이전트 페르소나 및 오케스트레이션)**, **`src/hooks/` (인텐트 라우팅 파이프라인)**, 그리고 **`src/adapt/` (자가 적응형 기억 엔진)** 모듈의 통합 분석 내용을 기반으로, 시스템의 전체 동작 메커니즘을 유기적으로 결합한 통합 상세 설계 문서 (Integrated Architecture & Detailed Design Specification)를 작성했습니다.


# Oh-My-Codex (OMX) 오케스트레이션 및 다중 에이전트 서브시스템 통합 상세 설계 문서

**Integrated Design Specification: Hooks Pipeline, Agent Personas, and Self-Adaptive Memory Engine**

## 1. 시스템 개요 및 아키텍처 철학 (System Overview & Philosophy)

Oh-My-Codex (OMX)는 사용자의 고차원적 목표(Goal)를 분석하여 자율적으로 소프트웨어를 설계, 구현, 검증하는 **다중 에이전트 오케스트레이션 프레임워크**입니다. 본 시스템은 복잡한 상태 제어 논리(TypeScript)와 정적 행동 지침(Markdown)을 철저히 분리하여 설계되었습니다. 이를 통해 제어 엔진의 수정 없이도 에이전트의 페르소나와 가이드라인을 유연하게 확장할 수 있는 아키텍처를 구현합니다.

### 1.1 핵심 서브시스템의 삼각 관계

OMX의 핵심 구동체는 다음과 같은 3대 엔진의 유기적인 결합으로 가동됩니다.

1. **인텐트 및 라우팅 엔진 (`src/hooks/`)**: 사용자 입력을 감지하고 작업 규모를 분류하여 최적의 실행 레인으로 바인딩합니다.
2. **다중 에이전트 코어 (`src/agents/`)**: 기획자(Planner), 실행자(Executor), 검증자(Verifier) 등의 분업화된 에이전트 페르소나가 상호 협력하고 합의하는 실행 레이어입니다.
3. **자가 적응형 기억 엔진 (`src/adapt/`)**: 매 턴의 결과와 사용자 피드백을 분석하여 프로젝트 특화 규칙 및 기술 스택을 자율 학습하는 인공지능의 해마 역할을 담당합니다.


## 2. 서브시스템별 상세 설계 (Component Detailed Design)

### 2.1 인텐트 파악 및 라우팅 파이프라인 (`src/hooks/`)

사용자의 프롬프트가 유입되면 `codex-native-hook.ts` 진입점을 시작으로 정밀하게 조율된 라우팅 체인이 기동됩니다.

```
[사용자 입력 프롬프트]
         │
         ▼
┌────────────────────────────────────────────────────────┐
│ 1단계: keyword-detector.ts (명시적/암시적 키워드 스캔)   │
│   - $autopilot, $ralph, $deep-interview 등 변형 정규화 │
└────────────────────────────────────────────────────────┘
         │
         ├─► [키워드 감지 성공]: 스킬 활성화 및 상태 기록 후 즉시 라우팅
         │
         ▼ [키워드 미감지 시 폴백(Fallback)]
┌────────────────────────────────────────────────────────┐
│ 2단계: triage-heuristic.ts (11단계 선점형 결정 트리)    │
│   - PASS: 단답형 인사, 명시적 오프아웃(just chat)      │
│   - LIGHT: 10단어 이하 탐색(?), 15단어 이하 소형 편집  │
│   - HEAVY: 5단어 초과 명령형 동사 스타터 및 복합 구현  │
└────────────────────────────────────────────────────────┘
         │
         ▼
┌────────────────────────────────────────────────────────┐
│ 3단계: task-size-detector.ts (작업 규모 최종 검증)     │
│   - small(≤50단어), medium(51~200단어), large(>200단어) │
│   - Escape Hatch 접두사(quick:, tiny:) 감지 시 small 강제 │
└────────────────────────────────────────────────────────┘

```

* **`keyword-registry.ts` (단일 진실 공급원 - SSOT)**: 약 50여 개의 고유 명령어 및 유도 패턴의 우선순위를 정의합니다. 경합 발생 시 `compareKeywordMatches()` 정렬 체인에 의해 우선순위 내림차순 $\rightarrow$ 매칭 길이 내림차순 $\rightarrow$ 사전식 알파벳 오름차순으로 정렬되어 결정론적 라우팅을 보장합니다.
* **`triage-config.ts` (Fail-Closed 안전장치)**: 전역 설정 파싱 실패 나 변형 발생 시, 시스템 오작동 확산을 차단하기 위해 Triage 모드를 즉시 비활성화(`false`) 상태인 `invalid`로 강제 격리 조치합니다.

### 2.2 다중 에이전트 협업 및 오케스트레이션 코어 (`src/agents/`)

`src/agents/` 모듈은 역할 기반 제어를 구체화하며, `prompts/` 디렉토리의 역할 마크다운 지침을 기반으로 개별 페르소나를 모델링합니다.

#### 2.2.1 핵심 에이전트 페르소나 매트릭스

* **Planner (기획/설계 에이전트 - `planner.md`)**:
* **책임**: 고차원적 구현 목표를 수신하여 아키팩트(PRD, Architecture Spec, Task Breakdown)를 설계합니다.
* **원칙**: 결과 중심적 사고("outcome-first")를 엄수하며, 전체 코드베이스의 의존성 및 정합성을 선제적으로 고려합니다.


* **Executor (실행/구현 에이전트 - `executor.md`)**:
* **책임**: Planner가 정의한 Task Breakdown 단위에 따라 실제 파일 생성, 소스 코드 수정, 리팩토링 및 도구 실행을 전담합니다.
* **원칙**: 불필요한 공백이나 컨텍스트 훼손 없이 정확한 델타 패치(Delta patch)를 수행합니다.


* **Verifier (검증/테스트 에이전트 - `verifier.md`)**:
* **책임**: Executor의 산출물이 기획서의 요구사항을 충족하는지 근거 중심(Grounded & Verified)으로 전수 검증합니다.
* **원칙**: 단위 테스트 빌드, linter 구동, 정적 코드 분석, 런타임 실행 결과 파싱 등의 실증적 도구를 강제 집행합니다.



#### 2.2.2 무거운 오케스트레이션 및 스킬 워크플로우

* **Autopilot Mode**: 복수의 에이전트 페르소나 간 자율 순차/병렬 협업 루프를 제어하며, 장기 목표가 달성될 때까지 단계를 스스로 갱신합니다.
* **Ralph Mode**: "Don't stop, keep going" 원칙에 따라 기술적 블로킹(Blocking)이 발생하기 전까지 사용자 개입 없이 연속적인 컨텍스트 자율 최적화 루프를 수행합니다.
* **Deep-Interview Mode**: 사용자의 입력 모호성이 임계치를 초과할 때 기동됩니다. 입력 잠금(`input_lock`) 필드를 활성화하여 사용자의 성급한 승인 문자(`yes`, `y`, `proceed`)를 일시 차단하고, 의도가 완전히 명확해질 때까지 심층 질의 응답 라운드를 강제 유지합니다.

### 2.3 자가 적응형 기억 엔진 (`src/adapt/`)

에이전트가 시간이 흐름에 따라 프로젝트의 암묵적 룰을 학습하고 진화할 수 있도록 지원하는 기억 시스템입니다.

```
[턴 완료 이벤트 (turn-complete)]
               │
               ▼
┌────────────────────────────────────────────────────────┐
│ tech-stack-detector.ts (환경 분석)                    │
│   - package.json 의존성 및 소스 import 구문 실시간 스캔 │
└────────────────────────────────────────────────────────┘
               │
               ▼
┌────────────────────────────────────────────────────────┐
│ rule-extractor.ts (규칙 추출)                           │
│   - 자연어 프롬프트 지시 및 코드 변경점(diff)의 특징 분석 │
└────────────────────────────────────────────────────────┘
               │
               ▼
┌────────────────────────────────────────────────────────┐
│ feedback-loop.ts (피드백 반영)                          │
│   - 사용자의 부정적 피드백을 '부정적 제약 조건'으로 변환   │
└────────────────────────────────────────────────────────┘
               │
               ▼
┌────────────────────────────────────────────────────────┐
│ project-memory.ts & context-optimizer.ts (압축 및 저장) │
│   - 원자적 결합을 통해 project-memory.json 에 영구 각인 │
│   - 사용 빈도가 낮고 오래된 컨텍스트 삭제 (Eviction)      │
└────────────────────────────────────────────────────────┘

```


## 3. 동적 컨텍스트 오버레이 파이프라인 (Runtime Overlay Pipeline)

고정된 시스템 프롬프트 템플릿인 `AGENTS.md`에 실시간 프로젝트 스냅샷과 세션 메타데이터를 결합하기 위해 멱등성이 보장되는 마커 시스템을 운영합니다.

### 3.1 마커 시스템 및 오버레이 인젝션 구조

`agents-overlay.ts`는 파일 내의 특수 주석 마커 쌍을 탐색하여 컨텍스트를 주입합니다.

* **일반 세션 마커**: `$\sim$`
* **팀 수행원 세션 마커**: `$\sim$`

주입되는 동적 데이터 패키지는 다음 요소를 압축 통합합니다.

1. `codebase-map.ts`가 생성한 고속 코드베이스 구조체 스냅샷 (최대 1,000자 제약)
2. `.omx/notepad.md`에 기록된 개발자의 최우선 순위 메모 사항
3. `.omx/project-memory.json`에서 로드한 학습된 기술 스택 및 제약 조건 요약본
4. `explore-routing.ts`가 분석한 로컬 탐색용 명령어(`omx explore`) 분기 유도 지침

### 3.2 배타적 동시성 제어 (Exclusive File Locking)

다중 비동기 프로세스나 외부 MCP 도구가 `AGENTS.md`를 동시에 수정하여 손상시키는 문제를 방지하기 위해 파일 시스템 기반의 배타적 디렉토리 락(`agents-md.lock/`) 매커니즘을 적용합니다. 좀비 락(Lock) 상태를 감지하기 위해 기존 락 소유자의 PID 생존 여부를 `process.kill(pid, 0)`으로 감지하여 비정상 사멸 시 자동으로 락을 회수 및 파기합니다.


## 4. 데이터 모델 및 트랜잭션 스키마 (Data Model Specification)

### 4.1 스킬 활성 상태 스키마 (`SkillActiveState`)

에이전트의 현재 행동 단계와 잠금 상태를 추적하는 상태 파일의 코어 구조입니다.

```typescript
interface SkillActiveState {
  version: 1;                                    // 상태 스키마 버전 사양
  active: boolean;                               // 현재 스킬의 활성화 플래그
  skill: 'autopilot' | 'ralph' | 'deep-interview' | 'ralplan' | 'team'; // 활성 스킬 유니온
  keyword: string;                               // 스킬을 트리거한 원본 키워드 토큰
  phase: string;                                 // 스킬 내부의 세부 실행 단계 (e.g., 'intent-first', 'planning')
  activated_at: string;                          // ISO 8601 타임스탬프 기반 활성화 시점
  updated_at: string;                            // 최종 상태 변경 시점
  source: 'keyword-detector';                    // 상태를 기록한 주체 엔진 정보
  session_id?: string;                           // 바인딩된 고유 세션 식별자
  input_lock?: {                                 // 인터뷰 모드 전용 입력 차단 제어 객체
    active: boolean;
    scope: 'deep-interview-auto-approval';
    blocked_inputs: string[];                    // 무조건 차단할 입력어 리스트 (['yes', 'y', 'proceed'])
    message: string;                             // 차단 발생 시 사용자 인터페이스에 출력할 경고 메시지
  };
  deep_interview_config?: {                      // 인터뷰 실행 제약 조건
    profile: string;                             // 'strict' 등 평가 프로파일 명칭
    threshold: number;                           // 모호성 평가 통과 임계치
    max_rounds: number;                          // 최대 허용 허들 라운드 수
    enable_challenge_modes: boolean;             // 챌린지 모드 활성화 여부
  };
}

```

## 5. 마크다운 계약 검증 아키텍처 (Prompt Guidance Contract)

에이전트가 통제를 벗어나 위험한 파괴적 명령을 내리거나 무한 루프에 진입하는 것을 컴파일 및 실행 전 단계에서 원천 차단하기 위해 `prompt-guidance-contract.ts` 검증 구조를 운영합니다. 외부 마크다운 지침 파일들은 로딩 시 반드시 아래의 정규식 계약 모델 조건을 충족해야 작동 가능합니다.

```
[외부 마크다운 파일 로딩] ──► [prompt-guidance-contract.ts 실행]
                                           │
          ┌────────────────────────────────┴────────────────────────┐
          ▼ (정규식 계약 불일치)                                     ▼ (계약 조건 완벽 일치)
┌────────────────────────────────────┐                    ┌────────────────────────────────────┐
│ [컴파일/기동 거부 및 오류 격리]      │                    │ [정상 실행 컨텍스트 바인딩 승인]     │
│   - 시스템 비정상 폭주 예방        │                    │   - 런타임 오버레이 파이프라인 진입 │
└────────────────────────────────────┘                    └────────────────────────────────────┘

```

* **AUTO-CONTINUE 계약 검증**: `AUTO-CONTINUE.*clear.*already-requested.*low-risk` 패턴 검사를 통해 위험도가 낮고 사용자가 승인한 연속 작업에 한해서만 에이전트가 컨티뉴 루프를 수행할 수 있도록 명문화했는지 확인합니다.
* **ASK 제약 제어 검증**: `ASK only.*destructive.*irreversible.*credential-gated` 패턴을 매칭하여 영구적이거나 파괴적인 도구 사용, 혹은 보안 인증 장벽 진입 직전에만 사용자에게 확인을 요청하도록 한정했는지 강제 검증합니다.
* **검증 실증성 계약**: `task is grounded and verified` 문구 추적을 통해 Verifier 에이전트가 추측성 판단을 내리지 못하도록 원천 차단합니다.


## 6. 제어 임계값 및 운영 정책 요약 (System Thresholds & Policies)

OMX 프레임워크의 자원 고갈(토큰 오염) 방지와 연산 안정성 제어를 위한 핵심 수치적 명세 가이드라인입니다.

| 제어 항목 식별자 | 명세 값 | 작동 레이어 및 물리적 의미 |
