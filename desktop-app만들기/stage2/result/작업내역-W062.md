## 2. 목표 (DoD 요약)

- Renderer 가 보유한 첨부를 main 으로 옮길 때 **binary blob 을 IPC 로 흘리지 않는다**.
- 원본을 `%APPDATA%/oh-my-codex/attachments/<id>/<safe-basename>` 로 복사한 후 path/메타만 반환.
- basename sanitize + path traversal/symlink/SHA-256/size 4중 무결성 가드.
- 7일 경과 미사용 첨부 자동 만료 + 부트 시 고아 디렉터리 정리.
- `attachment_register / get / delete / list` 4종 IPC + zod 스키마 + accept/reject 회귀.

## 4. 핵심 API · 정책

### 4.1 IPC 4종

| 명령 | 인자 | 반환 |
|---|---|---|
| `attachment_register` | `args[0]` = JSON `{ name, mime, size, sourcePath?, base64?, sha256?, messageId? }` (페이로드 ≤ 5MB) | `{ id, path, sha256, size, name }` |
| `attachment_get` | `args[0]` = id | `{ row }` (binary 미반환) |
| `attachment_delete` | `args[0]` = id | `{ id, removed }` |
| `attachment_list` | `args` = `[messageId?, limit?, offset?]` | `{ rows, limit, offset }` |

### 4.2 보안 가드 (4중)

1. **basename sanitize**: 마지막 segment + `[^\w.\-]→_` + 100자 상한 + 빈/`.`/`..` → `file`.
2. **path traversal**: `path.relative(root, final)` 검증 — `..` 시작 / 절대 / 빈 문자열 거부.
3. **symlink 거부**: sourcePath 는 절대 경로 + `fs.realpath` 가 원경로와 동일해야 함.
4. **SHA-256 + size 무결성**: main 재계산 결과가 caller 선언 값과 일치해야 함.

### 4.3 TTL + 정리

- 기본 만료: `createdAt + 7d` (`AttachmentRepoOptions.ttlMs` 로 override).
- 부트 시 `cleanupExpired(now)` + `cleanupOrphans()` 자동 호출.
- 부트 로그: `[attachments] bootstrap ok (expired=N, orphans=M)`.

### 4.4 마이그레이션 슬롯 deviation

- 기획 문서(W062 티켓)는 `004_attachments.sql` 로 명기했으나, 실제 슬롯 `004` 는 WIN-054(`004_tool_calls_v2.sql`) 가 점유 중.
- **결정**: `005_attachments.sql` 로 충돌 회피. 매뉴얼 §8.2 와 티켓 체크리스트에 deviation 주석 명기.

## 6. 후속 / Known Limits

- 동일 sha256 dedup(refCount) 은 차후 최적화로 보류 (현재는 등록마다 새 디렉터리 생성).
- `attachment_list` 응답은 메타만 — binary 는 WIN-063 의 `attachment_read_text`/`file://` CSP 로 분리.
- `tool_call` 등 외부 도구 호출 시 첨부 path 를 인자로 전달하는 결합은 WIN-064 에서 처리.
- LocalProcessTransport 의 `allowedCwdRoots` 에는 첨부 디렉터리를 **의도적으로 미포함** (read-only intent) — WIN-044 legacy state 와 동일한 패턴.
