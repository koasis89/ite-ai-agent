## 2. 목표

Phase 4 채팅 UI 가 의존하는 **TUI/색상(ANSI)/리사이즈/raw-input** 지원 워커 transport 를 ConPTY(`node-pty`) 위에 구현한다.
단, `node-pty` 는 **네이티브 빌드 의존성**(Windows 의 경우 VS Build Tools / Python) 을 요구하므로
**미설치 / 빌드 실패 환경에서도 데스크탑 앱이 정상 부팅** 되어야 한다 → optionalDependency + capability fallback 아키텍처를 채택한다.

## 4. 신규 API 요약

### `WorkerTransport` 인터페이스 확장

```ts
interface WorkerTransport {
  // ...기존
  resize?(cols: number, rows: number): void;  // WIN-030
}
```

### `PtyLocalTransport` 표면

```ts
class PtyLocalTransport implements WorkerTransport {
  readonly kind: 'pty-local';
  constructor(opts?: PtyLocalTransportOptions);
  isAvailable(): boolean;                      // capability 캐시 기반
  start(spec: WorkerTransportStartSpec): Promise<void>;
  send(data: string): Promise<void>;           // raw input → pty.write
  resize(cols: number, rows: number): void;    // SIGWINCH 등가
  onOutput(h): Unsubscribe;                    // ANSI 보존, stream='stdout' 로 통합 전달
  onExit(h): Unsubscribe;
  stop(): Promise<void>;                       // 멱등, SIGTERM→grace→SIGKILL
  isAlive(): boolean;
}

// capability helpers
loadPtyModule(loader?): Promise<MinimalPtyModule | null>
isPtyTransportAvailable(loader?): Promise<boolean>
tryRegisterPtyLocalTransport(registry, opts?, setAsDefault?): Promise<boolean>
```

### Capability / Fallback 매트릭스

| 조건 | 결과 default kind | 비고 |
|---|---|---|
| `OMX_DESKTOP_FORCE_LOCAL_PROCESS=1` | `local-process` | 운영 측 강제 우회 |
| `node-pty` 로드 성공 | `pty-local` | TUI/색상/리사이즈 가능 |
| `node-pty` 로드 실패 (미설치 또는 ABI 불일치) | `local-process` | 자동 fallback, 부팅 차단 없음 |

## 7. 후속 / 한계

1. **수동 GUI 검증** — xterm.js 패널 합류 시점(WIN-031/032/033) 에 TUI 워커 1회 기동 → 색상 보존 / 리사이즈 / 키 입력 echo 시각 확인. (체크리스트 [ ] 유지)
2. **prebuilt 빌드 검증** — `node-pty` 가 실제 설치된 환경에서 `electron-builder --win` 으로 빌드 후 `node-pty` 네이티브 모듈이 asarUnpack 으로 풀려 require 되는지 통합 테스트. 현재 PR 범위에서는 optionalDependency 미설치 상태로 둠.
3. **Electron ABI 매칭** — Electron 35.7.5 와 `node-pty@1.x` 의 ABI 호환성은 `@electron/rebuild` 로 보장. 실제 빌드 머신에서 `electron-rebuild -f -w node-pty` 실행 필요 (후속 패키징 워크플로우에 추가 예정).
4. **stderr 처리** — node-pty 는 stdout/stderr 가 합쳐진 PTY 한 채널이므로 `stream: 'stdout'` 으로 일괄 라우팅. 옵션 A 의 별도 stderr 채널과 약간의 의미 차이가 있으나, 상위 채팅 UI 는 둘을 동일 토큰 스트림으로 취급하므로 영향 없음.
