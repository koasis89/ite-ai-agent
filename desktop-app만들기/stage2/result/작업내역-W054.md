## 4. 핵심 API · 정책

### 4.1 마이그레이션 004 (`tool_calls` 재구축)
- 기존 `tool_calls` 행을 `tool_calls_v2` 로 INSERT SELECT 마이그레이션:
  - `tool_id := command CONTAINS ':' ? command : 'builtin:' || command`
  - `status := CASE WHEN exit_code = 0 THEN 'ok' WHEN exit_code IS NULL THEN 'ok' ELSE 'failed' END`
  - `stdout_truncated/stderr_truncated := 0`, `permission_* := NULL`, `error_reason := NULL`
- `DROP tool_calls` → `ALTER RENAME tool_calls_v2 → tool_calls` 후 인덱스 3개 재생성. FK `message_id → messages(id) ON DELETE CASCADE` 보존 (WIN-042 cascade 회귀 안정).

### 4.2 ToolCallHistoryRepo
- `record({id, messageId?, toolId, command, args, exitCode?, durationMs?, stdout?, stderr?, stdoutTruncated?, stderrTruncated?, status, permissionDecision?, permissionScope?, errorReason?, createdAt?})`
- `list({limit?<=200, toolId?})` — `created_at DESC, id DESC` 정렬
- `getById(id)` / `count()` — prepared statement 사용
- `args` 는 JSON 문자열로 직렬화 저장, raw row 복원 시 안전 파싱(비문자열 원소 거부 → 빈 배열 fallback)

### 4.3 ToolRouter (`desktop/main/tools/tool-router.ts`)
- 구성: `{ catalog?, broker?, repo?, executors: { omx?, mcp? }, maskSensitive?, now? }`
- `call({toolId, args, sessionId?, messageId?}, hooks?)` 흐름:
  1. `parseToolId('<server>:<name>')` — 잘못된 형식이면 `Error` throw (IPC 단에서 `COMMAND_FAILED` 로 흡수)
  2. catalog hit + `enabled=false` → `persistDenied('tool-disabled')` + `status='denied'` 즉시 반환 (executor 미호출)
  3. broker hit + decision='deny' → `persistDenied('permission:<source>', 'deny', source)` + `status='denied'` + `permissionDecision='deny'`
  4. `server === 'builtin'` → `executors.omx(name, args, hooks)`; `timedOut` 이거나 `exitCode≠0` 이면 `status='failed'` + `errorReason='timeout'|'exit N'`
  5. 그 외 → `executors.mcp(server, name, args, hooks)` → `status='ok'`, `result` 그대로 전달. `progress` chunk 는 stdout 컬럼에 누적 저장
  6. catch — `status='failed'` + `errorReason=err.message`, `historyId` 보존
- `replay(historyId, overrides?, hooks?)` — `repo.getById(historyId)` 의 `toolId`/`args` 로 `call()` 재진입. broker 가 다시 통과.
- `list({limit?, toolId?})` — repo 위임.
- `historyId = tc-${now}-${randomUUID().slice(0,8)}` 형식. IPC 단에서 `^tc-[0-9]{10,16}-[0-9a-f]{4,32}$` 정규식으로 검증.
- 마스킹: argsJson 직렬화 전 각 인자에 `defaultMaskSensitive` 적용(또는 옵션 주입). 보안 영구 정책: stdout/stderr 8KB 상한 + `truncated` 플래그는 executor outcome 에서 가져옴.

### 4.4 IPC 명령(3종)
- `tool_call` — `args=[toolId(TOOL_ID_RE), argsJson]`. `argsJson` 은 `<=8KB` + `JSON.parse → string[]` 이며 길이 `<=10`, 각 원소 `<=1024 chars`. `toolRouterBackend.call({toolId, args, messageId})` 위임. 결과: `{command, ok, toolId, status, exitCode, durationMs, stdout?, stderr?, stdoutTruncated, stderrTruncated, result, permissionDecision, permissionScope, errorReason, historyId}`. backend 미주입 시 `COMMAND_FAILED`.
- `tool_history_list` — `args=[limit?(1..200), toolId?(TOOL_ID_RE)]`, limit 기본 50. 결과: `{command, limit, toolId|null, entries: ToolCallRow[]}`.
- `tool_history_replay` — `args=[historyId(TOOL_HISTORY_ID_RE)]`. broker 가 다시 통과해야 실행됨. backend 미주입 시 `COMMAND_FAILED`.

### 4.5 슬래시 라우터 통합 (`/tool`)
- `SlashSpec.transformArgs?: (rest: readonly string[]) => string[]` 훅을 추가해 잔여 토큰을 IPC args 로 가공.
- `/tool <toolId> [args...]` → `command='tool_call'`, `args=[toolId, JSON.stringify(rest.slice(1))]` 으로 직렬화 — `tool_call` 의 (toolId, argsJson) 형태 정합.
- `tool_history_list` / `tool_history_replay` 는 슬래시 미노출 (`SLASH_EXCLUDED`) — 이력 패널 전용.

### 4.6 Streaming bridge 정합
- `ToolCallHooks.onChunk(stream: 'stdout'|'stderr'|'mcp-progress', chunk, truncated)` 인터페이스로 일원화. main bootstrap 단계에서 hook → `eventBus.publish('command.progress', { channel: stream, chunk, truncated })` 으로 wiring (WIN-034 streaming 채널 재사용).
