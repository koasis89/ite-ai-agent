## 4. 핵심 API · 정책

### 4.1 PermissionStore
- `set({toolId, sessionId?, decision, grantedAt?, expiresAt?})` — `(tool_id, COALESCE(session_id,''))` UNIQUE 충돌 시 UPDATE (decision/granted_at/expires_at).
- `check(toolId, sessionId?)` — 우선순위 매트릭스:
  1. 영구 deny → 즉시 반환
  2. 영구 allow → 후보로 보류 (세션 deny 가 있으면 그것이 우선)
  3. 세션 deny (만료 미경과) → 즉시 반환
  4. 세션 allow (만료 미경과) → 영구 allow 있으면 그것을, 없으면 세션 allow
  5. 모두 미설정 → `null`
- `list()` — `granted_at DESC` 정렬
- `revoke(toolId, 'session'|'permanent'|'all')` — 삭제 행 수 반환
- `purgeExpired()` — `expires_at <= now` 인 세션 행 일괄 제거

### 4.2 PermissionBroker
- 입력: `{toolId, server, toolName, description?, argsSummary?, sessionId?}`
- 흐름:
  1. `toolId.startsWith('builtin:omx_')` && `omxAllowExecOverride() === false` → 즉시 `{decision:'deny', source:'override-deny'}` (WIN-023 `OMX_DESKTOP_ALLOW_EXEC=0` 흡수)
  2. store.check → hit 면 `{decision, source: hit.scope}`
  3. miss → `prompt(input)` 호출 후 scope 별 영속:
     - `permanent` → `session_id=null`, `expires_at=null`
     - `session` → `session_id=<sid>`, `expires_at=now+sessionTtlMs` (기본 8h). sessionId 부재 시 once 로 강등
     - `once` → 영속 없음, 결과만 반환
  4. prompt 콜백 throw → 안전한 deny-once
- `onChange(event)` — granted/denied 시 system 채팅 메시지 출력 용도(상위 layer 가 wiring).

### 4.3 PermissionDialog
- 4선택지 버튼(once-allow / session-allow / permanent-allow / once-deny), ESC = once-deny.
- `maskArgsSummary(input)` helper — `password|passwd|secret|api[_-]?key|token|authorization` 키 값을 `key=***` 로 마스킹.
- 인자 요약은 호출자가 사전 마스킹한 문자열만 받음 (raw 보관 금지).

### 4.4 IPC
- `tool_permission_list` — 인자 0개, `{command, permissions:[{id,toolId,sessionId,decision,grantedAt,expiresAt}]}` 반환. backend 부재 시 빈 배열.
- `tool_permission_revoke` — 인자 `[toolId, 'session'|'permanent'|'all']`, `{command, toolId, scope, deleted}` 반환. backend 부재 시 `COMMAND_FAILED: permission backend not initialised`.
- 두 명령 모두 슬래시 매핑에서 제외 (`SLASH_EXCLUDED`) — 설정 UI 전용.

### 4.5 omx_* 통합
- 기존 `isOmxExecAllowed()` 조기 차단은 유지 (`OMX_DESKTOP_ALLOW_EXEC=0` → `exec-disabled` 메시지). 기존 ipc-contract.test 의 거부 케이스 호환.
- 추가로 `permissionBackend?.check` 가 주입된 경우 broker 통과를 강제. deny 시 `COMMAND_FAILED: ${command}: permission denied (${source})`. backend 부재(테스트/단독) 시 기존 흐름 유지.
