## 4. 핵심 API · 정책

### 4.1 PermissionBroker catalog 게이트 (W052#2 해소)
- 우선순위 (높음 → 낮음): **catalog 비활성** → env override → store 영구 → store 세션 → prompt
- 게이트 실패 시:
  - `result = { decision: "deny", source: "override-deny" }`
  - `onChange({ kind: "denied", toolId, scope: "once" })` 통지
  - store / prompt 진입 **없음** (호출 영속도 일어나지 않음)
- 회귀: `permission-broker-catalog-gate.test.ts` 4건
  1. 비활성 즉시 deny + prompt 미호출
  2. 활성 시 기존 prompt 경로 정상 (`once / allow`)
  3. catalog 미주입 시 게이트 비활성화 (기존 동작 유지)
  4. catalog 비활성이 `omxAllowExecOverride=true` 보다 우선

### 4.2 `tool.permission.changed` 이벤트 (W053#3 해소)
- `OmxEventType` 16번째 타입으로 등록 (기존 15종 보존)
- detail: `{ kind: "granted"|"denied", toolId, scope: "permanent"|"session"|"once" }`
- 발행 시점: PermissionBroker 가 grant 영속(`set`) 직후 + 게이트/once-deny 시점
- 발행 주체: 메인 프로세스 (`bootstrapToolStack` 에서 broker `onChange` 콜백)

### 4.3 `bootstrapToolStack()` 결선 (W052#4 + W054#2 해소)
1. SQLite 가 초기화되지 않은 경우: 3 backend 모두 `null` 로 set (Phase 4 이전 호환 폴백). 로그 경고 없음 (silent).
2. `ToolEnabledStore` 로딩 (`userData/mcp-tools.json`)
3. `mcpServerRegistry = null` (v1 config schema 미정의 — Phase 7 확장 여지)
4. `ToolCatalogStore` 생성 — `enabledRepo` + `mcpSource` (registry 부재 시 `[]`)
5. `setToolCatalogBackend` 결선 (`list` / `search` / `get` / `toggleTool`)
6. `PermissionStore` (SQLite 003 migration) 생성 + `PermissionBroker` 결선:
   - `catalog.isEnabled(id)` → 카탈로그 게이트
   - `omxAllowExecOverride` → env `OMX_ALLOW_EXEC` 체크
   - `prompt` → 현재는 deny-once 폴백 (1회 경고 로그; Phase 7 다이얼로그 대체 예정)
   - `onChange` → `eventBus.publish(EventBus.event<ToolPermissionChangedDetail>(...))`
7. `setPermissionBackend` 결선 (list / revoke / check pass-through)
8. `ToolCallHistoryRepo` (SQLite 004) + `ToolRouter` 결선:
   - executors.omx: `omxCliMatrix[name]` 으로 인자 prefix → `runOmxSubcommand` 스트리밍
   - executors.mcp: `mcpServerRegistry.callTool(server, name, kwargs)` (args[0] = JSON object string 컨벤션)
9. `setToolRouterBackend` 결선 — `hooks.onChunk` 매핑 (`stdout|stderr` → 그대로 / 그외 → `info`)

### 4.4 ToolRouter 스트리밍 (W054#3 해소)
- `tool_call` / `tool_history_replay` switch 가 `parsed.data.messageId` 가 있으면 `hooks` 를 합성:
  - 도구 청크 발생 시 `bumpEvent("command.progress")` + `eventBus.publish(EventBus.event<CommandProgressDetail>("command.progress", {stream, chunk, messageId, truncated}, commandId))`
- backend boundary 매핑: `tool-router` 내부 `"stdout"|"stderr"|"mcp-progress"` → events.ts 호환 `"stdout"|"stderr"|"pane"|"info"` (`mcp-progress` → `info`)

### 4.5 ToolHistoryPanel 마운트 (W054#4 해소)
- 호스트: `#tool-history-mount` (renderer/index.html, `data-win="WIN-054"` 태그)
- 마운트 위치: `app.ts` 의 세션 리스트 초기화 직후 (DOM 준비 보장)
- IPC 어댑터: `window.omx.runCommand` 를 `ToolHistoryPanel` 의 `ipc.runCommand` 시그니처에 맞게 어댑팅 (`readonly string[]` → 스프레드 `[...req.args]`)
- 토스트: `onToast` 콜백을 타임라인 `addTimeline(...)` 으로 라우팅
