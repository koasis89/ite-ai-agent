# WIN-064 — 채팅 메시지 첨부 슬롯 + 도구 호출 인자 전달

## 1. 식별

- 티켓: WIN-064 (Phase 7 / Sprint SP-07 / Epic EP-02)
- 우선순위: P0 — Phase 7 최종 사용자 가치 마무리
- 선행: WIN-061 (drag-drop UI), WIN-062 (attachment-repo + IPC 4종), WIN-063 (프리뷰), WIN-054 (`tool_call`), WIN-031 (메시지 ViewModel)
- 기준 브랜치: webapp

## 2. 목표

1. InputBox 의 미확정 첨부를 메시지 전송 시점에 확정(승격)한다.
2. 채팅 메시지 버블에 첨부 칩을 렌더한다.
3. `@<attachmentId>` 토큰을 도구 호출 인자에서 해석해 경로(또는 메타데이터 객체)로 치환한다.
4. 권한 다이얼로그 / 호출 이력 패널에 첨부 메타를 노출한다.
5. 세션 소유권 / 미해석 토큰 / 잘못된 id 에 대한 보안 회귀를 통과한다.

## 3. 구현 작업

### 3.1 작업 내용

- `attachment-token-resolver.ts` (신규, pure): `parseAttachmentToken` / `hasAttachmentTokens` / `resolveAttachmentTokens` + `AttachmentTokenError` (코드 `INVALID_TOKEN` / `ATTACHMENT_NOT_FOUND` / `FOREIGN_SESSION`). 매니페스트 hint(`string` / `object`)에 따라 경로 또는 JSON 메타로 치환.
- `message-attachment-link.ts` (신규, pure renderer): `extractAttachmentTokens`, `formatAttachmentLabel`, `renderAttachmentChips` (DOM 헬퍼, `innerHTML` 미사용).
- `AttachmentRepo.linkToMessage(id, messageId)` 추가: `UPDATE attachments SET message_id=? WHERE id=? AND message_id IS NULL` — 멱등성/재시도 안전.
- IPC `attachment_link_message` 추가 (allowedCommands 27 → 28). `AttachmentLinkMessageArgsSchema = z.tuple([attachmentId, messageId(<=200, [A-Za-z0-9_.-])])`. `handleRunCommand` 케이스에서 `attachmentBackend.linkToMessage(...)` 호출 후 `{ ok: linked, data: { command, id, messageId, linked } }`.
- `ToolRouter` 에 `attachmentResolver` 옵션 + `ToolCallInput.allowedMessageIds` 옵션 도입. `call()` 은 카탈로그 enabled 통과 직후 토큰이 있으면 resolver 호출 → 실패 throw 시 `persistDenied` 로 `denied` 이력 기록 + executor 미호출. 성공 시 `effectiveInput` 이 broker.check 인자 요약과 executors(omx/mcp)에 전달.
- `desktop/main/index.ts` 의 `setAttachmentBackend({ … })` 에 `linkToMessage` 와이어링 + `new ToolRouter({ attachmentResolver: … })` 와이어링 (resolver 안에서 `resolveAttachmentTokens({ backend: { get: id => attachmentRepo.get(id) }, allowedMessageIds })` 호출).
- `ChatMessage` 인터페이스 확장: 선택적 `attachments?: ReadonlyArray<{id, name, mime, size}>` + `ChatViewModel.append(role, content, attachments?)` 시그니처 확장 (기존 호출자 호환).
- `ChatPanel.renderMessage`: 메시지 본문 다음에 `<ul class="chat-msg-attachments" data-testid="chat-msg-attachments">` 칩 리스트 렌더 (모든 속성/본문 `escapeAttr` 통과).
- `ToolHistoryPanel.RawHistoryRow` 확장: 선택적 `attachments?: ReadonlyArray<{id, name}>` + 펼친 상세에 `attachments (N): <names>` 라인.
- `PermissionDialogPayload` 확장: 선택적 `attachments?: ReadonlyArray<{id, name}>` + `addMeta("첨부", "N건: <names>")` 행.

## 4. 검증 작업

### 4.1 검증 내역

- `npm run build:desktop` — TypeScript 컴파일 + 마이그레이션 자산 복사 (오류 0).
- `npm run test:phase2:common:compiled` — Phase 2 공통 게이트 재실행.
- 신규 테스트 `tool-call-attachment.test.ts`:
  - driver-independent: 토큰 패턴 파싱, `hasAttachmentTokens`, 정상 해석, `object` 힌트 JSON 직렬화, `INVALID_TOKEN` / `ATTACHMENT_NOT_FOUND` / `FOREIGN_SESSION` 거부, `allowedMessageIds` 통과, `ToolRouter` resolver 성공 / resolver throw → `denied`.
  - DB-backed (itDb): `AttachmentRepo.linkToMessage` 정상 갱신 + 멱등성(중복 호출 false), 잘못된 입력 false, 실제 첨부 등록 후 `ToolRouter` 가 절대 경로로 치환해 executor 호출(e2e).
