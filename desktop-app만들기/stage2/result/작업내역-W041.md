## 4. 핵심 API · 정책

### 4.1 스키마 v1

```ts
ConfigV1 = {
  schemaVersion: 1,
  ui: { theme: 'light'|'dark'|'system', fontSize: number(8..48) },
  transport: { preferred: 'pty'|'local-process' },
  exec: { omxAllowed: boolean },
  telemetry: { optIn: boolean },
}
DEFAULT_CONFIG_V1 = { ..., ui: { theme: 'system', fontSize: 14 }, transport: { preferred: 'pty' }, exec: { omxAllowed: true }, telemetry: { optIn: false } }
```

- 모든 객체 `strict()` — 미지 필드는 검증 실패 → 백업+복원.

### 4.2 ConfigStore.load() 결정 표

| 입력 상태 | 동작 | 결과 플래그 |
|---|---|---|
| 파일 없음 (ENOENT) | migration 0→target + atomic write | `created=true`, `migrated=[{0,1}]` |
| 유효 v1 (target=1) | 로드 only | `created=false`, `migrated=[]` |
| JSON 파싱 실패 | 백업(`.bak.<ts>`) + 기본값 write | `restored=true`, `backupPath` |
| `schemaVersion` 누락 | 백업 + 기본값 | `restored=true` |
| 스키마 v1 위반 | 백업 + 기본값 | `restored=true` |
| 미래 버전 (`> target`) | 백업 + 기본값 | `restored=true` |
| 구버전 (`< target`) | migration chain + atomic write | `migrated=[{from,to},...]` |
| 마이그레이션 throw | 백업 + 기본값 | `restored=true` |

### 4.3 보안 정책

- **외부 입력 차단**: `set()` 은 `SETTABLE_KEYS = ['ui','transport','exec','telemetry']` 화이트리스트만 허용. `'schemaVersion'`/`'__proto__'` 등은 `'key not allowed'` throw.
- **값 검증**: 모든 `set` 값은 `ConfigV1Schema.safeParse` 통과 후에만 디스크 반영. 실패 시 기존 값 유지.
- **atomic write**: `tmp + rename` 패턴 — write 중 크래시에도 원본 손상 없음, 동시에 `.tmp` 잔여 없음을 회귀로 검증.
- **백업 보존**: 손상 입력은 `config.json.bak.<timestamp>` 로 원형 그대로 보존 — 사용자가 수동 복구 가능.
- **다운그레이드 금지**: 미래 버전 입력은 백업+기본값. 마이그레이션 체인에서 명시적 throw.

### 4.4 마이그레이션 확장 패턴

```ts
// 새 vN 도입 절차:
// 1) schema.ts: CONFIG_SCHEMA_VERSION = N, ConfigVN_Schema 추가
// 2) migrations/index.ts: vNMigration { from: N-1, to: N, apply: (vN-1) => vN }
//    PRODUCTION_MIGRATIONS 끝에 추가 — 순서는 from 오름차순 권장.
// 3) ConfigStore.load() 의 v1 분기에 vN 검증 분기 추가.
// 4) config-store.test.ts 에 vN-1 → vN dry-run 회귀 + production 체인 길이 SSOT 갱신.
```
