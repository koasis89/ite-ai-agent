## 2. 목표

MCP 표준(JSON-RPC 2.0 over stdio) 서버를 spawn 해 도구 카탈로그(`tools/list`)와
호출(`tools/call`) 능력을 확보한다. 이후 WIN-052 ~ WIN-054 의 도구 UI / 권한 /
실행 라우터가 본 클라이언트를 사용한다.

## 4. 핵심 API · 정책

### 4.1 JSON-RPC 프레이밍 (Content-Length / LSP 호환)

```
Content-Length: <N>\r\n
\r\n
<N bytes of JSON>
```

- 인코더는 `Buffer.byteLength(body, "utf8")` 로 N 산정.
- 디코더는 누적 버퍼에서 헤더를 찾으면 `Content-Length` 만 추출(다른 헤더는 무시),
  본문은 **utf-8 byte 단위**로 정확히 N 바이트만 잘라낸 뒤 다음 메시지를 이어 처리.
- 가드: 헤더 4096자 초과 / Content-Length 누락·음수·비숫자 / 본문 1MB 초과 → 즉시 throw.
- 인바운드 1MB 초과 + 아웃바운드 1MB 초과 모두 거부.

### 4.2 McpClient

- 생성자 옵션: `{ transport: WorkerTransport, requestTimeoutMs?=30000, logger?, now? }`
- ID 발급: 1부터 증가하는 정수. string id 응답 수신 시 로그 후 무시.
- 응답 매칭: `pending: Map<number, {resolve,reject,timer,method}>`. 도착 시 `clearTimeout` + resolve/reject.
- 응답 zod 검증: `InitializeResultSchema` / `ToolsListResultSchema` / `ToolsCallResultSchema` 모두 호출자 측에서 `safeParse` 후 실패 시 throw.
- `initialize` 응답 수신 직후 표준에 따라 `notifications/initialized` 알림 자동 전송.
- transport 비정상 종료(`onExit`) 시 모든 pending 요청 reject.
- 프레이밍 오류 시 pending 일괄 fail + closed 상태 전환.

### 4.3 McpServerRegistry

- 부트: `enabled && autoStart` 매니페스트 순차 spawn (start → initialize → tools/list 캐시 → status=`running`).
- 실패 시: `restartCount` 증가, 다음 spawn 까지 delay = `backoffBaseMs * 2^attempt` (기본 1000 * 2^n → 1s, 2s, 4s).
- `restartCount >= maxRestarts` (기본 3) 도달 시 `status = "disabled"` + 더 이상 재시작하지 않음.
- 호출 시 `running` 아닌 서버는 거부.
- `transportFactory(manifest)` 주입 — 테스트는 FakeTransport, 운영은 `LocalProcessTransport` 인스턴스 생성기.
- `schedule(fn, ms)` 주입으로 결정적 단위 테스트 가능.

### 4.4 보안 정책 (불변 / 승계)

- `LocalProcessTransport` 가 강제하는 보안 가드 (`shell:false`, `allowedCommands` 화이트리스트, `allowedCwdRoots`, `envAllowList`, kill 3s grace) 100% 승계 — `McpServerRegistry` 는 transport 만 wrapping.
- `config.json.mcp.servers[].command` 는 `allowedCommands` 화이트리스트에 **자동 등록되지 않음**. 사용자가 명시적으로 허용해야 하며, 권한 UI 는 WIN-053 에서 제공.
- JSON-RPC 페이로드 1MB 초과 → 인/아웃바운드 모두 즉시 throw.
- 응답 zod 검증 실패 → 호출 거부 (`McpClient.listTools/callTool` 에서 throw).
