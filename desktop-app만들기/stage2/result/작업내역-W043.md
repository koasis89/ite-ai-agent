## 2. 목표

Phase 3 (WIN-021) 의 main-side in-memory 50건 ring buffer (`desktop/ipc/commands.ts` `commandHistory`) 를 SQLite 백엔드(`command_history` 테이블)로 교체하여 명령 실행 이력이 앱 재시작에도 보존되도록 한다. Phase 3 의 `history_list` IPC 응답 스키마 (`{ command, limit, total, entries[] }`) 와 한도(`1..200`) 는 100% 유지하고, 비밀번호·토큰 패턴은 직렬화 단계에서 마스킹한다. 기본 보존은 10,000 건이며 초과분은 LRU 로 삭제된다.

## 4. 핵심 API · 정책

### CommandHistoryRepo
- `record({command, args, status, commandId?, exitCode?, startedAt, completedAt?, durationMs?, errorReason?})` → 마스킹된 args 를 JSON 직렬화 후 insert, 이어서 `enforceMaxRows()`
- `list(limit, offset=0)` → `started_at DESC, id DESC` 정렬, 1..10000 한도, offset>=0
- `search({command?, from?, to?, limit?})` → 모두 nullable, prepared statement
- `purgeOlderThan(ts)` → `started_at < ts` 삭제, 영향 행수 반환
- `enforceMaxRows()` → 초과분만큼 `LIMIT N` 으로 가장 오래된 row 제거

### 마스킹 규칙
- 인라인: `\b(password|token|secret|api[_-]?key)\s*=\s*\S+` → `$1=***` (case-insensitive)
- 플래그: `--password|--token|--secret|--api_key|--api-key|--apikey` 다음 토큰 → `***`
- 단일 토큰 마스킹만 보장 (멀티-스페이스 분리 값은 호출 측 책임)

### `history_list` 응답 (불변)
```ts
{ ok: true, data: { command: "history_list", limit, total, entries: CommandHistoryEntry[] } }
```
- `entries[]` 는 newest-first (Phase 3 와 동일하게 `slice(-limit).reverse()` 의미)
- `limit` 검증 1..200 유지
