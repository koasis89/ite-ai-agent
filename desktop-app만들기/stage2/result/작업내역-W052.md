## 4. 핵심 API · 정책

### IPC 계약
- `tool_list(args[0]?=query)` → `{ command:'tool_list', query: string|null, tools: ToolDescriptor[] }`
  - `query` 가 비면 전체 목록. `tools` 정렬: builtin 그룹 우선, 그 후 server asc, name asc.
- `tool_toggle(args[0]=toolId, args[1]='on'|'off')` → `{ command:'tool_toggle', toolId, enabled: boolean }`
  - 검증: `toolId` 는 `^[a-z0-9][a-z0-9_-]{0,63}:[a-zA-Z0-9_][a-zA-Z0-9_.-]{0,255}$` (builtin: 와 외부 `<serverId>:<toolName>` 모두 커버).
  - 미등록 toolId → `COMMAND_FAILED` (`unknown toolId '...'`).

### 카탈로그 SSOT (`ToolCatalogStore`)
- `BUILTIN_TOOLS` 3종 — id `builtin:omx_*`, server `builtin`, origin `builtin`, schema `{type:'object', properties:{}}`.
- MCP 도구 id 규약: `<serverId>:<toolName>`. origin `mcp`.
- `enabled` 오버레이: `ToolEnabledRepository.isEnabled(id, defaultValue=true)`. 명시 false 만 비활성.
- `toggleTool(id, enabled)` — 미등록 id throw, 등록되면 repository 영속 + onChange 통지.
- `assertEnabled(id)` — main 측 호출 게이트(WIN-054 `/tool` 라우터에서 사용 예정).

### 영속 (`ToolEnabledStore`)
- 파일: `%APPDATA%/oh-my-codex/mcp-tools.json` (`{ schemaVersion:1, enabled:{ [toolId]: boolean } }`).
- atomic write (tmp + rename). JSON parse 실패/schema 불일치 → 백업 없이 기본값 복원(파일 자체가 보조 캐시이므로 lossy 허용).
- ConfigStore v1 strict 회피 정책: WIN-053 SQLite `tool_permissions` 도입 시 흡수 여부 재검토.

### 보안
- `tool_toggle` 의 `toolId` regex 화이트리스트 패턴 + catalog 화이트리스트 이중 가드.
- `on/off` 외 값은 zod enum 으로 `INVALID_REQUEST`.
- 비활성 도구의 실제 호출 거부는 WIN-054 (도구 실행 라우터)에서 `ToolCatalogStore.assertEnabled` 사용.
