## 3) 중복 확정 항목

### A. Sidecar 스냅샷 취득/검증 로직 이중 구현

현황:

- desktop/ipc/commands.ts 의 `sidecar_get_snapshot` 분기에서 teamName 검증 + `getSidecarSnapshot` 직접 호출
- desktop/main/index.ts 의 sidecar IPC 핸들러에서 teamName sanitize + `safeGetSidecarSnapshot` 호출

문제:

- 동일 책임(팀명 검증/스냅샷 취득/오류 fallback)이 두 경로에 중복됨
- 검증 규칙도 미세하게 다름
  - commands 쪽: 길이 제한(1..64) 포함
  - main 쪽: 문자셋만 검사하고 길이 상한 미명시

보완:

1. sidecar 공통 서비스 함수 1개로 통합
2. `commands.ts` 와 `main/index.ts` 모두 해당 함수 호출만 하도록 변경
3. 팀명 규칙을 단일 정규식으로 고정(문자셋 + 길이 상한)

우선순위: 높음

### B. 입력 토크나이저/파서 이중 구현

현황:

- desktop/renderer/app.ts: `parseCommandLine`
- desktop/renderer/chat/slash-router.ts: `tokenizeSlashInput`

문제:

- 둘 다 입력 분해기이지만 규칙이 다름
  - app 파서: single quote 지원, 미닫힘 인용을 관대 처리
  - slash 파서: double quote 중심, 미닫힘 인용 에러
- 동일 사용자 입력이 경로마다 다르게 처리될 수 있음

보완:

1. renderer 공용 tokenizer 모듈 신설(예: desktop/renderer/chat/tokenizer.ts)
2. slash/명령패널이 같은 파서 코어를 쓰고 옵션만 다르게 적용
3. 인용/이스케이프 규칙을 매뉴얼 한 곳에 명시

우선순위: 중간

## 4) 중복 위험 항목

### C. tool 호출 이력의 이중 저장 관점

현황:

- command_history: 모든 명령 실행 공통 기록
- tool_calls: tool 전용 상세 기록(stdout/stderr/권한결정 포함)

판단:

- 의도된 이중 기록일 수 있으나, tool 호출은 두 저장소에 동시에 남아 조회 기준이 분산됨
- 향후 리포트/통계에서 중복 카운트 위험

보완:

1. 정책 확정: tool 호출의 기준 저장소를 `tool_calls` 로 고정
2. `command_history` 에는 tool 호출을 요약 인덱스만 저장하거나(참조 ID만), 제외 정책 적용
3. 리포트 쿼리 레이어에서 중복 카운트 방지 규칙 명시

우선순위: 중간

## 9) 짧은 검증 시나리오 (tool_call 중복 저장 제외)

목적:

- `tool_call`/`tool_history_replay`가 `command_history`에 중복 저장되지 않는지 확인
- `history_list` 화면의 카운트 변화가 정책과 일치하는지 확인

사전 조건:

- 데스크탑 앱 실행 상태
- 명령 실행 패널에서 `history_list`, `state_get_status`, `tool_call` 실행 가능

절차:

1. `history_list 200` 실행 후 `total` 값을 `T1`로 기록
2. `tool_call builtin:omx_doctor []` 실행
3. `history_list 200` 실행 후 `total` 값을 `T2`로 기록

기대 결과:

- `T2 = T1 + 1`
- 증가한 1건은 1단계에서 실행한 `history_list` 자기 자신 적재분
- 2단계 `tool_call` 실행분은 `command_history` 카운트에 포함되지 않음

추가 교차 확인(권장):

4. `state_get_status` 실행
5. `history_list 200` 실행 후 `total` 값을 `T3`로 기록

기대 결과:

- `T3 = T2 + 2`
- 증가 2건의 구성: 직전 `history_list` 1건 + `state_get_status` 1건
- 즉, tool 호출은 제외되고 일반 명령만 누적됨
