# 작업문서 기반 개발 보완/검증 체크리스트

대상:
- 작업문서 폴더의 md 문서를 기준으로 개발을 수행하는 모든 참여자

목표:
- 문서 기반 개발 시 누락되기 쉬운 기술/운영/품질 검증 항목을 표준화한다.

---

## 1. 문서 사용 순서 (권장)

1. change-winapp.md
- 전환 방향, 아키텍처 원칙, 우선순위 확인

2. change-winapp-phase1-tickets.md
- 구현 티켓 단위 작업 계획 수립

3. change-winapp-실무등록용-백로그.md
- Epic/Sprint/Story/Task 매핑과 실행 단위 확인

4. taiga-스크럼-백로그-스프린트-등록-절차서.md
- 실제 Taiga 등록 및 운영 절차 수행

---

## 2. 개발 착수 전 필수 검증 (DoR)

- [ ] 목표 범위 확정: 이번 작업이 어느 Story/Task까지인지 확정
- [ ] 선행 의존성 확인: 선행 Task 완료 여부 확인
- [ ] 환경 점검: Node 버전, 패키지 설치, 빌드/테스트 명령 확인
- [ ] 브랜치 전략 확인: 작업 브랜치명/PR 대상 브랜치 확정
- [ ] 보안 점검: 문서 내 계정/토큰 평문 노출 여부 확인
- [ ] 산출물 정의: 코드/문서/테스트/운영증적 중 무엇을 남길지 확정

---

## 3. 문서-코드 매핑 검증

아래 규칙을 작업 시작 전에 확인한다.

1. 티켓과 코드 경로 매칭
- 티켓의 대상 경로가 실제 리포지토리 경로와 일치하는지 확인

2. 식별자 일관성
- EP/SP/US/TS 식별자가 문서마다 동일한지 확인

3. Taiga 객체 매칭
- 실무등록용-백로그의 ID 매칭표와 Taiga 실제 객체 ID가 일치하는지 확인

4. 우선순위 정합성
- Highest/High 항목이 현재 Sprint에 포함되어 있는지 확인

---

## 4. 구현 중 검증 포인트

## 아키텍처
- [ ] CLI adapter와 core 실행 경계가 유지되는가
- [ ] desktop/main, preload, renderer 경계를 침범하지 않는가
- [ ] tmux 의존 코드를 새로운 transport 경계 안으로 제한했는가

## 보안
- [ ] preload에서 최소 API만 노출했는가
- [ ] renderer에 nodeIntegration이 비활성화되어 있는가
- [ ] IPC 입력 검증(allowlist 또는 스키마 검증)이 있는가

## 품질
- [ ] 새 기능에 대한 단위/스모크 테스트가 추가되었는가
- [ ] 기존 핵심 명령 회귀가 없는가
- [ ] 실패 시 에러 메시지와 로그가 진단 가능 수준인가

---

## 5. 완료 검증 (DoD)

- [ ] 코드 변경 완료 및 빌드 성공
- [ ] 테스트 통과(신규 + 회귀)
- [ ] 작업 문서 업데이트(필요 시)
- [ ] Taiga 상태 업데이트(To Do -> In Progress -> Done)
- [ ] PR 생성 및 리뷰 피드백 반영
- [ ] 운영 영향(배포/설정/로그) 변경 사항 기록

---

## 6. Taiga 운영 검증

- [ ] Epic/Sprint/Story/Task 링크가 연결되어 있는가
- [ ] Story Point가 비어있지 않은가
- [ ] Assignee가 필요한 항목에 지정되어 있는가
- [ ] Milestone이 누락된 Story가 없는가
- [ ] 종료된 Sprint에 잔여 미완료 항목 정리 정책이 적용됐는가

## 8. 권장 산출물 묶음

작업 종료 시 최소 아래 4개는 남긴다.

1. 코드 변경(PR)
2. 테스트 결과(명령/로그)
3. Taiga 상태 변경 이력
4. 문서 갱신(변경점/운영 주의사항)

이 4개를 남기면 인수인계와 회고 품질이 크게 좋아진다.
