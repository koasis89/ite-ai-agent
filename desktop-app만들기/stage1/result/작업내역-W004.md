## 2. 작업 목표
WIN-004의 완료 기준에 맞춰 CLI/Desktop 공통으로 사용할 이벤트 스트림 계약과 emitter 인터페이스를 정의하고,
코어 실행기(`executeCommand`)가 optional emitter를 받아 이벤트를 발생시키도록 연결한다.

핵심 요구사항:
- 이벤트 타입 정의 (`command.started`, `command.progress`, `command.completed`, `command.failed`)
- `emit(event)` 인터페이스 정의
- 코어 실행기와 emitter 연동

- 수정: 한글해석-분석본/작업문서/change-winapp-phase1-tickets.md
- 신규: 한글해석-분석본/작업문서/작업내역폴더/작업내역-W004.md

## 4. 실제 변경 내역

### 4.1 이벤트 계약/인터페이스 신규 추가
`src/core/events.ts`를 신규 생성했다.

구현 내용:
1) emitter 인터페이스
- `CommandEventEmitter`
  - `emit(event: CommandEvent): void | Promise<void>`

2) 이벤트 생성 유틸
- `createStartedEvent(...)`
- `createProgressEvent(...)`
- `createCompletedEvent(...)`
- `createFailedEvent(...)`

3) 타입 재사용
- 기존 `src/core/contracts/command.ts`의 `CommandEvent`/`CommandResult`/`CommandRuntime` 계약을 재사용해 타입 중복을 피함

### 4.2 코어 실행기 emitter 연동
`src/core/execute-command.ts`에 아래를 반영했다.

1) 입력 계약 확장
- `ExecuteCommandInput`에 optional 필드 추가:
  - `runtime?: CommandRuntime`
  - `correlationId?: string`
  - `emitter?: CommandEventEmitter`

2) 안전한 emitter 호출 경계 추가
- 내부 `emitEvent(...)` 유틸을 통해 emitter 예외가 명령 실행을 깨지 않도록 격리

3) 실행 이벤트 발생 흐름 추가
- 명령 실행 전: `command.started`
- 핸들러 실행 중: `command.progress`
- 핸들러 성공 후: `command.completed`
- 알 수 없는 명령/예외 시: `command.failed`

4) 실패 코드 표준화
- unknown command: `CommandErrorCode.UNKNOWN_COMMAND`
- handler 예외: `CommandErrorCode.COMMAND_FAILED`

### 4.3 기준 문서 동기화
