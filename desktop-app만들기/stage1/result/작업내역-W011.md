- 작업 브랜치: webapp

## 2. 작업 목표
WIN-011 DoD에 따라 tmux 의존을 어댑터로 국한할 수 있는 공통 Worker Transport 인터페이스를 신규 정의하고, 기존 tmux 로직을 인터페이스로 노출하는 어댑터(`TmuxTransport`)를 제공한다.

핵심 요구사항:
- `WorkerTransport` 인터페이스 정의 (start / send / onOutput / onExit / stop / isAvailable / isAlive)
- `TmuxTransport` 어댑터 구현 (기존 `src/team/tmux-session.ts` 함수 위임)
- Transport 레지스트리 도입 (kind별 팩토리, 기본 transport 선택 정책)
- 인터페이스 계약 단위 테스트 확보
- 전체 빌드 무결성 유지 (회귀 없음)

설계 원칙:
- 코어는 transport 구체 구현(tmux, child_process 등)을 모른다.
- tmux 직접 호출은 어댑터 내부로 한정 가능한 표면을 제공한다.
- 한 번 `start()`한 인스턴스는 단일 워커 컨텍스트에 바인딩된다.
- `stop()`은 멱등이며 핸들러 예외는 격리한다.

### 3.3 구현 상세

1) `src/core/transport.ts`
- 타입/인터페이스
  - `WorkerTransportKind` (식별자)
  - `WorkerTransportStartSpec` (command/args/cwd/env/workerLabel)
  - `WorkerOutputChunk` (stream: 'stdout' | 'stderr' | 'pane', data, timestamp)
  - `WorkerOutputHandler`, `WorkerExitHandler`, `Unsubscribe`
  - `WorkerTransport` 인터페이스
    - `kind: WorkerTransportKind`
    - `isAvailable(): boolean`
    - `start(spec): Promise<void>`
    - `send(data): Promise<void>`
    - `onOutput(handler): Unsubscribe`
    - `onExit(handler): Unsubscribe`
    - `stop(): Promise<void>`
    - `isAlive(): boolean`
- 레지스트리
  - `WorkerTransportFactory`
  - `WorkerTransportRegistry` 클래스
    - `register / unregister / has / setDefault / getDefaultKind / create / createDefault / listKinds`
  - `workerTransportRegistry` (프로세스 공용 싱글톤 인스턴스)

2) `src/core/tmux-transport.ts`
- 타입
  - `TmuxTransportBinding` (sessionName / workerIndex / workerPaneId / workerCli)
  - `TmuxTransportOptions` (binding 주입형)
- 클래스 `TmuxTransport implements WorkerTransport`
  - `kind = 'tmux'`
  - `isAvailable()` → `isTmuxAvailable()` 위임
  - `start()` → 가용성 확인 후 active 마킹(중복 시작 금지)
  - `send()` → `tmuxSendToWorker(...)` 위임
  - `onOutput / onExit` → 핸들러 Set + Unsubscribe 반환
  - `stop()` → 멱등, exit 핸들러 통지, 핸들러 예외 격리
  - `isAlive()` → `isWorkerAlive(...)` 위임
  - `pushOutput(chunk)` → 외부 캡처 결과를 transport 구독자에게 푸시 (WIN-013에서 활용)
- 책임 경계 명시
  - 실제 pane 생성/파괴는 team runtime 책임 (createTeamSession/destroyTeamSession)
  - 본 어댑터는 단일 워커 (session, paneId) 컨텍스트에 한정된 transport 표면만 제공

3) `src/core/__tests__/transport.test.ts`
- `core/transport — WorkerTransport contract` (3 subtests)
  - lifecycle: start → send → onOutput → stop (멱등 stop 포함)
  - send before start throws
  - double start throws
- `core/transport — Registry` (3 subtests)
  - register / has / listKinds / create / setDefault / createDefault / unregister
  - create unknown / setDefault unknown / createDefault no-default 모두 throw
  - 프로세스 공용 `workerTransportRegistry` 싱글톤 확인
- `core/tmux-transport — adapter shape` (4 subtests)
  - binding/인터페이스 표면 확인
  - send before start throws (실어댑터)
  - output/exit 구독 등록/해제
  - `pushOutput`이 핸들러 예외와 무관하게 정상 핸들러에 전달
