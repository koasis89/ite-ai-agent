## 2. 작업 목표
WIN-012 DoD에 따라 tmux 없이 Windows에서 워커를 기동할 수 있는 기본 transport(`LocalProcessTransport`)를 신규 도입하고, Desktop Main 부팅 시 OS에 맞춰 기본 transport로 등록한다.

핵심 요구사항:
- `LocalProcessTransport`(Node.js `child_process.spawn` 기반) 구현 — `WorkerTransport` 인터페이스 완전 충족
- stdout/stderr → `onOutput` 연결, stdin → `send` 연결, 종료 → `onExit` 통지
- 명령 인자 escaping/검증 (path traversal, null byte 거부)
- 작업 디렉터리 제한 (`allowedCwdRoots` 화이트리스트)
- 명령 화이트리스트(`allowedCommands`), env 화이트리스트(`envAllowList`) 옵션
- Desktop Main에서 Windows의 경우 `local-process`를 기본 transport로 setDefault
- 회귀 없는 빌드/테스트 무결성 유지

설계 원칙(설계노트 §5.1과 정합):
- `shell: false` 강제 — 인자 escaping 우회 차단
- `windowsHide: true` — 자식 콘솔 창 미노출
- `stop()` 멱등 + SIGTERM → grace → SIGKILL 이스컬레이션
- 핸들러 예외 격리 (한 핸들러의 throw가 다른 구독자에 영향 없음)
- TTY/색상/리사이즈가 필요한 워커는 본 transport 범위 외 → 후속(`PtyLocalTransport`, node-pty)

### 3.3 구현 상세

1) `src/core/local-process-transport.ts`
- 옵션 타입 `LocalProcessTransportOptions`
  - `allowedCwdRoots?: readonly string[]` (cwd 화이트리스트)
  - `allowedCommands?: readonly string[]` (명령 화이트리스트, basename 또는 정확 일치)
  - `envAllowList?: readonly string[]` (env 화이트리스트)
  - `killGracePeriodMs?: number` (기본 3000ms)
- 내부 가드 함수
  - `validateArg(arg)` — null byte / `..` path traversal 거부
  - `isCwdAllowed(cwd, roots)` — `path.relative` 기반 안전 비교
  - `isCommandAllowed(command, allowList)` — 확장자(`.exe/.cmd/.bat/.ps1`) 제거 후 basename 비교
  - `filterEnv(source, allowList)` — 키 통과
- 클래스 `LocalProcessTransport implements WorkerTransport`
  - `kind = 'local-process'`
  - `isAvailable()` → 항상 true (Node 내장)
  - `start(spec)` →
    - command/args 검증, cwd/env 가드 적용
    - `spawn(command, args, { cwd, env, shell:false, windowsHide:true, stdio:['pipe','pipe','pipe'] })`
    - stdout/stderr `utf-8` 인코딩 설정 후 `dispatchOutput`로 fan-out
    - `error` 이벤트 발생 시 stderr로 흘려 운영 가시화 + `notifyExit`
  - `send(data)` → `child.stdin.write(data, 'utf-8', cb)` Promise 래핑, stdin 종료 시 throw
  - `onOutput / onExit` → Set 기반 구독, Unsubscribe 반환
  - `stop()` → 멱등. SIGTERM → grace(setTimeout) → SIGKILL, exit 이벤트로 settled, 핸들러 격리
  - `isAlive()` → `started && !stopped && exitCode===null && !killed`
- `registerLocalProcessTransport(registry, options, setAsDefault)` 헬퍼

2) `desktop/main/index.ts`
- 신규 import: `registerLocalProcessTransport`, `workerTransportRegistry`
- 신규 함수 `bootstrapWorkerTransports()`
  - 중복 등록 가드(`registry.has('local-process')`)
  - `allowedCwdRoots: [process.cwd(), app.getPath('userData')]`
  - `setAsDefault = (process.platform === 'win32')` → Windows 에서만 default 지정
- `app.whenReady()` 콜백 첫 단계로 호출

3) `src/core/__tests__/local-process-transport.test.ts`
- `basic lifecycle` (8 서브테스트)
  - kind/isAvailable/초기 isAlive
  - send before start throws
  - double start throws
  - stdout 캡처 + exit code 0 보고
  - stderr 분리 캡처 + non-zero exit
  - stdin send → child echo
  - onOutput unsubscribe 후 미수신
  - stop 멱등 (exit 핸들러 1회만 호출)
  - 핸들러 예외 격리
- `security guards` (7 서브테스트)
  - 경로 traversal 인자 거부
  - null byte 인자 거부
  - 명령 화이트리스트 미일치 거부
  - 명령 화이트리스트 basename 일치 통과
  - cwd 화이트리스트 외부 거부
  - cwd 화이트리스트 내부 통과
  - env 화이트리스트 외 키 드롭(`SECRET` → 빈 문자열)
- `registry integration` (1 서브테스트)
  - `registerLocalProcessTransport` 등록 / setDefault 동작
