## 범위

이 노트는 Rust 씬 어댑터 마이그레이션 이후 남은 고가치 런타임 이음새 간격을 기록합니다. 의도적으로 문서로만 작성되었으며 런타임 동작을 변경하지 않습니다.

## 요약

team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션)/런타임 스택에는 더 이상 Rust 계약 인터페이스이 누락되지 않습니다. 현재
문제 #1108 심 강화 패스, 발송/사서함 이중 쓰기 간격이 닫힙니다.
나머지 위험은 더 좁습니다.

1. **team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 메타데이터 해상도는 여전히 여러 파일에 걸쳐 대체됩니다**
2. **호환성 리더는 여전히 레거시/현재 우선순위 논리를 유지합니다**

이는 후속 솔기 강화 간격이며 Rust 런타임 방향이 잘못되었다는 증거는 아닙니다.

## 남은 솔기 간격

### 1. Rust 런타임 ← TS team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 상태 이중 쓰기

상태: **문제 #1108로 해결됨**

증거:

- `src/team/state/dispatch.ts:130-140`
- `src/team/state/mailbox.ts:45-49`

현재 상태:

- Rust 브리지/호환 파일은 이제 정식 발송 및 메일함 인터페이스입니다.
- 레거시 TS 디스패치/사서함 파일은 브리지가 비활성화되거나 사용할 수 없거나 읽을 수 없는 저하된 파트에 대해서만 대체 전용으로 유지됩니다.
- 감시자/런타임 경로가 좁아져 브리지 소유 성공 경로가 더 이상 의미상 이중 쓰기 레거시 사서함/디스패치 상태가 되지 않습니다.

위험:

- 잔여 회귀는 이제 활성 이중 쓰기 분기보다 잘못된 폴백 활성화로 인해 발생할 가능성이 더 높습니다.
- 테스트/문서가 표준 소유자 규칙을 명시적으로 유지하지 않는 한 향후 리팩터링으로 인해 실수로 폴백 동작이 확대될 수 있습니다.

원하는 최종 상태:

- Rust는 의미론적 전환을 소유합니다
- TS는 어댑터 로컬 또는 프레젠테이션 로컬 메타데이터만 저장합니다.

### 2. team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 메타데이터 확인은 여전히 ​​여러 파일에 걸쳐 있습니다.

증거:

- `src/team/api-interop.ts:423-438`

현재 상태:

- 작업자 ID 메타데이터가 먼저 확인됩니다.
- 그럼 `manifest.v2.json`
- 그럼 `config.json`

위험:

- 작업 디렉터리 또는 상태 루트 확인은 하나의 표준 소스 대신 대체 순서에 따라 달라질 수 있습니다.

원하는 최종 상태:

- team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 상태 루트/작업 디렉터리 확인을 위한 하나의 표준 메타데이터 소스
- 나머지 파일은 파생된 호환성 보기만 됩니다.

### 3. 런타임 소유권 계약과 컷오버 현실

상태: **발송/사서함 소유권에 대한 문제 #1108로 해결됨**

증거:

- `src/runtime/bridge.ts:2-6`
- `src/team/state/dispatch.ts:130-140`
- `src/team/state/mailbox.ts:45-49`

현재 상태:

- 브리지 계약과 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 파견/사서함 쓰기 경로는 이제 Rust 소유권에 동의합니다.
- 남은 소유권 작업은 발송/사서함 외부에 있으며 광범위한 메타데이터/대체 단순화에 중점을 둡니다.

위험:

- 문서가 정확하게 유지되지 않으면 기여자는 여전히 광범위한 메타데이터/대체 레이어를 소유권 레이어로 잘못 읽을 수 있습니다.
- 향후 호환성 작업으로 인해 계약/테스트 적용 범위 없이 실수로 JS 표준 쓰기가 다시 도입될 수 있습니다.

원하는 최종 상태:

- Rust는 런타임 상태 전환에 대한 유일한 의미론적 소유자입니다.
- TS는 얇은 리더기/전달 어댑터로만 남아 있습니다.

### 4. 호환성 판독기는 여전히 대체 우선순위 논리를 수행합니다.

증거:

- `src/hud/state.ts:107-123`
- `src/team/api-interop.ts:432-436`

현재 상태:

- 이는 마이그레이션 안전에 유용하지만 읽기 경로를 대상 아키텍처보다 더 복잡하게 유지합니다.

위험:

- 미래 형식 드리프트는 단일 표준 판독기 경계에서 실패하는 대신 대체 동작 내부에 숨길 수 있습니다.

원하는 최종 상태:

- 호환성 리더는 지원되는 마이그레이션 파트에 필요한 최소한의 폴백 동작만 유지합니다.
- 새로운 경로는 하나의 정식 Rust 작성 인터페이스을 먼저 읽습니다.
