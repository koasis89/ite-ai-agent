# 심층 인터뷰 1단계 성적표 형식 검증

날짜: **2026-03-25**
범위: 1단계 심층 인터뷰 질문 강화 패스에 대한 수동 기록 형식 검증.

이 파트은 결정론적 계약 테스트를 보완합니다. 심층 인터뷰 스킬(Skill)은 컴파일된 런타임이 아닌 지침 인터페이스이므로 이 문서는 대표 프롬프트에 대해 업데이트된 질문 흐름 계약을 검증하고 인터뷰가 이제 OMX 불변성을 삭제하지 않고 더 많은 압력을 가하는지 확인합니다.

## 검증 초점

아래 연습에서는 네 가지 필수 축을 확인합니다.

1. 질문 개수 압박
2. 깊이 압력
3. 가정 조사
4. 후속 압박

또한 보존된 불변성을 다시 확인합니다.
- 기술 계약에 모호성 게이팅이 남아 있습니다.
- 준비 게이트에는 여전히 명시적인 `Non-goals` 및 `Decision Boundaries`이 필요합니다.
- 컨텍스트 스냅샷/기록/사양 아티팩트가 여전히 존재합니다.
- 실행 핸드오프 계약이 여전히 존재합니다.
- 브라운필드 확인은 여전히 ​​발견된 증거를 인용하고 있습니다.

## 시나리오 1 - 그린필드 CLI

**즉각적인**
> 작업 관리 CLI를 구축하고 싶습니다.

**이 변경 후 예상되는 1단계 흐름**

1. **1라운드**
   - 보조자: 사용자가 CLI를 원하는 이유와 현재 워크플로의 어떤 오류가 요청을 트리거했는지 묻습니다.
2. **라운드 2**
   - 사용자: 저장소 전체에서 임시 작업을 계속 잊어버린다고 설명합니다.
   - 보조자: 동일한 이음새를 따라가며 달력/알림 흐름보다 CLI를 더 좋게 만드는 가정을 묻습니다.
3. **라운드 3**
   - 사용자: repo-local 컨텍스트와 빠른 캡처가 필요하다고 말합니다.
   - 보조자: 첫 번째 버전의 범위에서 명시적으로 벗어나야 하는 것이 무엇인지 묻습니다.
4. **4라운드**
   - 사용자: 동기화 없음, 다중 사용자 없음, GUI 없음이라고 말합니다.
   - 보조자: 빠른 캡처가 엄격한 작업 구조와 충돌하는 경우 어떤 절충안이 허용되지 않는지 묻습니다.

**합격 기준**
- 질문 수 압력: **통과** — 4라운드 전에 결정화가 발생하지 않습니다.
- 심도 압박: **통과** — 2라운드에서는 기능뿐만 아니라 기본 가정도 강화합니다.
- 가정 조사: **통과** — CLI 대 알림 가정이 즉시 도전됩니다.
- 후속 조치 압박: **통과** — 두 번째 질문은 일찍 요약하는 대신 신선한 내용을 담고 있습니다.

## 시나리오 2 — 브라운필드 인증 변경

**즉각적인**
> 이 브라운필드 앱에서 인증을 변경해야 합니다.

**이 변경 후 예상되는 1단계 흐름**

1. **비행 전**
   - 어시스턴트는 먼저 코드베이스 증거를 수집합니다.
2. **1라운드**
   - 보조자: "`src/auth/session.ts`에서 토큰 새로 고침 처리를 찾았습니다. 이 변경 사항이 해당 패턴을 따라야 할까요, 아니면 대체해야 할까요?"와 같은 증거 기반 확인 질문을 합니다.
3. **라운드 2**
   - 사용자: 새로 고침 루프가 실제 오류를 숨기기 때문에 교체하라고 합니다.
   - 보조자: 루프가 증상이 아닌 근본 원인임을 보여주는 증거가 무엇인지 묻습니다.
4. **라운드 3**
   - 사용자: 새로 고침 재시도가 업스트림 401 이탈을 숨긴다고 말합니다.
   - 보조자: 수정 사항에 미들웨어와 세션 저장소가 모두 필요한 경우 확인 없이 OMX가 취할 수 있는 결정 경계를 묻습니다.
5. **4라운드**
   - 사용자: 소유권 경계에 응답합니다.
   - 보조자: 명시적으로 범위를 벗어나야 하는 인증 동작을 묻습니다.

**합격 기준**
- 질문 수 압박: **통과** — 초기 브라운필드 확인 후에도 인터뷰가 계속 압박을 받고 있습니다.
- 심도 압박: **통과** — 2라운드에서는 대체 주장을 받아들이는 대신 근본 원인 증거를 강제합니다.
- 가정 조사: **통과** — 뒷받침되지 않는 브라운필드 주장은 직접적으로 이의를 제기합니다.
- 후속 조치 압력: **통과** — 각 답변은 계속 진행하기 전에 더 좁은 다음 질문을 생성합니다.
- 브라운필드 불변: **통과** — 질문은 추상적인 것이 아니라 증거에 근거한 상태로 유지됩니다.

## 시나리오 3 - 모호한 워크플로 요청

**즉각적인**
> 최초 기여자의 온보딩을 개선합니다.

**이 변경 후 예상되는 1단계 흐름**

1. **1라운드**
   - 보조자: 요청의 동기가 된 구체적인 기여자 실패 또는 포기 신호가 무엇인지 묻습니다.
2. **라운드 2**
   - 사용자: 설정 중에 새로운 기여자가 길을 잃었다고 합니다.
   - 보조자: 문서 검색 가능성이나 검토 대기 시간 대신 실제 차단 장치를 설정하는 가정이 무엇인지 묻습니다.
3. **라운드 3**
   - 사용자: 대부분의 이탈은 첫 번째 성공적인 로컬 실행 전에 발생한다고 말합니다.
   - 보조자: 단순화 모드를 사용하고 최초 기여자의 가장 작은 성공 결과가 무엇인지 묻습니다.
4. **4라운드**
   - 사용자: "앱을 실행하고 간단한 PR을 제출하세요."라고 말합니다.
   - 보조자: OMX가 단독으로 선택할 수 있는 결정 경계와 승인을 위해 에스컬레이션해야 하는 결정 경계를 묻습니다.

**합격 기준**
- 질문 수 압력: **통과** — 흐름은 핸드오프 전 최소 4라운드 동안 유지됩니다.
- 깊이 압력: **통과** — 솔루션 형성 전에 차단제 주장에 도전합니다.
- 가정 조사: **통과** — 보조자는 설정이 실제로 주요 실패 모드인지 여부를 테스트합니다.
- 후속 조치 압력: **통과** — 각 답변은 다음 경계 설정 질문으로 강화됩니다.
