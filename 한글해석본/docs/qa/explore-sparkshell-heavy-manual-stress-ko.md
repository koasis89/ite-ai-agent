# 무거운/수동 응력 파트: `omx-explore` 및 `omx-sparkshell`

날짜: **2026-03-11**
범위: 결정적 CI 파트을 보완하는 옵트인/수동 스트레스 시나리오:
- `npm run test:explore`
- `npm run test:sparkshell`

이 문서에서는 시끄러운 운영자 환경, 반복 호출 또는 판단 기반 증거 검토에 의존하기 때문에 기본 CI에서 의도적으로 제외된 과도한/수동 시나리오를 다룹니다.

## 전제 조건

저장소 루트에서:

```bash
npm run build:full
```

`npm run build:full`은 TypeScript 빌드와 함께 패키지된 탐색 하네스 및 Sparkshell 네이티브 바이너리를 다룹니다. 로컬 조사 중에 디버그 카고 빌드 경로를 특별히 원하는 경우에만 `npm run build:explore`을 별도로 사용하세요.

가능하면 깨끗하고 일회용 작업 공간을 사용하십시오. tmux-pane 적용 범위가 필요한 경우 tmux 내부에서 실행하고 먼저 대상 창 ID를 확인하세요.

## 검증된 명령 기준선

이 문서 전달 중에 다음 명령 형식이 확인되었습니다.

| Command | Result |
|---|---|
| `node bin/omx.js explore --help` | PASS |
| `node bin/omx.js sparkshell --help` | PASS |
| `node bin/omx.js sparkshell git --version` | PASS |
| `node bin/omx.js version` | PASS |
| `for i in $(seq 1 20); do node bin/omx.js sparkshell git --version; done` | PASS (bounded sample verified locally) |
| `seq 1 5 | xargs -I{} -P5 node bin/omx.js sparkshell git --version` | PASS (bounded sample verified locally) |

이 문서 전달 중 환경 제한 사항:
- `tmux -V`은 작동했지만 분리된 tmux 소켓 액세스는 `error connecting to /tmp/tmux-1000/default (Operation not permitted)`로 실패했습니다. 따라서 아래의 `--tmux-pane` 예제는 **문서화되어 있지만 이 세션에서 엔드투엔드 확인되지는 않습니다**.

## 증거 수집 템플릿

모든 과중/수동 시나리오 기록의 경우:
- 시나리오 ID
- 명령 실행
- 캡처된 stdout/stderr 또는 스크린샷/스니펫
- 관찰된 요약/지침/대체 동작
- 합격/불합격 결과
- 잔여 노트

## 대형/수동 파트 진출 기준

수동 시나리오는 다음 사항이 모두 참인 경우에만 통과됩니다.
1. 필요한 설정을 따랐으며,
2. 예상했던 증거가 포착됐고,
3. 실패 신호가 **나타나지** 않았습니다.
4. 결정론적 파트은 이후에도 계속 통과합니다.

최종 결정론적 재검사(나중 Ralph 확인 스윕에 필요하며 이 문서 전용 패스에서는 다시 실행하지 않음):

```bash
npm run test:explore
npm run test:sparkshell
```

## 시나리오 1: 큰 잡음이 있는 tmux-pane 캡처

**기본 CI에서 제외되는 이유**
- 라이브 tmux 서버와 현실적인 창 소음에 따라 다릅니다.
- 증거의 질은 부분적으로 운영자가 판단합니다. 요약은 정확한 표현이 아닌 중요한 사실을 보존해야 합니다.

**목표**
- 창 꼬리가 크고 시끄러울 때 `omx-sparkshell --tmux-pane`이 운영자에게 중요한 사실을 보존하는지 확인하세요.
- 요약/대체 경로가 여전히 운영자에게 올바른 다음 작업을 가리키는지 확인하세요.

**설정**
1. 혼합 신호 + 산만한 출력을 방출하는 창으로 tmux 세션을 시작합니다.
2. 시나리오를 실행하기 전에 반드시 보존해야 할 사실을 미리 선언하세요.
3. 창 ID를 캡처합니다.

꼭 보존해야 할 권장 사실:
- 실패한 명령 이름,
- 0이 아닌 최종 종료 또는 실패 상태,
- 실행 가능한 복구 또는 검사 힌트,
- 다음 작업에 필요한 파일/경로 식별자.

**명령 모양**

```bash
omx sparkshell --tmux-pane <pane-id> --tail-lines 400
```

**캡처할 증거**
- 원시 창 꼬리 샘플,
- 요약된 출력,
- 반드시 보존해야 할 사실 체크리스트,
- 출력의 다음 단계 명령이 실제로 충분한지 여부.

**실패 신호**
- 반드시 지켜야 할 사실이 누락되었습니다.
- 출력에 잘못된 다음 명령이 나와 있습니다.
- 출력은 노이즈로 인해 실제 오류 원인을 숨깁니다.

## 시나리오 2: 반복/동시 호출 스트레스

**기본 CI에서 제외되는 이유**
- 반복 및 동시성은 비용이 더 많이 들고 환경에 민감할 수 있습니다.
- 유용한 신호는 단일 실행이 아닌 전체 안정성에서 나옵니다.

**목표**
- 반복 실행으로 인해 대체 신뢰성이나 안내 품질이 저하되지 않는지 확인하세요.
- 동시 직접 명령 사용으로 인해 혼란스럽거나 사용할 수 없는 운영자 출력이 생성되지 않는지 확인하십시오.

**설정**
1. 안정적인 직접 명령과 대체 지향 명령 하나를 선택합니다.
2. 직접 명령을 반복해서 실행합니다.
3. 제한된 동시 호출 버스트를 실행합니다.

검증된 직접 명령 예:

```bash
node bin/omx.js sparkshell git --version
```

권장되는 반복 실행 형태:

```bash
for i in $(seq 1 20); do node bin/omx.js sparkshell git --version; done
```

권장되는 동시 실행 형태:

```bash
seq 1 5 | xargs -I{} -P5 node bin/omx.js sparkshell git --version
```

**캡처할 증거**
- 실행 횟수,
- 0이 아닌 출구,
- 표준 오류 스냅샷,
- 실행 전반에 걸쳐 출력 형식/지침이 표류하는지 여부.

**실패 신호**
- 간헐적인 실행 실패,
- 일관성이 없거나 잘린 지침 텍스트,
- 이유 없이 동일한 오류 모드에 대해 다른 복구 메시지가 표시됩니다.

## 시나리오 3: 유사 퍼즈 요약 말뭉치

**기본 CI에서 제외되는 이유**
- 코퍼스 디자인은 의도적으로 적대적이며 안정적인 고정 장치보다 빠르게 발전할 수 있습니다.
- 검토에는 미리 선언된 필수 보존 사실에 대한 의미론적 검사가 필요합니다.

**목표**
- 스트레스 `omx-explore` 및 `omx-sparkshell`은 산만한 신호 아래에 신호를 묻어두는 출력으로 요약됩니다.

**설정**
1. 다음을 사용하여 3-10개의 텍스트 설비를 구축하십시오.
   - 하나의 명확한 중대한 실패/성공 사실,
   - 많은 방해 요소 라인,
   - 실행 가능한 다음 단계 힌트 하나 이상,
   - 선택적으로 충돌하는 아슬아슬한 선택 요소.
2. 각 고정 장치에 대해 도구를 실행하기 전에 반드시 보존해야 할 사실을 작성하십시오.

제안된 말뭉치 차원:
- 거의 일치하는 파일 경로가 중복되었습니다.
- 하나의 실제 차단기로 여러 경고,
- 성공 텍스트 뒤에 나중에 실패 텍스트가 옵니다.
- 중요한 라인이 초기, 중간, 후반에 나타나는 긴 꼬리.

**캡처할 증거**
- 설비 ID,
- 보존해야 할 사실 목록,
- 관찰된 요약 출력,
- 누락/보존된 사실 체크리스트.

**실패 신호**
- 요약은 장식적인 소음을 유지하지만 실제 차단 요소를 제거합니다.
- 요약은 서로 다른 경로/오류를 오해의 소지가 있는 하나의 설명으로 축소합니다.
- 요약에는 실행 가능한 유일한 다음 단계가 생략되어 있습니다.

## 시나리오 4: 운영자 연습 검증

**기본 CI에서 제외되는 이유**
- 작업자가 도구 출력을 실제로 사용할 수 있는지 여부를 검증합니다.
- 엄격한 문자열 일치 이상의 판단이 필요합니다.

**목표**
- 도구에 표시된 복구/검사 명령을 문자 그대로 따랐을 때 충분한지 확인하세요.

**설정**
1. 하나의 직접 명령 시나리오와 하나의 tmux-pane 시나리오를 선택하십시오.
2. 도구에서 제안하는 정확한 다음 단계 명령을 기록합니다.
3. 먼저 편집하지 말고 해당 명령을 따르십시오.

유용한 직접 명령 기준:

```bash
node bin/omx.js sparkshell --help
node bin/omx.js explore --help
```

**캡처할 증거**
- 원본 도구 출력,
- 다음 단계 명령을 복사했습니다.
- 해당 명령을 실행한 결과,
- 추가 숨겨진 지식이 필요한지 여부.

**실패 신호**
- 제안된 명령이 구문상 잘못되었습니다.
- 시나리오에 대한 명령의 방향이 잘못되었습니다.
- 운영자는 복구하기 위해 문서화되지 않은 추가 컨텍스트를 추측해야 합니다.
