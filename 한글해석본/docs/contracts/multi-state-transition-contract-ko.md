# 다중 상태 전환 호환성 계약

이 문서는 승인된 작업에 대한 첫 번째 통과 피어 워크플로 상태 모델을 고정합니다.
`.omx/plans/prd-multi-state-compat.md`의 다중 상태 호환성 출시.

## 정식 진실 소스

런타임은 워크플로 상태를 다음의 조합으로 처리해야 합니다.

- `.omx/state/{scope}/<mode>-state.json` 아래의 모드 상태 파일
- `.omx/state/{scope}/skill-active-state.json` 아래의 정식 작업 흐름 열거

`skill-active-state.json`은 표준 활성 세트 인벤토리입니다. 기존 최상위 수준
`skill` 또는 `phase`과 같은 필드는 호환성 메타데이터로 남아 있을 수 있지만
여러 워크플로 구성원이 있는 경우 신뢰할 수 있는 활성 집합을 재정의하면 안 됩니다.
라이브 중입니다.

## 승인된 1차 통과 조합

이 롤아웃에서 허용되는 활성 세트 모양은 의도적으로 좁습니다.

- 추적된 워크플로를 위한 독립형 단일 워크플로 상태
- `team + ralph`
- `team + ultrawork`

결과 활성 세트는 피어 상태입니다. 두 멤버 모두 의미상 기본이 아닙니다.
먼저 활성화되었거나 레거시 최상위 수준을 차지했기 때문에
`skill` 필드.

## 독립형 전용 워크플로

이러한 워크플로는 이 패스에서 독립형으로 유지되며 중복 시도를 거부해야 합니다.

- `autopilot`
- `autoresearch`

거부된 중복은 현재 상태를 변경하지 않고 유지해야 합니다.

## 전환 규칙

표준 전환 도우미는 모든 작성자에 대해 세 가지 질문에 답해야 합니다.
워크플로 상태를 변경하는 소비자:

1. 요청된 전환이 현재 활성 세트에서 허용됩니까?
2. 허용되는 경우 결과 활성 세트는 무엇입니까?
3. 거부된 경우 어떤 운영자 지침이 표시되어야 합니까?

조합이 명시적으로 승인될 때까지 기본 규칙은 거부입니다.
돌연변이.

## 잘못된 전환 UX

거부된 모든 전환은 다음을 충족해야 합니다.

1. 현재 활성 상태를 변경하지 않고 유지
2. 거부된 조합의 이름을 명시적으로 지정하세요.
3. 재시도하기 전에 호환되지 않는 상태를 지우는 방법을 설명하세요.
4. 지원되는 투명 인터페이스을 모두 언급하십시오.
   - `omx state ...`
   - `omx_state.*` MCP 도구

운전자 안내 모양의 예:

> `<active-set>`이 활성 상태인 동안에는 `<requested>`을(를) 활성화할 수 없습니다. 지우기
> 먼저 `omx state ...` 또는 `omx_state.*` MCP 도구를 통해 호환되지 않는 상태,
> 그런 다음 전환을 다시 시도하십시오.

### 운영자 복구 예

CLI 패리티 인터페이스:

- `omx state clear --input '{"mode":"team"}' --json`
- `omx state clear --input '{"mode":"ralph","all_sessions":true}' --json`

MCP 패리티 인터페이스:

- `omx_state.state_clear({ mode: "team" })`
- `omx_state.state_clear({ mode: "ralph", all_sessions: true })`

## 브라운필드 소비자 기대

다음 인터페이스은 대신 동일한 전환 의미 체계를 사용해야 합니다.
자신만의 우선순위 규칙을 재창조합니다.

- `src/state/skill-active.ts` — 표준 활성 세트 지속성/동기화
- `src/hooks/keyword-detector.ts` — 키워드로 트리거되는 활성화는 또는
  단일 소유자를 덮어쓰는 대신 허용 목록에 따라 거부
- `src/modes/base.ts` — 모드 시작 유효성 검사는 동일한 전환을 따라야 합니다.
  규칙을 적용하고 동일한 운영자 지침을 내보냅니다.
- `src/mcp/state-server.ts` — 상태 쓰기/지우기는 결합된 상태를 유지해야 합니다.
  올바르게 삭제된 멤버만 제거합니다.
- `src/hud/state.ts` — HUD 렌더링은 승인된 결합 상태를 표시해야 합니다.
  레거시 최상위 메타데이터가 신뢰할 수 없는 경우
- `src/hooks/agents-overlay.ts` — AGENTS 오버레이 활성 모드 보고가 나열되어야 합니다.
  모든 활성 승인 회원
- `src/scripts/codex-native-hook.ts` — 중지/계속 논리는 다음을 준수해야 합니다.
  결합된 상태이며, 해당 멤버가 클리어되면 차단을 중지합니다.

## 범위 동작

세션 범위 상태는 존재하는 경우 권한을 유지합니다. 루트 범위는 여전히
호환성 대체만 가능합니다. 승인된 결합 세트의 한 멤버 지우기
전체 결합 상태를 실수로 삭제해서는 안 됩니다.

## 회귀 기대

테스트에서 다음 사항이 입증된 경우에만 구현이 완료된 것으로 간주되어야 합니다.

1. 표준 활성 상태는 다중 항목 활성 세트를 보유할 수 있습니다.
2. `team + ralph`은 두 활성화 주문 모두에서 허용됩니다.
3. `team + ultrawork`은 두 활성화 주문 모두에서 허용됩니다.
4. 지원되지 않는 겹침은 변형 없이 거부됩니다.
5. 거부 메시지에는 `omx state` 및 `omx_state.*`이 모두 언급되어 있습니다.
6. HUD/오버레이/스톱후크 소비자는 결합된 세트를 일관되게 존중합니다.
7. `autopilot` 및 `autoresearch`은 지원되지 않는 중첩 시도를 여전히 거부합니다. `autopilot -> ralplan`은 유일한 검토 기반 계획 루프백 예외입니다.
