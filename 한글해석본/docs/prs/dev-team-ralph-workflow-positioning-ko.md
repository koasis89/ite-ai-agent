# PR 초안: `ralplan -> team -> ralph` 작업 흐름 문서화

## 대상 지점
`dev`

## 요약
이 PR에서는 OMX의 가장 강력한 제어 워크플로 중 하나인 `ralplan -> team -> ralph`에 대해 설명합니다. 목표는 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 모드가 존재한다는 것뿐만 아니라 `$ultrawork`이 이미 병렬 실행을 제공하는 경우에도 이것이 중요한 이유를 README에서 설명하도록 만드는 것입니다.

중요한 점은 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 모드가 단순한 팬아웃이 아니라는 점입니다. 이는 코디네이션(Coordination)되고 검사 가능하며 런타임을 인식하는 실행입니다. 작업자는 차단 인식을 공유할 수 있고, 실행은 tmux 창과 내구성 상태를 통해 계속 표시되며, 리더는 복구 및 수명 주기 명령에 대한 더 강력한 제어권을 유지합니다. 이를 앞쪽에 `ralplan`, 뒤쪽에 `ralph`과 결합하면 빠르고 운영적으로 규칙적인 작업 흐름이 생성됩니다.

## 변경 사항
- `$team`과 `$ultrawork` 사이의 위치 차이를 명확히 합니다.
- 권장되는 높은 제어 워크플로우에 대한 README 지침을 추가하십시오: `ralplan -> team -> ralph`
- `--followup team`과 같은 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 후속 계획에 대한 더 강력한 `ralplan` 지원을 제안하는 이슈 초안을 추가하세요.

## 이것이 좋은 이유
- 기여자와 사용자에게 실제 제품의 강점을 보다 명확하게 설명합니다.
- 고급 사용자가 단순한 병렬 팬아웃 대신 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 모드를 선택해야 하는 경우를 이해하는 데 도움이 됩니다.
- 신뢰할 수 있는 설계 방향의 틀을 정합니다. 즉, team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 인력 배치 및 후속 실행을 이미 예상하는 계획 결과입니다.
- `autopilot`을(를) 동일한 기본 워크플로에 대한 자동 연결 레이어로 설명할 수 있으므로 설명하기도 더 쉬워집니다.

## 디자인 방향
문서에서는 `ralplan`을 team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 후속 조치가 더욱 명확해져야 하는 위치로 지정합니다. 향후 작업에서 `ralplan`은 파트 권장 사항, 역할 배치 힌트 및 직접 `team` 핸드오프에 대한 지침을 실행할 수 있으며 `ralph`은 지속성 및 확인 계층으로 유지됩니다.

## 기대되는 결과
- OMX의 오케스트레이션 모델을 평가하는 사용자를 위한 더 나은 온보딩
- team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션) 모드가 UltraWork와 다른 이유에 대한 더 쉬운 설명
- 향후 기능 작업을 위한 계획부터 실행까지의 경로가 더욱 명확해졌습니다.
- 혼합 CLI team(멀티 worker(실제작업을 수행하는 실행단위) 인터페이스 실행 오케스트레이션)과 런타임 집약적인 엣지 케이스 작업에 대한 더 나은 지원
